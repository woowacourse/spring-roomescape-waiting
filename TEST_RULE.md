## 사이클 규칙

### 테스트 보호 대상

우리가 작성한 비즈니스 규칙과 사용자에게 보이는 핵심 동작이 변경 이후에도 깨지지 않는 것을 보호한다.

- (If-Then) 만약 단순 위임이 아닌 비즈니스 로직(분기·데이터 가공)을 가지는 기능이라면 → 단위 테스트로 검증한다.


- (If-Then) 만약 의존 요소와의 상호작용(결합 지점) 자체를 검증해야 한다면 → 통합 테스트를 작성한다.


- (If-Then) 만약 사용자 관점의 핵심 흐름(예약 생성, 예약 대기, 인증)을 보호해야 한다면 → E2E로 최소 시나리오만 검증한다.


- (우선순위) 서비스 계층 DB 선택 순서:
    - 1) 서비스 자체로 유의미한 로직이 없는가? 없으면 통합 (테스트 DB / 인메모리 DB)
    - 2) Fake Repository 구현 비용이 적절한가? 적절하면 Fake 
    - 3) 협력 객체의 호출 행위만 확인하면 되는가? 그러면 Mock


- (순서) 테스트 작성 순서
    - 1) 무엇을 테스트할지 구현과 분리해서 정한다. 
    - 2) 그 다음 테스트 도구(단위/통합, Fake/Mock/인메모리)를 선택한다.


- (금지) 단순 설정·등록, 공통 인프라, 단순 위임 계층은 테스트하지 않는다. 이유: 같은 것을 여러 계층에서 중복 검증하게 된다.

### 트랜잭션 경계 규칙

- (If-Then) 만약 변경이 실패했을 때 비즈니스 정책상 모순되는 상태가 사용자에게 보여진다면 → 해당 변경들을 하나의 트랜잭션으로 처리한다.


- (If-Then) 즉각적인 결과 반영이 필요없고 비동기 재시도로 처리할 수 있다면 → 트랜잭션에 묶지 않고 별도로 처리한다.


- (금지) 작업이 실패했을 때 사용자에게 모순된 데이터가 노출되면 안된다.


- (우선순위) 경계를 결정할 때 순서:
    - 1) 사용자가 잘못된 중간 상태를 보게 되는가?
    - 2) 데이터 정합성이 깨지는가?
    - 3) 1·2 모두 아니면 묶지 않는다 (성능·복잡도 비용)


- (If-Then) 트랜잭션이 보장해야 하는 작업 단위는 테스트로 보호한다.
