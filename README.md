# 방탈출 예약 대기

# 기능 요구사항

## 1. 테마(Theme)

- [x] 테마 이름은 null이거나 빈값일 수 없다.
- [x] 테마 이름은 **20자 이하**여야 한다.
- [x] 테마 설명은 null이거나 빈값일 수 없다.
- [x] 테마 설명은 **200자 이하**여야 한다.
- [x] 테마 썸네일은 null일 수 없으며, 형식이 올바른 이미지 URL/파일이어야 한다.

### 테마 추가
- [x] 관리자는 테마를 추가할 수 있다. (이미지 파일 업로드 포함, `multipart/form-data`)
- [x] 이미 존재하는 테마는 중복으로 추가할 수 없다.

### 테마 삭제
- [x] 관리자는 테마를 삭제할 수 있다.
- [x] 존재하지 않는 테마는 삭제할 수 없다.
- [x] 예약이 존재하는 테마는 삭제할 수 없다.
- [x] 예약 대기가 존재하는 테마는 삭제할 수 없다.

### 테마 조회
- [x] 사용자는 전체 테마 목록을 조회할 수 있다.
- [x] 최근 1주 동안 예약이 많았던 인기 테마 상위 N개를 조회한다.

---

## 2. 예약 시간(Time)

- [x] 예약 시작 시간은 null일 수 없다.
- [x] 예약 시작 시간의 형식은 "18:00"와 같은 형식을 따른다.

### 예약 시간 추가
- [x] 관리자는 새로운 예약 시간을 추가할 수 있다.
- [x] 동일한 예약 시작 시간은 중복될 수 없다.

### 예약 시간 삭제
- [x] 관리자는 예약 시간을 삭제할 수 있다.
- [x] 존재하지 않는 예약 시간은 삭제할 수 없다.
- [x] 예약이 존재하는 예약 시간은 삭제할 수 없다.
- [x] 예약 대기가 존재하는 예약 시간은 삭제할 수 없다.

### 예약 시간 조회
- [x] 사용자는 등록된 모든 예약 시간을 조회할 수 있다.
- [x] (테마/날짜와 함께 조회 시) 각 시간 슬롯이 이미 예약 가능 여부(`isReservable`)와 함께 응답된다.

## 3. 예약(Reservation)

- [x] 예약자 이름은 null이거나 빈값일 수 없다.
- [x] 예약자 이름은 **10자 이하**여야 한다.
- [x] 예약 날짜는 null일 수 없으며, "2026-05-11" 형식을 따른다.
- [x] 예약 시간(`timeId`)·테마(`themeId`)는 null일 수 없다.

### 예약 생성
- [x] 예약을 생성할 수 있다.
- [x] 동일한 날짜·시간·테마 조합의 예약은 중복될 수 없다. (DB UNIQUE 제약 + 서비스 검증)
- [x] 존재하지 않는 예약 시간 / 테마로 예약할 수 없다.
- [x] 이미 지난 날짜·시간으로는 예약할 수 없다.
- [x] 이미 예약 대기가 존재하는 슬롯에는 신규 예약을 생성할 수 없다. (대기 순번 끼어들기 방지 — 자세한 사유는 §4 정책 참조)
- [x] 동시에 같은 빈 슬롯에 신규 예약이 들어오면 한 쪽만 성공하고 나머지는 409로 응답한다. (DB UNIQUE 제약 + `DuplicateKeyException` → `ConflictException` 변환)

### 예약 조회
- [x] 전체 예약 목록을 조회할 수 있다.
- [x] 사용자는 자신의 이름으로 본인의 **예약 + 예약 대기** 통합 목록을 조회할 수 있다. 각 항목에는 `status`(`RESERVED` / `WAITING`)와 대기인 경우 `sequence`(순번)가 함께 내려간다.
- [x] 본인 예약/예약 대기 목록은 **예약 그룹과 대기 그룹이 각각 날짜 → 시작 시간(→ 대기 순번) 오름차순으로 정렬된 뒤** 이어붙여 응답된다.

### 예약 변경
- [x] 사용자는 자신의 예약 날짜·시간·테마를 변경할 수 있다.
- [x] 존재하지 않는 예약은 변경할 수 없다.
- [x] 다른 사람의 예약은 변경할 수 없다.
- [x] 이미 지난 예약은 변경할 수 없다.
- [x] 변경하려는 날짜·시간·테마에 이미 다른 예약이 존재하면 변경할 수 없다.
- [x] 변경하려는 슬롯에 이미 예약 대기가 존재하면 변경할 수 없다. (예약 생성과 동일한 정책)
- [x] 변경 직후, 변경 전(old) 슬롯에 1번 대기자가 있으면 같은 트랜잭션에서 예약으로 자동 승격된다.

### 예약 취소
- [x] 사용자는 자신의 예약을 취소할 수 있다.
- [x] 존재하지 않는 예약은 취소할 수 없다.
- [x] 다른 사람의 예약은 취소할 수 없다.
- [x] 이미 지난 예약은 취소할 수 없다.
- [x] 취소 직후, 같은 트랜잭션에서 동일한 슬롯의 1번 대기자가 있으면 예약으로 자동 승격 된다.

## 4. 예약 대기(Waiting)

- [x] 같은 날짜·시간·테마에 이미 예약이 존재할 때, 다른 사용자가 대기 신청을 할 수 있다.
- [x] 예약 대기에도 예약자 이름/날짜/시간/테마의 null·빈값 제약이 동일하게 적용된다.

### 예약 대기 생성
- [x] 해당 날짜·시간·테마에 **이미 예약이 존재해야만** 대기를 생성할 수 있다.
- [x] 본인이 이미 예약한 시간에는 본인이 대기를 신청할 수 없다.
- [x] 같은 사용자가 같은 슬롯(`name + date + time + theme`)에 대기를 중복 등록할 수 없다. (DB UNIQUE 제약)

### 예약 대기 삭제
- [x] 사용자는 자신의 예약 대기를 삭제할 수 있다.
- [x] 다른 사람의 예약 대기는 삭제할 수 없다.

### 예약 대기 순번
- [x] 같은 날짜·시간·테마 안에서 `created_at` 와 `id` 기준 오름차순 순번이 부여된다.

### 예약 대기 승격
- [x] 확정 예약이 취소되면 같은 슬롯의 1번 대기자가 자동으로 예약으로 승격된다.
- [x] 승격은 다음 세 시점에서 트리거된다 : 사용자 예약 취소, 관리자 예약 강제 삭제, 예약 변경(슬롯 이동).
- [x] 승격 대상 슬롯이 이미 과거 시점이면 승격하지 않는다 (관리자 강제 삭제 케이스 가드).
- [x] 대기 순번은 조회 시 계산되기 때문에, 승격 또는 일반 삭제 이후 별도의 순번 재정렬 작업은 필요하지 않다.

### 정책 및 제약 (사이클 2)
- [x] **확정 예약 취소 시 #1 대기자가 자동으로 예약으로 승격된다.**
- [x] **대기가 남아 있는 슬롯에는 신규 예약 생성·예약 변경(다른 슬롯에서 이동) 둘 다 불가하다.**
- [x] **대기가 남아 있는 테마/시간 자원은 삭제할 수 없다** (참조 무결성 검증 — `409 Conflict`).
- [x] **대기 생성·승격 흐름의 비관적 락은 `reservation` 행 → `waiting` 행 순서로 획득한다.** 
  - 대기 생성: `reservation` 행을 X-Lock으로 잡아 예약 존재를 확인한 뒤 `waiting`에 INSERT 한다. (확인-후-삽입 사이에 예약이 사라지는 TOCTOU 방지)
  - 승격: `reservation`을 DELETE 한 뒤, 1번 대기자 `waiting` 행을 `FOR UPDATE`로 잡고 새 reservation INSERT + waiting DELETE를 수행한다.
  - 역순으로 진입하는 흐름이 새로 추가되면 동시 실행 시 데드락이 발생하므로, 새 동시성 코드도 반드시 이 순서를 지킨다.

## 5. 에러 응답 설계

- [x] 서비스 정책 위반, 유효하지 않은 입력, 존재하지 않는 리소스 등에 대해 의도된 4xx 에러 응답을 반환한다.
- [x] 500(서버 에러)이 사용자에게 노출되지 않도록 한다.
- [x] 브라우저에서 에러 발생 시 사용자에게 의미 있는 메시지가 표시되어야 한다.

### 응답 바디 공통 형식

```json
{
  "message": "이미 존재하는 예약 건입니다.",
  "errors": []
}
```

- `message`: 사용자/프론트엔드에게 노출되는 핵심 메시지.
- `errors`: 필드 검증 실패 시 필드별 상세(`{field, message}`) 배열. 그 외 케이스에서는 빈 배열.

필드 검증 실패(`MethodArgumentNotValidException`) 응답 예:
```json
{
  "message": "입력값 이상",
  "errors": [
    { "field": "name", "message": "이름은 비어 있을 수 없습니다." }
  ]
}
```

### HTTP 상태 매핑 (`GlobalExceptionHandler`)

| 예외                            | HTTP 상태 |
|-------------------------------|---------|
| `MethodArgumentNotValidException` (Bean Validation 실패) | 400     |
| `HttpMessageNotReadableException` (JSON 파싱 실패) | 400     |
| `IllegalArgumentException` (도메인 값 객체 검증) | 400     |
| `BadRequestException`         | 400     |
| `ForbiddenException`          | 403     |
| `NotFoundException`           | 404     |
| `ConflictException`           | 409     |
| `UnprocessableEntityException`| 422     |

## API 명세서

### Theme API

| 구분    | 메서드      | URI                                | 설명             | 요청 바디                                         | 응답 바디                                                                                  | 상태 코드 |
|-------|----------|------------------------------------|----------------|-----------------------------------------------|----------------------------------------------------------------------------------------|-------|
| 사용자   | `GET`    | `/themes`                          | 테마 전체 조회       | -                                             | `[{ id, name, description, url }, ...]`                                                | 200   |
| 사용자   | `GET`    | `/themes/popular?limit={N}`        | 인기 테마 상위 N개 조회 | -                                             | `[{ id, name, description, url }, ...]`                                                | 200   |
| 사용자   | `GET`    | `/themes/{id}/schedule?date={ymd}` | 특정 테마/날짜 슬롯 조회 | -                                             | `[{ timeResponse: { id, startAt }, isReservable }, ...]`                                 | 200   |
| 관리자   | `POST`   | `/admin/themes`                    | 테마 추가          | `multipart/form-data` — `name, description, file` | `{ id, name, description, url }`                                                       | 201   |
| 관리자   | `DELETE` | `/admin/themes/{id}`               | 테마 삭제          | -                                             | -                                                                                      | 204   |

### Time API

| 구분  | 메서드      | URI                  | 설명          | 요청 바디           | 응답 바디                    | 상태 코드 |
|-----|----------|----------------------|-------------|-----------------|--------------------------|-------|
| 사용자 | `GET`    | `/times`             | 전체 시간 조회    | -               | `[{ id, startAt }, ...]` | 200   |
| 관리자 | `POST`   | `/admin/times`       | 시간 추가       | `{ startAt }`   | `{ id, startAt }`        | 201   |
| 관리자 | `DELETE` | `/admin/times/{id}`  | 시간 삭제       | -               | -                        | 204   |

### Reservation API

| 구분  | 메서드      | URI                                  | 설명                       | 요청 바디                                | 응답 바디                                                                                                                                              | 상태 코드 |
|-----|----------|--------------------------------------|--------------------------|--------------------------------------|----------------------------------------------------------------------------------------------------------------------------------------------------|-------|
| 사용자 | `GET`    | `/reservations`                      | 전체 예약 조회                 | -                                    | `[{ id, name, date, timeResponse, themeResponse }, ...]`                                                                                           | 200   |
| 사용자 | `GET`    | `/reservations?userName={name}`      | 본인 이름으로 예약 + 대기 통합 조회    | -                                    | `{ reservationDetailResponses: [{ id, name, date, timeResponse, themeResponse, status: "RESERVED"\|"WAITING", sequence: number\|null }, ...] }` | 200   |

## DB 스키마 (`src/main/resources/schema.sql`)

```sql
reservation_time (id, start_at)
theme            (id, name, description, url)
reservation      (id, name, date, time_id, theme_id)
                 UNIQUE (date, time_id, theme_id)
waiting          (id, name, date, time_id, theme_id, created_at)
                 UNIQUE (name, date, time_id, theme_id)
```

- `reservation`은 같은 슬롯에 한 건만 존재 (UNIQUE).
- `waiting`은 **같은 사용자**의 같은 슬롯 대기 중복만 DB 레벨에서 차단. 다른 사용자의 대기는 여러 건 쌓일 수 있고, 순번으로 구분한다.

## 토론 규칙 

### 1. 리소스 식별 기준

- URL에 들어갈 것
  - **경로**: 리소스(필요한 데이터)의 최소 단위
  - **쿼리스트링**: 그 리소스를 필터링하기 위한 수단
  - 가독성을 위해 하이픈(`-`)을 활용한다.
- URL 결정 순서:
  1. 사용자 시나리오 / 예상 화면을 적는다.
  2. 리소스(명사) + HTTP 메서드를 식별한다.
  3. POST는 body, GET은 쿼리스트링 키를 설계한다.

### 2. 서버/클라이언트 책임

- 클라이언트가 자체적으로 판단할 수 없는 비즈니스 로직이면, 서버에서 모두 가공해 제공한다. (클라이언트가 보낸 데이터를 신뢰하지 않는다.)

### 3. 관리자 / 사용자 API 분리

- 같은 자원이라도 **목적이 다르면** 분리한다. 같은 행위·같은 목적이면 분리하지 않는다.
- 인증 도입 시 관리자/사용자가 갈라질 가능성이 크기 때문.

### 4. 좋은 API의 정의

- URI만 보고도 리소스 구조가 추측 가능해야 한다.
- 에러 상황에 대해 클라이언트가 충분히 알 수 있어야 한다 — **어떤 값이 왜 문제인지** 응답에 담는다.

### 5. 계층 별 메서드 시그니처 통일

- 같은 계층 안에서 메서드 시그니처/네이밍을 일관되게 유지해 가독성과 활용성을 높인다.

### 6. 비즈니스 규칙 / 형식 위반 → 4xx

- 서버 → 클라이언트 에러 메시지는 다음 형태를 사용한다.
  ```json
  { "message": "...", "errors": [] }
  ```
- 에러 메시지 수신 대상은 **프론트엔드 개발자 + 최종 사용자**이며, 따라서 메시지에는 어떤 값이 왜 문제인지와 해결 방향까지 포함한다.

### 테스트 전략
1. 코드 리팩토링 시, 동일 요청에 대한 응답 값이 변경 되는 경우를 방지하기 위해 테스트 코드를 작성한다.
2. 계층 별 테스트 작성 방식을 분리한다

  * 통합테스트(및 E2E 테스트): controller 계층
  * 단위테스트: service, repository, domain 계층

3. Production DB는 사용하지 않고, Mocking은 service 계층에서만 사용한다.
4. 다른 로직을 단순 호출만 하는 메서드는 테스트하지 않는다.

### 트랜잭션 경계 규칙
1. 한 메서드에서 여러 데이터를 다루면(select 제외) 데이터의 원자성을 보장해야한다.
2. 비즈니스 요구사항 단위를 하나의 작업 단위로 묶는다.
3. 단일 작업의 경우 하나의 트랜잭션으로 묶지 않아도 된다.
4. 트랜잭션 실패 시, 사용자에게 에러의 원인과 함께 재시도를 요청한다.
