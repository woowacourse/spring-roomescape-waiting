## 에러 응답 명세

모든 에러 응답은 JSON 객체로 반환한다.

```json
{
  "code": "ERROR_CODE",
  "detail": "에러 상세 설명"
}
```

`code`는 클라이언트가 분기하기 위한 안정적인 값이다. `detail`은 API 소비자가 에러 원인을 이해하기 위한 설명이며, 클라이언트는 `detail`이 아닌 `code`를 기준으로 처리한다.

| error code | detail | 상태 코드 | 상황 |
|------------|--------|----------|------|
| INVALID_INPUT | name은 비어 있을 수 없습니다. | 400 | 요청 본문의 필수 문자열 값이 비어 있음 |
| INVALID_INPUT | name은 255자를 넘을 수 없습니다. | 400 | 요청 본문의 문자열 길이가 허용 범위를 초과함 |
| INVALID_INPUT | timeId는 비어 있을 수 없습니다. | 400 | 요청 본문의 필수 ID 값이 누락됨 |
| INVALID_INPUT | timeId는 양수이어야 합니다. | 400 | 요청 본문의 ID 값이 양수가 아님 |
| INVALID_INPUT | themeId는 비어 있을 수 없습니다. | 400 | 요청 본문의 필수 ID 값이 누락됨 |

## 미션 중 기록

### 규칙을 적용해서 변경한 코드

토론에서 정한 "외부 의존을 실제로 연결하지 않아도 핵심 로직을 검증할 수 있다면 필요에 따라 모킹을 사용한다"는 규칙을 `ReservationWaitingValidatorTest`에 적용했다.
예약 대기 등록 예외 정책은 실제 DB 저장·조회 자체보다, 주어진 예약/대기 상태에서 대기 신청을 허용하거나 거부하는 정책 분기를 확인하는 것이 중요하다고 보았다.
그래서 `ReservationRepository`, `ReservationWaitingRepository`를 실제 DB와 연결하지 않고 mock으로 대체했다. 
이를 통해 예약 가능한 슬롯에는 대기할 수 없음, 본인 예약 슬롯에는 대기할 수 없음, 중복 대기 불가, 지난 시간 대기 불가 같은 정책을 Repository 응답에 따라 검증했다.

### 테스트 작성이 어려웠던 코드

이번 미션에서는 예약 대기 신청, 취소, 조회, 내 예약 목록 조회까지 기능이 늘어나면서 작성해야 할 테스트 범위가 크게 늘었다. 
처음에는 몇 개의 테스트를 직접 작성했지만, 컨트롤러 테스트와 통합 테스트까지 작성해야 하는 상황에서 무엇을 어디까지 검증해야 할지 막막하게 느꼈다.
특히 컨트롤러 테스트는 요청 값, 응답 상태 코드, 응답 body, 예외 응답을 함께 검증해야 했고, 
통합 테스트는 실제 API 흐름에 필요한 사전 데이터와 검증 조건을 준비해야 해서 작성 비용이 크게 느껴졌다.
이 과정에서 AI를 활용해 테스트 초안을 만들고, 생성된 테스트가 어떤 기능 흐름과 예외 상황을 검증하는지 읽어보며 보완하는 방식으로 진행했다. 
다만 테스트를 직접 설계했다기보다 생성된 테스트를 따라가며 이해한 부분이 많아서, 앞으로는 기능별로 어떤 테스트가 필요한지 먼저 나누는 연습이 필요하다고 느꼈다.

### 막힌 순간

이번 미션에서는 페어와 함께 구현을 진행하는 과정에서 전체 코드의 맥락을 따라가는 데 어려움을 느꼈다.
요구사항은 이해했지만, 실제로 구현하려고 하니 어떤 순서로 작업해야 하는지 막막했다.
특히 도메인과 테이블 구조를 먼저 정리하지 않은 상태에서 컨트롤러와 서비스 코드를 함께 수정하려고 하니 한 번에 고려해야 하는 범위가 커졌다. 
예약 대기를 예약에 종속된 데이터로 볼지, 별도 데이터로 볼지, 순번을 저장할지 계산할지 같은 결정을 먼저 정리하지 않으면 이후 구현이 계속 복잡해진다는 것을 느꼈다.
이번 경험을 통해 기능을 추가하기 전에 도메인 개념, 스키마, 핵심 정책을 먼저 정리하는 연습이 필요하다는 것을 크게 느꼈다.

# 규칙 문서(그룹)

## 테스트가 보호하는 대상 규칙
(If-Then) 만약 메서드나 기능이 외부에 드러난 동작을 가진다면
→ 우리는 그 동작이 기대한 대로 유지되는지를 테스트한다.
이유: 테스트는 구현 코드 자체가 아니라, 리팩토링이나 수정 이후에도 기능이 깨지지 않았는지 알려주는 안전장치이기 때문이다.


## 테스트 단위 선택 기준
(If-Then) 만약 외부 의존 없이 메서드나 도메인 규칙의 동작을 검증할 수 있다면
→ 단위 테스트로 검증한다.
이유: 빠르고, 어떤 동작이 깨졌는지 가장 명확하게 알 수 있기 때문이다.

(If-Then) 만약 Spring, DB, Repository, Controller처럼 여러 계층의 연결이 검증 대상이라면
→ 통합 테스트로 검증한다.
이유: 순수 자바 단위 테스트만으로는 외부 의존과 계층 간 결합에서 발생하는 문제를 잡기 어렵기 때문이다.

(If-Then) 만약 HTTP 요청부터 DB 저장까지 사용자의 실제 기능 흐름을 보호해야 한다면
→ E2E 테스트로 검증한다.
이유: 기능이 전체 흐름에서 제대로 동작하는지를 확인해야 하기 때문이다.

(우선순위) 같은 검증을 어디서 할지 결정할 때 순서:
1. 외부 의존 없이 검증 가능한가? 가능하면 단위 테스트
2. 계층 간 연결이나 Spring 의존성이 검증 대상인가? 통합 테스트
3. 사용자 시나리오 전체가 검증 대상인가? E2E 테스트

(참고)
통합 테스트: 서버 내부 부품들이 잘 붙어 있는지 확인 / 내부 연결 검증
E2E 테스트: 사용자가 하는 행동이 끝까지 성공하는지 확인 / 실제 사용자 흐름 검증


## DB·HTTP 의존 코드 테스트 기준
(If-Then) 만약 DB와 연결된 Repository나 Service 흐름을 검증해야 한다면
→ 운영 DB가 아닌 테스트용 DB 또는 인메모리 DB를 사용한다.
이유: 운영 DB를 사용하면 데이터가 오염될 수 있고, 외부 환경 변화 때문에 테스트가 깨질 수 있기 때문이다.

(If-Then) 만약 외부 의존을 실제로 연결하지 않아도 핵심 로직을 검증할 수 있다면
→ 필요에 따라 모킹을 사용한다.
이유: 테스트 범위를 좁히고 핵심 동작만 빠르게 확인할 수 있기 때문이다.


## 테스트 제외 기준
(If-Then) 만약 getter처럼 값을 그대로 반환하기만 한다면
→ 별도 테스트하지 않는다.
이유: 깨질 만한 동작이 거의 없기 때문이다.

(If-Then) 만약 다른 객체의 메서드를 그대로 호출만 한다면
→ 별도 테스트하지 않는다.
이유: 실제 기능 테스트에서 이미 확인되기 때문이다.

(우선순위) 테스트 제외 여부를 정할 때 순서:
1. 기능 결과가 달라질 수 있는가? → 테스트한다
2. 중요한 규칙이나 예외 상황이 있는가? → 테스트한다
3. 단순 반환이나 단순 호출뿐인가? → 테스트하지 않는다
