## 에러 응답 명세

모든 에러 응답은 JSON 객체로 반환한다.

```json
{
  "code": "ERROR_CODE",
  "detail": "에러 상세 설명"
}
```

`code`는 클라이언트가 분기하기 위한 안정적인 값이다. `detail`은 API 소비자가 에러 원인을 이해하기 위한 설명이며, 클라이언트는 `detail`이 아닌 `code`를 기준으로 처리한다.

| error code | detail | 상태 코드 | 상황 |
|------------|--------|----------|------|
| INVALID_INPUT | name은 비어 있을 수 없습니다. | 400 | 요청 본문의 필수 문자열 값이 비어 있음 |
| INVALID_INPUT | name은 255자를 넘을 수 없습니다. | 400 | 요청 본문의 문자열 길이가 허용 범위를 초과함 |

## 트랜잭션/동시성 고민 내용

### 예약 대기 등록 시 해당 슬롯의 예약이 변경/삭제될 경우

> **도메인 규칙**  
> 예약 대기는 예약이 존재하는 슬롯에만 신청할 수 있다.

1. 예약 대기 등록을 위해 예약 조회 -> 존재 확인
2. 다른 쓰레드에서 해당 예약 삭제
3. 대기 등록  
-> 예약 없는 슬롯에 대기만 남을 수 있음

**현재 처리**  
예약 대기 저장이 끝날 때까지 해당 예약이 변경/삭제되지 않도록 조회하는 예약 row에 락을 건다.


### 예약 삭제/변경 시 대기 자동 승격이 필요한 경우

> **도메인 규칙**  
> 예약 삭제/변경으로 기존 슬롯이 비면 첫 번째 예약 대기를 예약으로 자동 승격한다.

1. 예약 삭제/변경을 위해 예약 조회
2. 해당 예약의 기존 슬롯에 있는 첫 번째 대기 조회
3. 예약 삭제 또는 변경
4. 첫 번째 대기를 예약으로 등록
5. 첫 번째 대기 삭제  
-> 중간에 일부 작업만 성공하면 예약은 비었는데 대기가 그대로 남거나, 예약으로 승격됐는데 대기 row도 남는 상태가 될 수 있음

**현재 처리**  
예약 삭제/변경부터 대기 승격, 승격된 대기 삭제까지 하나의 트랜잭션으로 묶는다.  
삭제/변경 대상 예약 row와 자동 승격 대상이 되는 첫 번째 대기 row에 락을 건다.


### 예약 변경 중 변경하려는 슬롯을 다른 요청이 먼저 예약할 경우

> **도메인 규칙**  
> 하나의 슬롯에는 하나의 예약만 존재할 수 있다.

1. A가 예약을 다른 슬롯으로 변경하기 위해 예약 가능 여부 확인
2. B가 같은 슬롯에 예약 생성
3. A가 예약 변경 시도  
-> A가 먼저 요청을 시작했더라도 저장 시점에는 이미 예약된 슬롯일 수 있음

**현재 처리**  
예약 가능 여부를 먼저 검증하되, 최종 저장 시점에는 DB unique 제약으로 한 번 더 막는다.  
이 경우 A는 `이미 예약된 시간입니다.` 예외 응답을 받는다.


### 자동 승격 대상 대기와 사용자 대기 취소가 동시에 일어날 경우

> **도메인 규칙**  
> 사용자가 대기 취소 성공 응답을 받았다면 실제로 대기가 취소되어야 한다.

1. B가 예약 대기 취소를 위해 대기 조회
2. A가 같은 슬롯 예약을 삭제하고 B의 예약 대기를 자동 승격
3. A가 승격된 대기 row 삭제
4. B가 대기 삭제 시도  
-> B가 취소 성공 응답을 받았지만 실제로는 이미 예약으로 승격된 상태일 수 있음

**현재 처리**  
사용자 대기 취소 시 삭제된 row가 0건이면 `존재하지 않는 예약 대기입니다.` 예외를 응답한다.

**판단**   
사용자 대기 취소는 일반 조회 후 삭제하므로, 성공 응답이 실제 결과와 달라지지 않도록 deleteCount를 확인하도록 했다.

### 관리자가 과거 예약을 삭제할 경우

> **도메인 규칙**  
> 과거 예약 삭제는 데이터 정리로 볼 수 있지만, 과거 대기가 예약으로 승격되면 안 된다.

1. 관리자가 과거 예약 삭제
2. 해당 슬롯에 과거 대기 존재
3. 첫 번째 대기가 예약으로 자동 승격  
-> 이미 지난 시간에 새로운 예약이 생길 수 있음

**현재 처리**  
관리자가 과거 예약을 삭제하면 예약만 삭제하고 대기 자동 승격은 하지 않는다.

**판단**  
관리자의 과거 예약 삭제 자체는 허용하되, 자동 승격은 미래 예약 삭제/변경에만 적용하는 것이 자연스럽다고 판단했다.

### 정리(동시성 처리 기준)

모든 동시성 경합을 고려하고 과하게 막지 않는다.  
데이터 정합성이 깨지는 경우를 우선적으로 막는다.  
그 외의 자원 경쟁은 DB 제약과 실패 응답으로 처리한다.

## 미션 중 기록 - 사이클1

### 규칙을 적용해서 변경한 코드

토론에서 정한 "외부 의존을 실제로 연결하지 않아도 핵심 로직을 검증할 수 있다면 필요에 따라 모킹을 사용한다"는 규칙을 `ReservationWaitingValidatorTest`에 적용했다.
예약 대기 등록 예외 정책은 실제 DB 저장·조회 자체보다, 주어진 예약/대기 상태에서 대기 신청을 허용하거나 거부하는 정책 분기를 확인하는 것이 중요하다고 보았다.
그래서 `ReservationRepository`, `ReservationWaitingRepository`를 실제 DB와 연결하지 않고 mock으로 대체했다.
이를 통해 예약 가능한 슬롯에는 대기할 수 없음, 본인 예약 슬롯에는 대기할 수 없음, 중복 대기 불가, 지난 시간 대기 불가 같은 정책을 Repository 응답에 따라 검증했다.


### 테스트 작성이 어려웠던 코드

이번 미션에서는 예약 대기 신청, 취소, 조회, 내 예약 목록 조회까지 기능이 늘어나면서 작성해야 할 테스트 범위가 크게 늘었다.
처음에는 몇 개의 테스트를 직접 작성했지만, 컨트롤러 테스트와 통합 테스트까지 작성해야 하는 상황에서 무엇을 어디까지 검증해야 할지 막막하게 느꼈다.
특히 컨트롤러 테스트는 요청 값, 응답 상태 코드, 응답 body, 예외 응답을 함께 검증해야 했고,
통합 테스트는 실제 API 흐름에 필요한 사전 데이터와 검증 조건을 준비해야 해서 작성 비용이 크게 느껴졌다.
이 과정에서 AI를 활용해 테스트 초안을 만들고, 생성된 테스트가 어떤 기능 흐름과 예외 상황을 검증하는지 읽어보며 보완하는 방식으로 진행했다.
다만 테스트를 직접 설계했다기보다 생성된 테스트를 따라가며 이해한 부분이 많아서, 앞으로는 기능별로 어떤 테스트가 필요한지 먼저 나누는 연습이 필요하다고 느꼈다.


### 막힌 순간

이번 미션에서는 페어와 함께 구현을 진행하는 과정에서 전체 코드의 맥락을 따라가는 데 어려움을 느꼈다.
요구사항은 이해했지만, 실제로 구현하려고 하니 어떤 순서로 작업해야 하는지 막막했다.
특히 도메인과 테이블 구조를 먼저 정리하지 않은 상태에서 컨트롤러와 서비스 코드를 함께 수정하려고 하니 한 번에 고려해야 하는 범위가 커졌다.
예약 대기를 예약에 종속된 데이터로 볼지, 별도 데이터로 볼지, 순번을 저장할지 계산할지 같은 결정을 먼저 정리하지 않으면 이후 구현이 계속 복잡해진다는 것을 느꼈다.
이번 경험을 통해 기능을 추가하기 전에 도메인 개념, 스키마, 핵심 정책을 먼저 정리하는 연습이 필요하다는 것을 크게 느꼈다.


## 미션 중 기록 - 사이클2

### 규칙을 적용해서 변경한 코드

토론에서 정한 "한 유스케이스 안의 일부 변경만 성공했을 때 사용자가 잘못된 예약/대기 상태를 보게 된다면 하나의 작업 단위로 묶는다"는 규칙을 `ReservationService`에 적용했다.

예약 삭제나 변경으로 기존 슬롯이 비면 첫 번째 대기가 예약으로 자동 승격되어야 한다.
이때 예약 삭제/변경, 첫 번째 대기 조회, 예약 등록, 승격된 대기 삭제 중 일부만 성공하면 사용자가 보는 예약/대기 상태가 깨질 수 있다고 보았다.

그래서 `deleteByUser`, `deleteByAdmin`, `updateByUser`를 하나의 트랜잭션으로 처리하고,
공통 흐름은 `deleteAndPromoteWaiting`, `promoteWaiting`으로 분리했다.
또 삭제/변경 대상 예약과 자동 승격 대상 대기는 `FOR UPDATE`로 조회해 같은 대상을 동시에 변경하지 못하도록 했다.

다만 모든 경합을 락으로 막지는 않았다.
예약 변경 중 변경하려는 슬롯을 다른 요청이 먼저 예약하는 경우처럼 자원 경쟁에 가까운 상황은 DB unique 제약과 실패 응답으로 처리했다.


### 테스트 작성이 어려웠던 코드

자동 승격 후 남은 대기 순번이 재정렬되는 흐름을 테스트하는 부분이 어려웠다.
서비스 단위 테스트에서는 첫 번째 대기가 예약으로 등록되고 대기 row가 삭제되는지는 확인할 수 있지만,
실제로 남은 대기의 순번이 다시 1번으로 보이는지는 Repository 조회 로직까지 함께 지나가야 확인할 수 있었다.

그래서 `MissionStepTest`에서 실제 API 흐름으로 테스트했다.
예약 1개와 대기 2개를 만든 뒤 예약을 삭제하거나 변경했을 때,
첫 번째 대기는 예약 상태가 되고 두 번째 대기는 대기 순번이 1번으로 바뀌는지 확인했다.

이 테스트를 작성하면서 단위 테스트와 E2E 테스트의 역할을 다시 구분하게 되었다.
자동 승격 메서드가 어떤 Repository 메서드를 호출하는지는 서비스 테스트에서 확인하고,
사용자가 실제로 보게 되는 예약/대기 상태와 순번은 E2E 테스트로 확인하는 것이 더 적합하다고 판단했다.


### 막힌 순간

자동 승격을 구현하면서 어떤 동시성 상황까지 막아야 하는지 기준을 잡는 부분이 가장 막혔다.

처음에는 예약 삭제/변경, 대기 조회, 예약 등록, 대기 삭제가 모두 이어지기 때문에 모든 작업에 강하게 락을 걸어야 하는지 고민했다.
하지만 모든 경합을 막으려고 하면 구현 복잡도와 락 범위가 커지고, 오히려 어떤 상황을 왜 막는지 설명하기 어려워진다고 느꼈다.

그래서 기준을 다시 세웠다.
데이터 정합성이 깨지는 경우와 성공 응답이 실제 상태와 달라지는 경우는 막고,
단순히 동시에 같은 자원을 선점하려는 상황은 DB 제약과 실패 응답으로 처리하기로 했다.

이 기준을 세운 뒤에는 자동 승격 대상 대기와 사용자 대기 취소가 동시에 일어나는 경우가 보였다.
사용자가 대기 취소 성공 응답을 받았는데 실제로는 이미 예약으로 승격된 상태라면 응답이 실제 상태와 달라진다.
그래서 사용자 대기 취소에서는 deleteCount를 확인해 실제로 삭제된 row가 없으면 예외를 응답하도록 변경했다.

# 규칙 문서(그룹)

## 테스트가 보호하는 대상 규칙
(If-Then) 만약 메서드나 기능이 외부에 드러난 동작을 가진다면
→ 우리는 그 동작이 기대한 대로 유지되는지를 테스트한다.
이유: 테스트는 구현 코드 자체가 아니라, 리팩토링이나 수정 이후에도 기능이 깨지지 않았는지 알려주는 안전장치이기 때문이다.


## 테스트 단위 선택 기준
* 격리를 하면서 controller는 테스트하지 않는다. 인터셉터, 핸들러도 역시(공인된 라이브러리나 프레임워크 같은 기능의 경우를 뜻함)
* 서비스는 레포지토리를 모킹하면서 자신의 퍼블릭 메서드에 대한 단위 테스트를 진행한다.
* 레포지토리는 @JdbcTest를 사용해서 통합 테스트를 진행한다.
* 도메인은 메서드 단위로 단위 테스트를 진행한다.
* 만약 HTTP 요청부터 DB 저장까지 사용자의 실제 기능 흐름을 보호해야 한다면 E2E 테스트로 검증한다.

(우선순위) 같은 검증을 어디서 할지 결정할 때 순서:
1. 외부 의존 없이 검증 가능한가? 가능하면 단위 테스트
2. 계층 간 연결이나 Spring 의존성이 검증 대상인가? 통합 테스트
3. 사용자 시나리오 전체가 검증 대상인가? E2E 테스트


## DB·HTTP 의존 코드 테스트 기준
(If-Then) 만약 DB와 연결된 Repository나 Service 흐름을 검증해야 한다면
→ 운영 DB가 아닌 테스트용 DB 또는 인메모리 DB를 사용한다.
이유: 운영 DB를 사용하면 데이터가 오염될 수 있고, 외부 환경 변화 때문에 테스트가 깨질 수 있기 때문이다.

(If-Then) 만약 외부 의존을 실제로 연결하지 않아도 핵심 로직을 검증할 수 있다면
→ 필요에 따라 모킹을 사용한다.
이유: 테스트 범위를 좁히고 핵심 동작만 빠르게 확인할 수 있기 때문이다.


## 테스트 제외 기준
(If-Then) 만약 getter처럼 값을 그대로 반환하기만 한다면
→ 별도 테스트하지 않는다.
이유: 깨질 만한 동작이 거의 없기 때문이다.

(If-Then) 만약 다른 객체의 메서드를 그대로 호출만 한다면
→ 별도 테스트하지 않는다.
이유: 실제 기능 테스트에서 이미 확인되기 때문이다.

(우선순위) 테스트 제외 여부를 정할 때 순서:
1. 기능 결과가 달라질 수 있는가? → 테스트한다
2. 중요한 규칙이나 예외 상황이 있는가? → 테스트한다
3. 단순 반환이나 단순 호출뿐인가? → 테스트하지 않는다


### 트랜잭션 경계 기준
(If-Then) 만약 한 유스케이스 안의 일부 변경만 성공했을 때 사용자가 잘못된 예약/대기 상태를 보게 된다면
→ 그 변경들은 하나의 작업 단위로 묶는다.
이유: 사용자가 보는 도메인 상태도 경계 판단의 근거가 되기 때문이다.

(If-Then) 만약 어떤 작업이 핵심 데이터 변경 이후의 후행 작업이고 실패해도 예약/대기 상태가 깨지지 않는다면
→ 핵심 트랜잭션과 분리한다.
예: 알림, 통계, 화면 재조회

(If-Then) 만약 트랜잭션 실패가 일시적 실패라면
→ 우리는 사용자에게 다시 시도할 수 있음을 안내한다.
이유: 요청 자체가 잘못된 것이 아니라, 동시 요청·락 대기·일시적인 처리 실패처럼 타이밍이나 시스템 상태 때문에 실패한 경우이기 때문이다.

(If-Then) 만약 요구사항에서 정의한 비즈니스 예외가 발생한다면
→ 우리는 해당 예외의 의미가 사용자에게 드러나도록 응답한다.
이유: 요구사항 기반 예외는 사용자가 이해해야 하는 도메인 규칙 위반이므로, 단순히 “다시 시도하세요”로 뭉개지 않는다.

(우선순위) 트랜잭션 경계를 정할 때 순서:
1. 일부만 성공했을 때 데이터 정합성이 깨지는가?
2. 사용자가 납득하기 어려운 중간 상태를 보게 되는가?
3. 1~2가 아니라면 하나의 경계로 묶지 않는다.
