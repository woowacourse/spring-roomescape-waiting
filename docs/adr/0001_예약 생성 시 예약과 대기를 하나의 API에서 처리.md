# 예약 생성 API에서 예약과 대기를 함께 처리한다

## 상태

Accepted

## 배경

예약 기능에 예약 대기 개념이 추가되면서, 예약이 이미 존재하는 슬롯에 대해 사용자를 대기열에 등록해야 한다.

이때 API를 다음 두 방식 중 하나로 설계할 수 있다.

1. `POST /reservations` 하나로 예약과 대기 생성을 모두 처리한다.
2. `POST /reservations`와 `POST /waitlists`를 분리한다.

UI에서는 예약 가능한 슬롯과 예약 대기 가능한 슬롯을 다르게 보여줄 수 있다.
하지만 인기 있는 서비스에서는 클라이언트가 보고 있는 슬롯 상태가 항상 최신이라고 보장할 수 없다.
사용자가 예약 가능 상태를 보고 요청하더라도, 요청이 서버에 도착하는 시점에는 다른 사용자가 먼저 예약했을 수 있다.

따라서 예약 가능 여부의 최종 판단은 서버에서 이루어져야 한다.

## 결정

`POST /reservations` 하나로 예약과 예약 대기 생성을 함께 처리한다.

이 API는 확정 예약만 생성하는 API가 아니라, 특정 날짜, 시간, 테마에 대한 예약 신청 API로 해석한다.

서버는 요청을 받은 시점의 실제 상태를 기준으로 다음과 같이 처리한다.

- 해당 슬롯에 예약이 없으면 예약을 생성하고 `RESERVED` 상태를 응답한다.
- 해당 슬롯에 이미 예약이 있으면 대기를 생성하고 `WAITING` 상태와 대기 순번을 응답한다.

응답에는 예약 또는 대기의 상태를 명확히 표현한다.

```json
{
  "id": 1,
  "status": "RESERVED",
  "waitingOrder": null
}
```

또는

```json
{
  "id": 2,
  "status": "WAITING",
  "waitingOrder": 1
}
```

## 근거

사용자의 핵심 의도는 “이 슬롯을 이용하고 싶다”는 것이다.

확정 예약인지 대기 예약인지는 서버가 현재 상태에 따라 결정할 수 있다.
특히 인기 있는 서비스에서는 UI에 표시된 상태와 서버의 실제 상태가 달라질 수 있으므로,
서버가 요청 시점에 예약 가능 여부를 판단하는 것이 더 안전하다.

또한 예약이 불가능해진 경우 클라이언트가 다시 대기 신청 API를 호출하게 하면,
그 사이에 다른 사용자가 먼저 대기열에 들어갈 수 있다. 하나의 요청 안에서 예약 불가 여부를 확인하고 즉시 대기 등록까지 처리하면 사용자의 신청 시점을 더 자연스럽게 보존할 수 있다.

이 방식은 사용자 경험도 단순하게 만든다. 사용자는 하나의 요청만 보내면 되고,
서버는 결과로 예약 확정 또는 대기 등록 여부를 알려준다.

## 고려한 대안

### `POST /reservations`와 `POST /waitlists`를 분리한다

예약과 대기를 별도의 리소스로 명확히 표현할 수 있다.

- `POST /reservations`: 예약 가능한 슬롯에만 예약 생성
- `POST /waitlists`: 이미 예약된 슬롯에만 대기 생성

이 방식은 리소스 의미가 명확하고, 예약과 대기의 생명주기를 분리하기 좋다. 특히 UI에서 사용자가 “예약하기”와 “예약 대기하기”를 명시적으로 다르게 선택해야 하는 정책이라면 적합하다.

하지만 클라이언트가 본 슬롯 상태와 서버의 실제 상태가 달라질 수 있다. 예약 가능으로 보고 `POST /reservations`를 호출했는데 이미 예약된 상태라면 실패 응답을 받은 뒤 다시
`POST /waitlists`를 호출해야 한다. 이 과정에서 사용자 경험이 끊기고, 대기 순번 경쟁에서도 불리해질 수 있다.

## 결과

`POST /reservations`는 예약 신청 API로 사용한다.

클라이언트는 응답의 `status` 값을 기준으로 예약 확정과 예약 대기를 구분한다.

- `RESERVED`: 확정 예약
- `WAITING`: 예약 대기

예약과 대기는 응답 상태로 구분되며, 취소 시에는 각 리소스의 생명주기에 맞는 API를 사용한다.

- 예약 취소: `DELETE /reservations/{id}`
- 예약 대기 취소: `DELETE /waitlists/{id}`

## 주의할 점

`POST /reservations`가 실제로는 대기를 생성할 수 있으므로, API 문서에서는 이를 단순한 “예약 생성”이 아니라 “예약 신청” 또는 “예약 및 대기 신청”으로 설명해야 한다.

또한 응답의 `id`가 예약 ID인지 대기 ID인지는 반드시 `status`와 함께 해석해야 한다.

향후 정책이 바뀌어 사용자가 예약과 대기를 반드시 명시적으로 선택해야 한다면, `POST /waitlists` 분리를 다시 검토한다.
