# ADR-0006: 예약 취소 시 첫 번째 대기를 자동으로 예약 전환한다

## 상태

Proposed

## 맥락

사이클2 요구사항은 예약 대기를 예약으로 전환하는 흐름을 추가하라고 요구한다.

선택지는 크게 두 가지다.

- 예약 취소 시 대기 1번을 자동으로 예약 전환한다.
- 관리자가 대기를 수동으로 승인하거나 거절한다.

현재 프로젝트는 사용자 예약/대기 관리 흐름을 이미 가지고 있고, `WaitingLine` 도메인을 통해 같은 슬롯의 대기열, 첫 번째 대기자, 순번 계산 정책을 표현하기 시작했다.

또한 추가 요구사항으로 "함께 일어나야 하는 데이터 변경을 트랜잭션으로 묶고, 중간 실패 시 데이터 일관성이 유지되는지 테스트로 확인"해야 한다. 따라서 예약 취소와 대기 전환이 하나의 일관성 경계로 묶이는 흐름을 명확히 설계해야 한다.

## 결정

사이클2에서는 자동 전환 방식을 선택한다.

예약이 취소되면 같은 슬롯의 대기열을 조회하고, 첫 번째 대기를 예약으로 전환한다.

기본 흐름은 다음과 같다.

```text
예약 취소 요청
-> 취소 대상 예약과 슬롯 확인
-> 권한 및 과거 예약 여부 검증
-> 같은 슬롯의 대기 목록을 id 오름차순으로 조회
-> WaitingLine.first()로 첫 번째 대기 선택
-> 예약 삭제
-> 첫 번째 대기를 같은 슬롯의 예약으로 생성
-> 전환된 대기 삭제
```

위 변경은 하나의 트랜잭션으로 처리한다.

같은 슬롯에 대기가 없다면 기존처럼 예약만 취소한다.

대기 순번은 별도 컬럼으로 갱신하지 않는다. 전환된 대기가 삭제되면 남은 대기 목록을 `WaitingLine`으로 해석할 때 자연스럽게 순번이 재계산된다.

## 대안

### 대안 1: 수동 승인

장점:

- 관리자가 대기 전환 여부를 직접 제어할 수 있다.
- 승인/거절 API와 화면을 통해 운영 흐름을 명확히 보여줄 수 있다.
- 전환 조건이 복잡해져도 관리자가 판단할 여지가 있다.

단점:

- 관리자 승인 API, 거절 API, 대기 목록 관리 화면이 추가로 필요하다.
- 예약 취소 후 슬롯이 비어 있는 상태가 유지될 수 있다.
- 트랜잭션 일관성 학습 포인트가 자동 전환보다 약하다.

### 대안 2: 자동 전환

장점:

- 예약 취소와 대기 전환이 하나의 도메인 흐름으로 연결된다.
- `WaitingLine.first()`를 자연스럽게 활용할 수 있다.
- 트랜잭션 경계와 중간 실패 시 롤백 테스트를 학습하기 좋다.
- 사용자는 대기 1번이 자동으로 예약되는 자연스러운 흐름을 경험할 수 있다.

단점:

- 예약 취소 유스케이스가 단순 삭제보다 복잡해진다.
- 예약 삭제, 예약 생성, 대기 삭제의 순서와 실패 처리를 신중히 다뤄야 한다.
- 자동 전환 대상이 될 수 없는 예외 상황이 생기면 정책을 추가로 결정해야 한다.

## 근거

이번 리팩토링의 핵심 학습 목표는 도메인 책임 강화와 트랜잭션 일관성이다.

자동 전환은 예약 취소와 대기열 정책이 직접 연결되므로 `WaitingLine` 도메인을 활용하기 좋다. 또한 여러 데이터 변경이 함께 일어나야 하므로 트랜잭션 경계를 테스트로 검증하기 좋다.

수동 승인은 운영 기능으로는 의미가 있지만, 현재 프로젝트 단계에서는 관리자 UI/API가 추가되어 구현 범위가 커진다. 반면 자동 전환은 기존 사용자 예약 취소 흐름에 자연스럽게 통합된다.

## 결과

좋아지는 점:

- 대기열의 첫 번째 대기를 예약으로 전환하는 정책이 명확해진다.
- 예약 취소 유스케이스가 사이클2 요구사항의 중심 흐름을 담당한다.
- 트랜잭션 일관성 테스트를 작성할 명확한 대상이 생긴다.

감수해야 하는 점:

- `ReservationService.deleteInternal(...)`은 더 이상 단순 삭제만 담당하지 않는다.
- 같은 슬롯의 대기 목록 조회가 필요하다.
- 예약 생성과 대기 삭제가 같은 트랜잭션에 포함되어야 한다.

후속 제약:

- 예약 취소 메서드에는 `@Transactional`이 필요하다.
- 전환 대상 대기 삭제와 예약 생성 순서를 정할 때 DB unique 제약을 고려해야 한다.
- 전환 중 실패를 테스트하기 위해 Repository fake 또는 통합 테스트 전략을 정해야 한다.

## 검증 방법

추가할 테스트 후보:

- 예약을 취소하면 같은 슬롯의 첫 번째 대기가 예약으로 전환된다.
- 전환된 대기는 대기 목록에서 삭제된다.
- 기존 2번 대기는 조회 시 1번으로 보인다.
- 같은 슬롯에 대기가 없으면 예약만 취소된다.
- 다른 사용자는 내 예약을 취소할 수 없고, 자동 전환도 일어나지 않는다.
- 전환 중 실패하면 예약 취소, 예약 생성, 대기 삭제가 모두 롤백된다.

전체 테스트:

```bash
./gradlew test
```

## 열린 질문

- 전환 대상 대기자가 같은 시간대에 다른 예약을 가지고 있는 경우 전환을 막을지 결정해야 한다.
- 예약 삭제 후 예약 생성 순서와 대기 삭제 순서를 어떤 방식으로 둘지 구현 중 검증해야 한다.
- 자동 전환 결과를 API 응답에 포함할지, 기존처럼 `204 No Content`를 유지할지 결정해야 한다.
- 관리자 예약 삭제에서도 자동 전환을 동일하게 적용할지 결정해야 한다.
