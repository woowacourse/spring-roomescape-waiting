# ADR 작성을 위한 AI 프롬프트

아래 프롬프트는 프로젝트를 진행하며 생긴 설계 고민, 선택지, 판단 근거를 ADR(Architecture Decision Record) 형식으로 남기기 위한 프롬프트다.

```text
너는 시니어 소프트웨어 엔지니어이자 아키텍처 의사결정 기록(ADR) 작성자다.

내가 설명하는 설계 고민을 바탕으로 ADR(Architecture Decision Record)을 작성해줘.

목표는 단순히 "무엇을 했다"를 기록하는 것이 아니라,
"왜 이 결정을 하게 되었는지", "어떤 대안을 검토했는지", "무엇을 포기했는지"를 남기는 것이다.

다음 원칙을 반드시 지켜줘.

1. 결과만 쓰지 말고, 왜 이 결정이 필요했는지 맥락을 먼저 정리해줘.
2. 선택한 결정과 선택하지 않은 대안을 분리해줘.
3. 각 대안의 장단점을 구체적으로 비교해줘.
4. "좋다/나쁘다"가 아니라 현재 프로젝트의 요구사항, 규모, 변경 가능성, 테스트 용이성, 유지보수성 관점에서 판단해줘.
5. 결정의 긍정적 결과뿐 아니라 비용, 단점, 나중에 생길 제약도 반드시 써줘.
6. 과하게 장황하지 않게, 프로젝트 문서에 바로 넣을 수 있는 Markdown 형식으로 작성해줘.
7. 아직 확정하기 애매한 부분은 "열린 질문"으로 따로 분리해줘.
8. 추상적인 표현만 쓰지 말고, 가능하면 현재 코드 구조나 예시 이름을 사용해줘.
9. 내가 제공하지 않은 사실은 단정하지 말고, 추정이라면 추정이라고 밝혀줘.
10. 마지막에는 이 ADR을 적용했을 때 확인해야 할 테스트나 검증 방법을 제안해줘.

ADR 형식은 다음을 사용해줘.

# ADR-XXXX: [결정 제목]

## 상태

Proposed

## 맥락

[어떤 문제가 있었는지, 왜 지금 결정이 필요한지 정리]

## 결정

[무엇을 선택했는지 명확히 작성]

## 대안

[검토한 선택지들을 나열하고 각각의 장단점 정리]

## 근거

[왜 이 선택이 현재 프로젝트에 적합한지 설명]

## 결과

[좋아지는 점, 감수해야 하는 비용, 후속 제약 정리]

## 검증 방법

[테스트, 코드 리뷰 체크포인트, 운영 확인 방법 등]

## 열린 질문

[아직 결정하지 않았거나 추가 확인이 필요한 부분]

## 관련

[관련 코드, 이슈, PR, 문서, 요구사항]

내 설계 고민은 다음과 같아.

[여기에 설계 고민을 작성]
```

## 더 깊게 고민하고 싶을 때 쓰는 사전 질문 프롬프트

ADR을 바로 작성하기 전에, AI에게 먼저 질문을 뽑게 하면 결정 품질이 좋아진다.

```text
다음 설계 고민에 대해 ADR을 작성하기 전에,
내가 더 생각해야 할 질문을 먼저 뽑아줘.

질문은 다음 관점에서 구성해줘.

1. 요구사항과 비즈니스 규칙
2. 책임 위치와 객체 역할
3. 변경 가능성
4. 테스트 용이성
5. 의존성 방향
6. 구현 비용과 복잡도
7. 나중에 되돌리기 어려운 결정인지 여부

각 질문은 내가 답할 수 있도록 구체적으로 작성해줘.
질문만 나열하지 말고, 왜 그 질문이 중요한지도 한 문장씩 설명해줘.

내 설계 고민은 다음과 같아.

[여기에 설계 고민을 작성]
```

## 사용 예시

```text
내 설계 고민은 다음과 같아.

예약 생성 시 중복 예약 검증을 ReservationService에서 할지,
Reservation 도메인 객체에서 할지,
Reservations 일급 컬렉션에서 할지 고민 중이야.

중복 기준은 같은 날짜, 같은 시간, 같은 테마다.
현재는 Service에서 reservationRepository.existsByDateAndTimeAndTheme(...)로 검사하고 있다.
하지만 도메인 규칙이 Service에 모이는 것 같아서 책임 위치를 다시 고민하고 있다.
```
