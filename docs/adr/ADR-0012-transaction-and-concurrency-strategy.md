# ADR-0012: 예약 취소 및 대기 승격의 트랜잭션 범위와 동시성 제어 전략

## 상태

Proposed

## 맥락

방탈출 예약 시스템에서 사용자가 예약을 취소하면 해당 슬롯의 대기자가 자동으로 승격되는 로직이 구현되어 있다. 이 과정에서 두 가지 핵심적인 설계 고민이 발생했다.

첫째, 예약 취소와 대기 승격을 하나의 트랜잭션으로 묶을 것인가, 아니면 실패 경계를 분리하여 별도로 다룰 것인가?
둘째, 여러 사용자가 동시에 취소나 신청을 할 때 발생하는 동시성 문제를 어디까지 제어할 것인가?

## 결정

1. **원자적 트랜잭션 유지**: 예약 취소와 대기 승격 로직을 하나의 `@Transactional` 범위 내에서 원자적으로 처리한다. 승격 실패 시 예약 취소도 함께 실패하도록 유지한다.
2. **비즈니스 가치 중심의 동시성 제어**: 모든 동시성 시나리오를 방어하기보다, "금전적/운영적 손실"이 발생할 수 있는 케이스를 최우선으로 제어한다.
3. **핵심 방어 대상**: '예약 취소로 인한 승격'과 '해당 대기자의 대기 취소'가 동시에 발생하여, 사용자는 취소된 것으로 인지함에도 불구하고 시스템상으로는 예약으로 승격되는 '유령 예약' 시나리오를 집중적으로 방어한다.

## 대안

### 대안 1: 이벤트 기반의 실패 경계 분리 (Spring Events 등 사용)
- **장점**: 알림 등 부수적인 로직 실패가 핵심 비즈니스(취소)에 영향을 주지 않음. 가용성이 높아짐.
- **단점**: 구현 복잡도가 상승하고, 승격 실패 시 데이터 정합성을 맞추기 위한 보상 트랜잭션이나 재시도 로직이 추가로 필요함.

### 대안 2: 전체 시스템에 대한 엄격한 락(Lock) 적용
- **장점**: 모든 동시성 이슈로부터 안전함.
- **단점**: 성능 저하가 발생할 수 있으며, 발생 가능성이 매우 낮은 케이스까지 방어하기 위해 과도한 구현 비용(Overhead)이 발생함.

## 근거

1. **도메인 복잡도 제어**: 방탈출 도메인 특성상 승격 로직 자체의 실패 가능성은 매우 낮다. 극히 드문 실패를 위해 이벤트 기반 구조나 복잡한 예외 처리를 도입하는 것은 오버헤드다. 승격 실패로 인한 에러 발생 시 사용자에게 일반적인 에러 안내를 하는 것으로 충분하다.
2. **현실적인 리스크 관리**: 예약 취소 실패는 매출 손실로 직결되지 않지만, 사용자가 취소했다고 확신하는 대기가 예약으로 승격되어 노쇼(No-show)가 발생하는 경우 매장의 신뢰도와 금전적 손실로 이어진다. 따라서 이 구체적인 race condition을 해결하는 것에 집중하는 것이 비용 효율적이다.

## 결과

- **좋아지는 점**: 도메인 로직이 단순하게 유지되면서도 데이터 정합성이 보장된다. 가장 위험한 동시성 시나리오에 대한 안전장치를 확보할 수 있다.
- **감수해야 하는 비용**: 승격 로직에서 오류가 발생하면 사용자의 단순 '예약 취소' 요청도 함께 실패하게 된다.
- **후속 제약**: 향후 알림 서비스 등 외부 의존성이 추가될 경우, 해당 시점에는 트랜잭션 경계 분리를 재검토해야 한다.

## 검증 방법

1. **동시성 테스트 (Race Condition Simulation)**: 
   - 스레드 A: 특정 대기자의 대기 취소 시도
   - 스레드 B: 같은 슬롯의 예약 취소(승격 유발) 시도
   - 결과: 대기가 취소되었다면 승격 대상에서 제외되거나, 승격 중이라면 대기 취소가 실패해야 함을 검증한다.
2. **원자성 테스트**: 승격 로직 중간에 강제로 예외를 발생시켰을 때, 앞선 예약 삭제가 정상적으로 롤백되는지 확인한다.

## 열린 질문

- 특정 슬롯에 대해 비관적 락(Pessimistic Lock)을 사용할 것인지, 혹은 버전 정보를 이용한 낙관적 락(Optimistic Lock)을 사용할 것인지에 대한 성능 테스트가 필요한가?
- 대기 순번 재정렬 요구사항과 맞물릴 때 락의 범위가 슬롯 단위로 충분한가?
