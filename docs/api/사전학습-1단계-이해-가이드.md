# 사전학습과 1단계 이해 가이드

이 문서는 Toss Payments 미션의 **사전학습**과 **요구사항 1단계**를 처음부터 다시 이해하기 위한 가이드다.

목표는 코드를 외우는 것이 아니라, 아래 질문에 스스로 답할 수 있게 되는 것이다.

- 왜 예약을 바로 확정하지 않고 `PENDING` 상태로 저장하는가?
- 결제 인증과 결제 승인은 무엇이 다르고, 우리 서버는 어느 단계에 개입하는가?
- `orderId`, `amount`, `paymentKey`는 각각 언제 만들어지고 왜 저장하는가?
- 왜 `PaymentService`가 Toss DTO를 직접 알면 안 되는가?
- 왜 금액 검증을 Toss 승인 API 호출보다 먼저 해야 하는가?

---

## 0. 먼저 큰 그림 보기

기존 예약 서비스의 흐름은 단순했다.

```text
사용자 예약 요청
  -> 서버가 예약 생성
  -> 곧바로 예약 확정(CONFIRMED)
```

Toss Payments 미션 1단계 이후에는 흐름이 바뀐다.

```text
사용자 예약 요청
  -> 서버가 결제 대기 예약 생성(PENDING)
  -> 서버가 orderId, amount를 저장해서 응답
  -> 브라우저가 Toss 결제창을 열어 결제 인증
  -> Toss가 successUrl 또는 failUrl로 사용자를 돌려보냄
  -> 브라우저가 서버에 결제 승인 요청
  -> 서버가 저장해 둔 amount와 요청 amount를 비교
  -> 서버가 Toss 결제 승인 API 호출
  -> 성공하면 예약 확정(CONFIRMED), paymentKey 저장
```

핵심 변화는 하나다.

> 예약 생성과 예약 확정 사이에 결제 승인이 끼어든다.

그래서 이제 예약은 처음부터 확정되지 않는다. 결제 전에는 `PENDING`, 결제 승인 후에는 `CONFIRMED`가 된다.

## 1. 용어부터 정리하기

결제 미션이 어려운 이유는 코드보다 용어가 먼저 낯설기 때문이다.

### PG사

PG는 Payment Gateway의 줄임말이다. 온라인 결제를 대신 처리해주는 외부 결제 서비스다.

이 미션에서는 Toss Payments가 PG사다. 우리는 카드 번호를 직접 다루지 않고 Toss에 결제 처리를 맡긴다.

### 외부 API

우리 서버 밖에 있는 다른 서버의 API다.

Toss 결제 승인 API는 우리 코드가 아니다. 그래서 항상 다음을 고려해야 한다.

- 네트워크가 실패할 수 있다.
- 응답이 느릴 수 있다.
- Toss가 에러 코드를 내려줄 수 있다.
- 응답이 우리가 기대한 값과 다를 수 있다.
- 같은 요청을 여러 번 보내면 부작용이 생길 수 있다.

1단계에서는 특히 **성공/에러 응답 처리**와 **금액 검증**에 집중한다.

### clientKey와 secretKey

Toss에는 키가 두 종류 있다.

| 키 | 쓰는 곳 | 노출 가능 여부 | 역할 |
| --- | --- | --- | --- |
| `clientKey` | 브라우저 | 공개 가능 | Toss 결제창 SDK 초기화 |
| `secretKey` | 서버 | 절대 노출 금지 | Toss API 인증 |

브라우저에는 `clientKey`만 내려준다. `secretKey`는 서버의 설정 파일 또는 환경 변수에 둔다.

### orderId

`orderId`는 우리 서비스가 만든 주문 번호다.

이 미션에서는 예약 생성 시 서버가 `orderId`를 만든다. 이후 Toss 결제창을 열 때도 같은 `orderId`를 사용하고, 결제 승인 콜백에서도 같은 `orderId`가 돌아온다.

즉 `orderId`는 이런 역할을 한다.

```text
서버에 저장된 결제 대기 예약
  <-> Toss 인증 결과
  <-> Toss 승인 요청
```

세 대상을 연결하는 식별자다.

### amount

`amount`는 최종 결제 금액이다.

중요한 점은 브라우저에서 돌아온 `amount`를 믿으면 안 된다는 것이다. 사용자는 브라우저 요청 값을 조작할 수 있다.

그래서 서버는 예약 생성 시점에 최종 금액을 저장해 둔다.

```text
예약 생성 시 저장한 amount = 믿을 수 있는 기준값
successUrl로 돌아온 amount = 검증해야 하는 값
```

두 값이 다르면 Toss 승인 API를 호출하지 않는다.

### paymentKey

`paymentKey`는 Toss가 발급한 결제 식별자다.

결제 인증이 성공하면 Toss가 `successUrl`의 쿼리 파라미터로 내려준다. 서버는 이 값을 이용해 Toss 승인 API를 호출한다.

승인 성공 후에는 DB에 저장한다. 나중에 결제 조회나 취소가 필요할 때 사용하기 때문이다.

### 결제 요청, 인증, 승인

Toss 문서에서 가장 중요한 흐름이다.

| 단계 | 누가 주로 처리하는가 | 의미 |
| --- | --- | --- |
| 결제 요청 | 브라우저 | Toss SDK로 결제창을 연다. |
| 결제 인증 | Toss, 카드사, 사용자 | 카드 정보 입력, 본인 확인, 카드 소유자 인증을 한다. |
| 결제 승인 | 우리 서버, Toss | 인증된 결제를 실제 결제로 확정한다. |

사용자 입장에서는 인증이 끝나면 결제가 끝난 것처럼 보일 수 있다. 하지만 서버 입장에서는 아직 끝난 게 아니다.

> 우리 서버가 승인 API를 호출해야 최종 결제가 완료된다.

## 2. 사전학습은 무엇을 위한 것인가

사전학습의 목표는 Toss 결제를 완벽히 외우는 것이 아니다.

목표는 외부 API를 붙일 때 필요한 기본 감각을 익히는 것이다.

### 사전학습에서 얻어야 할 것

| 학습 | 얻어야 하는 감각 |
| --- | --- |
| Toss 결제 흐름 문서 읽기 | 브라우저와 서버의 역할 분담 |
| 샌드박스 결제 | `paymentKey`, `orderId`, `amount`가 실제로 어떻게 생기는지 확인 |
| API 콘솔 승인 호출 | 서버가 나중에 코드로 호출할 API를 손으로 먼저 호출 |
| 학습 테스트 1 | `RestClient`, `onStatus`, 에러 매핑, 금액 검증 |
| 학습 테스트 2 | 타임아웃과 재시도의 위험 |
| 학습 테스트 3 | Rate Limit과 외부 API 보호 |

요구사항 1단계에 직접 연결되는 것은 주로 **학습 테스트 1**이다. 2, 3은 이후 단계나 더 깊은 운영 관점에서 중요하다.

---

## 3. 사전학습 1: Toss 결제 흐름 이해하기

공식 문서에서 봐야 하는 핵심은 다음이다.

- 결제는 요청, 인증, 승인으로 나뉜다.
- 성공 URL에는 `paymentKey`, `orderId`, `amount`가 온다.
- 서버는 승인 전에 저장해 둔 결제 정보와 성공 URL의 값을 비교해야 한다.
- 결제 요청 전에 `orderId`와 `amount`를 서버에 저장해야 한다.

이 흐름을 우리 미션에 대입하면 다음과 같다.

```text
1. 사용자가 날짜, 테마, 시간을 선택한다.
2. 서버에 예약 생성을 요청한다.
3. 서버는 아직 예약을 확정하지 않고 PENDING 예약을 만든다.
4. 서버는 orderId와 amount를 응답한다.
5. 브라우저는 orderId와 amount로 Toss 결제창을 연다.
6. 사용자가 카드 인증을 마치면 Toss가 successUrl로 돌려보낸다.
7. 브라우저는 successUrl의 paymentKey, orderId, amount를 서버에 보낸다.
8. 서버는 orderId로 PENDING 예약을 찾는다.
9. 서버는 저장된 amount와 요청 amount를 비교한다.
10. 일치하면 Toss 승인 API를 호출한다.
11. 승인 응답이 정상이라면 예약을 CONFIRMED로 변경한다.
```

여기서 가장 중요한 선은 8~10번이다.

```text
저장된 주문 정보 확인 -> 금액 검증 -> Toss 승인 API 호출
```

순서가 바뀌면 안 된다. 금액 검증보다 Toss 승인 API 호출이 먼저 일어나면, 조작된 금액으로 외부 결제를 시도할 수 있다.

## 4. 사전학습 2: 샌드박스와 API 콘솔

샌드박스와 API 콘솔은 코드를 쓰기 전에 결제 승인 흐름을 손으로 체험하기 위한 도구다.

### 샌드박스에서 하는 일

샌드박스에서는 실제 출금 없이 테스트 결제를 진행할 수 있다.

결제 인증을 마치면 성공 URL에 이런 값들이 붙는다.

```text
paymentKey=...
orderId=...
amount=...
```

이 값들은 나중에 우리 브라우저 코드가 받게 될 값과 같은 역할을 한다.

### API 콘솔에서 하는 일

API 콘솔에서는 서버가 호출할 승인 API를 코드 없이 직접 호출해본다.

```http
POST https://api.tosspayments.com/v1/payments/confirm
```

요청 바디는 세 필드다.

```json
{
  "paymentKey": "샌드박스에서 받은 값",
  "orderId": "샌드박스에서 받은 값",
  "amount": 10000
}
```

인증에는 테스트 시크릿 키를 쓴다.

이 실습을 통해 알아야 하는 것은 이것이다.

> 서버가 코드로 할 일은 결국 `paymentKey`, `orderId`, `amount`를 들고 Toss 승인 API를 호출하는 것이다.

## 5. 사전학습 3: 학습 테스트 1과 우리 코드의 연결

학습 테스트 저장소는 `woowacourse/spring-third-party-api`다.

요구사항 1단계와 가장 직접적으로 연결되는 모듈은 다음이다.

```text
learning-test-1-api-error
```

이 모듈에서 봐야 하는 핵심 코드는 4개다.

| 학습 테스트 코드 | 역할 | 우리 프로젝트의 대응 코드 |
| --- | --- | --- |
| `PaymentService` | 저장 금액 검증 후 게이트웨이 호출 | `roomescape.payment.application.PaymentService` |
| `PaymentGateway` | 외부 결제 호출 포트 | `roomescape.payment.application.port.out.PaymentGateway` |
| `TossPaymentGateway` | Toss API 호출 어댑터 | `roomescape.payment.adapter.out.toss.TossPaymentGateway` |
| `TossClientConfig` | Basic 인증이 적용된 `RestClient` 생성 | `roomescape.payment.config.TossPaymentConfig` |

학습 테스트의 `PaymentService`는 이런 구조다.

```text
orderId로 저장된 주문을 찾는다.
저장 금액과 요청 금액을 비교한다.
다르면 예외를 던진다.
같으면 PaymentGateway를 호출한다.
```

우리 프로젝트의 `PaymentService`도 같은 원칙을 따른다.

```text
orderId로 PENDING 예약을 찾는다.
예약 소유자가 현재 로그인 사용자와 같은지 확인한다.
저장 amount와 요청 amount를 비교한다.
PENDING 상태인지 확인한다.
PaymentGateway로 Toss 승인 요청을 보낸다.
Toss 응답이 요청과 일치하는지 확인한다.
예약 상태를 CONFIRMED로 변경하고 paymentKey를 저장한다.
```

학습 테스트보다 우리 프로젝트가 더 챙기는 점은 다음이다.

- 예약 소유자 검증
- 예약 상태 검증
- Toss 승인 응답의 `paymentKey`, `orderId`, `totalAmount`, `status` 검증
- 승인 성공 후 DB 상태 변경
- 결제 실패/취소 시 PENDING 예약 정리

## 6. 요구사항 1단계가 요구하는 것

요구사항 1단계는 한 문장으로 요약할 수 있다.

> 예약 생성 흐름에 Toss 결제 승인을 끼워 넣고, 결제가 성공해야 예약을 확정한다.

이를 세부 작업으로 나누면 다음과 같다.

### 6.1 테마와 슬롯에 가격이 필요하다

결제하려면 금액이 있어야 한다.

현재 구현에서는 `theme`과 `slot`에 `price`를 둔다.

- `theme.price`: 테마의 기본 가격
- `slot.price`: 실제 예약 가능한 특정 날짜/시간 슬롯의 가격

예약은 최종적으로 `slot.price`를 사용한다.

관련 파일:

- `src/main/resources/schema.sql`
- `src/main/java/roomescape/theme/domain/Theme.java`
- `src/main/java/roomescape/slot/domain/Slot.java`
- `src/main/java/roomescape/reservation/domain/Reservation.java`

### 6.2 예약 생성은 이제 PENDING을 만든다

기존에는 예약 생성 시 바로 `CONFIRMED`였다.

하지만 결제 미션에서는 결제가 끝나기 전까지 예약이 확정되면 안 된다.

그래서 예약 생성 시 다음 정보를 저장한다.

| 필드 | 의미 |
| --- | --- |
| `status` | 처음에는 `PENDING` |
| `order_id` | 서버가 생성한 주문 번호 |
| `amount` | 서버가 계산한 최종 결제 금액 |
| `payment_key` | 승인 전에는 `null`, 승인 후 저장 |

관련 코드:

- `Reservation.createPending(...)`
- `ReservationService.save(...)`
- `JdbcReservationRepository.save(...)`

흐름은 다음과 같다.

```text
POST /api/user/reservations
  -> Slot 조회
  -> 예약 가능 여부 확인
  -> orderId 생성
  -> Reservation.createPending(...)
  -> DB 저장
  -> orderId, amount, status=PENDING 응답
```

### 6.3 브라우저는 Toss 결제창을 연다

서버가 PENDING 예약을 만들면 브라우저는 응답으로 받은 `orderId`와 `amount`를 사용해 Toss 결제창을 연다.

관련 코드:

- `client/common.html`
- `client/common.js`
- `loadPaymentClientKey()`
- `requestTossPayment(order)`

중요한 점은 브라우저가 `secretKey`를 절대 알면 안 된다는 것이다. 브라우저는 서버의 `/api/user/payments/config`에서 `clientKey`만 받아온다.

```text
GET /api/user/payments/config
  -> clientKey 응답
```

### 6.4 successUrl 이후 서버가 승인한다

결제 인증이 성공하면 Toss는 성공 URL로 사용자를 돌려보낸다.

브라우저는 URL의 쿼리 파라미터에서 다음 값을 읽고 서버에 보낸다.

```text
paymentKey
orderId
amount
```

관련 코드:

- `handlePaymentRedirect()`
- `POST /api/user/payments/confirm`
- `PaymentController.confirm(...)`
- `PaymentService.confirm(...)`

서버는 바로 Toss를 호출하지 않는다. 먼저 검증한다.

```text
1. orderId로 예약 조회
2. 현재 로그인 사용자의 예약인지 확인
3. 저장된 amount와 요청 amount 비교
4. 예약 상태가 PENDING인지 확인
5. Toss 승인 API 호출
6. Toss 응답 검증
7. reservation 상태를 CONFIRMED로 변경
```

### 6.5 failUrl은 PENDING 예약을 정리한다

결제 실패 또는 사용자의 결제 취소가 발생하면 `failUrl`로 돌아온다.

이때 결제창을 열기 전 이미 PENDING 예약이 만들어져 있을 수 있다. 그대로 두면 사용자가 결제하지 않은 예약이 남는다.

그래서 서버는 실패 콜백을 받으면 해당 사용자의 PENDING 예약을 삭제한다.

관련 코드:

- `handlePaymentRedirect()`
- `POST /api/user/payments/fail`
- `PaymentController.fail(...)`
- `PaymentService.handleFailure(...)`
- `JdbcReservationRepository.deletePendingByOrderIdAndMemberId(...)`

단, 사용자가 결제창을 바로 닫거나 취소하면 `orderId`가 없을 수 있다. 그래서 `orderId`가 비어 있으면 그냥 종료한다.

## 8. 왜 포트와 어댑터로 나눴는가

이 프로젝트는 헥사고날 아키텍처를 사용한다.

결제 기능도 같은 원칙을 따른다.

```text
PaymentService
  -> PaymentGateway 인터페이스
      -> TossPaymentGateway 구현체
```

`PaymentService`는 Toss를 모른다.

`PaymentService`가 아는 것은 이것뿐이다.

```text
PaymentGateway.confirm(PaymentConfirmation)
  -> PaymentResult
```

Toss 전용 정보는 어댑터 안에 갇힌다.

- Toss 요청 DTO: `TossConfirmRequest`
- Toss 성공 응답 DTO: `TossPaymentResponse`
- Toss 에러 응답 DTO: `TossErrorResponse`
- Toss 에러 코드 매핑: `TossPaymentGateway.toDomainException(...)`

이렇게 나누면 나중에 PG사를 바꾸더라도 `PaymentService`의 핵심 흐름은 유지할 수 있다.

```text
TossPaymentGateway 제거
NaverPaymentGateway 추가
PaymentService는 그대로
```

물론 지금 미션 규모에서는 구조가 다소 무겁게 느껴질 수 있다. 그래도 이 미션의 학습 목표가 “외부 API를 우리 도메인 밖에 격리한다”에 있기 때문에, 포트와 어댑터는 의미가 있다.

## 9. 에러 처리를 어떻게 이해하면 좋은가

외부 API 에러는 단순히 `500`으로 뭉개면 안 된다.

예를 들어 Toss가 이런 에러를 줄 수 있다.

| Toss 에러 | 우리 서비스의 해석 |
| --- | --- |
| `ALREADY_PROCESSED_PAYMENT` | 이미 처리된 결제 |
| `REJECT_CARD_PAYMENT` | 카드 결제 거절 |
| `UNAUTHORIZED_KEY` | 서버 시크릿 키 설정 오류 |
| `NOT_FOUND_PAYMENT` | Toss가 해당 결제를 찾지 못함 |
| `FAILED_PAYMENT_INTERNAL_SYSTEM_PROCESSING` | Toss 내부 오류, 재시도 고려 가능 |

우리 코드는 이 값을 `ErrorCode`로 바꾼다.

```text
TossErrorResponse(code, message)
  -> TossPaymentGateway.toDomainException(...)
  -> EscapeRoomException(ErrorCode...)
  -> 우리 API 에러 응답
```

중요한 점은 Toss의 에러 DTO가 서비스 계층 밖으로 새지 않는다는 것이다.

`PaymentService`는 Toss의 `code` 문자열을 몰라도 된다. 이 번역은 어댑터의 책임이다.

---

## 10. `ALREADY_PROCESSED_PAYMENT`는 왜 조심해야 하는가

같은 `paymentKey`로 승인 API를 두 번 호출하면 Toss는 이미 처리된 결제라고 응답할 수 있다.

여기서 생각이 갈릴 수 있다.

```text
이미 처리됨 = 성공으로 봐도 될까?
이미 처리됨 = 위험한 상태로 봐야 할까?
```

이번 구현에서는 이렇게 처리한다.

### 이미 우리 예약이 `CONFIRMED`이고 paymentKey도 같다면

이미 같은 결제가 반영된 상태다. 새로고침이나 중복 요청일 수 있으므로 기존 확정 응답을 돌려줄 수 있다.

### 아직 우리 예약이 `PENDING`인데 Toss가 이미 처리됐다고 하면

우리 DB에는 아직 확정되지 않았는데 Toss는 이미 처리됐다고 말하는 상태다.

이 경우를 무작정 성공으로 바꾸면 위험하다. 예를 들어 다른 주문과 섞였거나, 이전 요청이 중간에 실패했거나, 서버 상태와 Toss 상태가 어긋난 상황일 수 있다.

그래서 PENDING 상태에서 `ALREADY_PROCESSED_PAYMENT`가 오면 안전하게 실패로 본다.

## 13. 헷갈릴 때 보는 체크리스트

### 결제 인증과 승인이 헷갈릴 때

- 인증은 사용자와 카드사가 한다.
- 승인은 서버가 Toss API를 호출해서 한다.
- 인증이 끝나면 `paymentKey`가 생긴다.
- 승인이 끝나야 실제 결제가 완료된다.

### orderId와 paymentKey가 헷갈릴 때

- `orderId`: 우리 서버가 만든 주문 번호
- `paymentKey`: Toss가 만든 결제 번호
- `orderId`는 인증 전부터 있다.
- `paymentKey`는 인증 성공 후 생긴다.

### amount 검증이 헷갈릴 때

- 브라우저에서 온 `amount`는 믿지 않는다.
- DB에 저장한 `amount`를 기준으로 삼는다.
- 다르면 Toss를 호출하지 않는다.

### clientKey와 secretKey가 헷갈릴 때

- 브라우저는 `clientKey`만 사용한다.
- 서버는 `secretKey`로 Toss API를 호출한다.
- `secretKey`가 브라우저에 노출되면 안 된다.

### 포트와 어댑터가 헷갈릴 때

- 포트는 우리 서비스가 기대하는 인터페이스다.
- 어댑터는 특정 외부 시스템에 맞춘 구현이다.
- `PaymentService`는 포트를 본다.
- `TossPaymentGateway`는 Toss를 본다.

### 14.2 가격 정보: 결제 금액의 기준 만들기

결제하려면 먼저 “얼마를 결제해야 하는지”가 서버에 있어야 한다.

그래서 DB에 가격 컬럼이 추가됐다.

```sql
CREATE TABLE theme
(
    ...
    price INT NOT NULL DEFAULT 10000,
    ...
);

CREATE TABLE slot
(
    ...
    price INT NOT NULL DEFAULT 10000,
    ...
);
```

관련 파일:

- `src/main/resources/schema.sql`
- `src/main/java/roomescape/theme/domain/Theme.java`
- `src/main/java/roomescape/slot/domain/Slot.java`

`Theme`에는 기본 가격이 있다.

```java
public class Theme {
    public static final int DEFAULT_PRICE = 10_000;

    private final int price;

    public Theme(Long id, String name, String description, String thumbnailUrl) {
        this(id, name, description, thumbnailUrl, DEFAULT_PRICE);
    }
}
```

이 코드는 기존 테스트와 기존 생성 흐름을 깨지 않기 위한 완충 장치다.

기존 코드에는 `Theme(id, name, description, thumbnailUrl)` 생성자가 많이 쓰이고 있었다. 가격 필드를 갑자기 필수로 바꾸면 기존 코드가 전부 깨진다. 그래서 기존 생성자는 유지하되 기본 가격 `10_000`을 넣도록 했다.

`Slot`은 실제 예약 가능한 날짜/시간/테마 조합이다.

```java
public class Slot {
    private final int price;

    public static Slot create(LocalDate date, ReservationTime time, Theme theme) {
        return new Slot(null, date, time, theme, theme.getPrice());
    }

    public static Slot of(Long id, LocalDate date, ReservationTime time, Theme theme, int price) {
        return new Slot(id, date, time, theme, price);
    }
}
```

여기서 중요한 결정은 예약 금액을 `theme.price`가 아니라 최종적으로 `slot.price`에서 가져온다는 점이다.

왜 그럴까?

테마는 기본 상품이고, 슬롯은 특정 날짜와 특정 시간에 실제로 판매되는 상품에 가깝다. 지금은 `theme.price`와 `slot.price`가 같아도, 나중에는 주말/야간/이벤트에 따라 슬롯별 가격이 달라질 수 있다.

그래서 예약은 `slot.price`를 최종 결제 금액으로 사용한다.

```java
public static Reservation createPending(long memberId, Slot slot, String orderId) {
    return new Reservation(null, memberId, slot, ReservationStatus.PENDING, orderId, slot.getPrice(), null);
}
```

리뷰 포인트:

- 결제 금액은 브라우저가 정하지 않는다.
- 결제 금액은 서버 도메인인 `Slot`에서 나온다.
- `theme.price`는 기본값, `slot.price`는 실제 예약 금액으로 이해하면 된다.

### 14.3 예약 생성: 결제 전 주문 정보를 저장한다

요구사항의 핵심은 “결제 요청 전에 주문 정보를 저장하라”이다.

이 요구사항은 `ReservationService.save(...)`에서 시작된다.

```java
public ReservationSaveResponse save(ReservationSaveRequest body, long memberId) {
    Slot slot = slotAssembler.assembleExisting(body.date(), body.timeId(), body.themeId());
    throwIfSlotUnavailableForReservation(slot.getId());
    Reservation reservation = reservationRepository.save(
            Reservation.createPending(memberId, slot, orderIdGenerator.generate())
    );

    return ReservationSaveResponse.from(reservation);
}
```

기존에는 예약 생성이 곧 예약 확정이었다.

1단계 이후에는 예약 생성이 “결제 대기 주문 생성”으로 바뀐다.

```text
ReservationService.save(...)
  -> Slot 조회
  -> 예약 가능 여부 확인
  -> orderId 생성
  -> Reservation.createPending(...)
  -> reservation 테이블에 저장
```

`OrderIdGenerator`는 서버가 주문번호를 만들기 위해 추가됐다.

```java
public class OrderIdGenerator {

    public String generate() {
        return "order_" + UUID.randomUUID().toString().replace("-", "");
    }
}
```

`orderId`를 서버가 만드는 이유는 클라이언트가 주문번호를 마음대로 정하지 못하게 하기 위해서다.

물론 `orderId` 자체가 보안 토큰은 아니다. 사용자가 URL에서 볼 수 있다. 그래서 뒤에서 반드시 로그인 사용자 소유권 검증도 함께 한다.

`Reservation`에는 결제에 필요한 필드가 추가됐다.

```java
public class Reservation {
    private final ReservationStatus status;
    private final String orderId;
    private final int amount;
    private final String paymentKey;
}
```

각 필드의 의미는 다음과 같다.

| 필드 | 저장 시점 | 의미 |
| --- | --- | --- |
| `status` | 예약 생성 시 | 처음에는 `PENDING`, 승인 후 `CONFIRMED` |
| `orderId` | 예약 생성 시 | 우리 서버가 만든 주문번호 |
| `amount` | 예약 생성 시 | 서버가 계산한 최종 결제 금액 |
| `paymentKey` | 승인 성공 후 | Toss가 발급한 결제 식별자 |

DB 저장도 이에 맞게 바뀐다.

```java
INSERT INTO reservation(member_id, slot_id, status, order_id, amount, payment_key)
VALUES (:memberId, :slotId, :status, :orderId, :amount, :paymentKey)
```

리뷰 포인트:

- 예약 생성 API가 더 이상 확정 예약을 만들지 않는다.
- 결제 전에 `orderId`와 `amount`가 DB에 저장된다.
- 이후 successUrl로 돌아온 값은 이 DB 값과 비교된다.

### 14.4 예약 생성 응답: 브라우저가 결제창을 열 수 있게 한다

브라우저가 Toss 결제창을 열려면 서버에서 받은 주문 정보가 필요하다.

그래서 `ReservationSaveResponse`에 결제창용 값이 추가됐다.

```java
public record ReservationSaveResponse(
        Long id,
        Long memberId,
        Long slotId,
        String orderId,
        int amount,
        ReservationStatus status,
        String orderName
) {
}
```

`orderName`은 Toss 결제창에 표시할 주문 이름이다.

```java
private static String buildOrderName(Reservation reservation) {
    return reservation.getSlot().getTheme().getName() + " " + reservation.getSlot().getStartAt();
}
```

즉 예약 생성 응답은 이제 단순히 “예약 ID”만 주는 것이 아니다.

```text
예약 ID
회원 ID
슬롯 ID
주문번호(orderId)
결제금액(amount)
예약상태(PENDING)
주문명(orderName)
```

브라우저는 이 응답을 받아 `requestPayment(...)`를 호출한다.

리뷰 포인트:

- `orderId`, `amount`는 Toss 결제창 요청에 그대로 쓰인다.
- 하지만 서버는 나중에 이 값을 다시 검증한다.
- 응답에 담긴 값은 “결제창을 열기 위한 값”이지 “최종 승인된 결제 결과”가 아니다.

### 14.5 브라우저 코드: 인증은 프론트에서, 승인은 서버에서

브라우저는 Toss SDK를 로드한다.

```html
<script src="https://js.tosspayments.com/v2/standard"></script>
<script src="./common.js"></script>
```

관련 파일:

- `client/common.html`
- `client/common.js`

브라우저는 서버에서 `clientKey`를 받아온다.

```javascript
async function loadPaymentClientKey() {
  if (paymentClientKey) return paymentClientKey;
  const config = await api("/user/payments/config");
  paymentClientKey = config.clientKey;
  return paymentClientKey;
}
```

`clientKey`를 서버 API로 내려주는 컨트롤러는 다음이다.

```java
@GetMapping("/config")
public ResponseEntity<ApiResponse<PaymentConfigResponse>> config() {
    return ResponseEntity.ok(ApiResponse.success(new PaymentConfigResponse(tossPaymentProperties.clientKey())));
}
```

브라우저가 결제창을 여는 코드는 `requestTossPayment(...)`다.

```javascript
async function requestTossPayment(order) {
  const clientKey = await loadPaymentClientKey();
  const tossPayments = window.TossPayments(clientKey);
  const payment = tossPayments.payment({ customerKey: "ANONYMOUS" });
  await payment.requestPayment({
    method: "CARD",
    amount: {
      value: order.amount,
      currency: "KRW",
    },
    orderId: order.orderId,
    orderName: order.orderName || `${selectedThemeName} ${selectedTimeLabel}`,
    customerName: currentUserLabel(),
    successUrl: paymentRedirectUrl("success"),
    failUrl: paymentRedirectUrl("fail"),
  });
}
```

이 코드가 하는 일은 “결제 인증”이다.

여기서 실제 승인 API를 호출하지 않는다. 브라우저는 카드 정보를 입력하고 인증을 마친 뒤, Toss가 돌려준 결과를 서버에 다시 넘기는 역할만 한다.

예약 버튼 클릭 흐름은 이렇게 이어진다.

```javascript
reservationOrder = await api("/user/reservations", {
  method: "POST",
  body: JSON.stringify({
    date,
    timeId: selectedTimeId,
    themeId: selectedThemeId,
    storeId: 1,
  }),
});
setStatus("결제창으로 이동합니다.");
await requestTossPayment(reservationOrder);
```

리뷰 포인트:

- 브라우저는 `clientKey`만 사용한다.
- 브라우저는 `secretKey`를 모른다.
- 브라우저는 결제를 확정하지 않는다.
- 브라우저는 Toss 인증 결과를 서버 승인 API로 전달한다.

### 14.6 successUrl 처리: 인증 결과를 서버 승인 요청으로 바꾼다

Toss 인증이 성공하면 브라우저는 `successUrl`로 돌아온다.

이때 URL에는 `paymentKey`, `orderId`, `amount`가 있다.

`common.js`는 이 값을 읽어서 우리 서버에 승인 요청을 보낸다.

```javascript
if (result === "success") {
  const response = await api("/user/payments/confirm", {
    method: "POST",
    body: JSON.stringify({
      paymentKey: params.get("paymentKey"),
      orderId: params.get("orderId"),
      amount: Number(params.get("amount")),
    }),
  });
}
```

이 코드는 Toss 승인 API를 직접 호출하지 않는다.

브라우저가 호출하는 것은 우리 서버다.

```text
브라우저
  -> POST /api/user/payments/confirm
      -> 우리 서버
          -> Toss 승인 API
```

이렇게 하는 이유는 Toss 승인 API에는 `secretKey`가 필요하기 때문이다. `secretKey`는 브라우저에 두면 안 된다.

리뷰 포인트:

- successUrl의 쿼리 파라미터는 검증 대상이다.
- 브라우저는 우리 서버에 승인 요청만 보낸다.
- 최종 승인 여부는 서버가 결정한다.

### 14.7 결제 컨트롤러: 웹 요청을 유스케이스로 넘긴다

결제 컨트롤러는 세 엔드포인트를 제공한다.

```java
@RestController
@RequestMapping("/api/user/payments")
public class PaymentController {

    @GetMapping("/config")
    public ResponseEntity<ApiResponse<PaymentConfigResponse>> config() { ... }

    @PostMapping("/confirm")
    public ResponseEntity<ApiResponse<PaymentConfirmResponse>> confirm(
            @RequestBody @Valid PaymentConfirmRequest request,
            @LoginMember AuthenticatedMember member
    ) { ... }

    @PostMapping("/fail")
    public ResponseEntity<ApiResponse<Void>> fail(
            @RequestBody PaymentFailRequest request,
            @LoginMember AuthenticatedMember member
    ) { ... }
}
```

여기서 가장 중요한 부분은 `@LoginMember AuthenticatedMember member`다.

결제 콜백은 URL에 `orderId`가 있으므로, 다른 사용자가 어떤 식으로든 `orderId`를 알아낼 가능성을 완전히 배제할 수 없다.

그래서 서버는 “이 `orderId`가 현재 로그인 사용자의 예약인가?”를 확인해야 한다.

컨트롤러는 로그인 사용자 ID를 서비스로 넘긴다.

```java
PaymentConfirmResponse response = confirmPaymentUseCase.confirm(request, member.id());
```

리뷰 포인트:

- 컨트롤러는 HTTP 요청/응답 변환을 담당한다.
- 결제 확정 규칙은 컨트롤러가 아니라 `PaymentService`에 있다.
- 로그인 사용자 ID를 서비스에 넘겨 소유권 검증이 가능하게 한다.

### 14.8 PaymentService: 1단계의 핵심 비즈니스 규칙

`PaymentService.confirm(...)`은 1단계에서 가장 중요한 코드다.

전체 흐름을 먼저 보자.

```java
@Transactional
public PaymentConfirmResponse confirm(PaymentConfirmRequest request, long memberId) {
    Reservation reservation = findReservationByOrderId(request.orderId());
    reservation.validateOwnedBy(memberId);
    validateAmount(reservation, request.amount());

    if (reservation.isConfirmedWith(request.paymentKey())) {
        return toConfirmResponse(reservation);
    }
    validatePending(reservation);

    PaymentResult result = paymentGateway.confirm(
            new PaymentConfirmation(request.paymentKey(), request.orderId(), request.amount())
    );
    validateGatewayResult(request, result);

    boolean confirmed = reservationRepository.confirmPayment(
            reservation.getId(),
            reservation.getOrderId(),
            request.paymentKey()
    );
    if (!confirmed) {
        throw new EscapeRoomException(ErrorCode.PAYMENT_NOT_PENDING, reservation.getOrderId());
    }

    return new PaymentConfirmResponse(...);
}
```

이 메서드는 아래 요구사항들을 한 번에 만족한다.

#### 1. orderId로 저장된 주문을 찾는다

```java
private Reservation findReservationByOrderId(String orderId) {
    return reservationRepository.findByOrderId(orderId)
            .orElseThrow(() -> new EscapeRoomException(ErrorCode.PAYMENT_NOT_FOUND, orderId));
}
```

Toss에서 돌아온 `orderId`를 기준으로, 결제 전에 저장해 둔 PENDING 예약을 찾는다.

#### 2. 현재 사용자의 예약인지 확인한다

```java
reservation.validateOwnedBy(memberId);
```

이 검증이 없으면 다른 사용자의 `orderId`로 결제 승인 또는 실패 처리를 시도할 수 있다.

#### 3. 금액을 검증한다

```java
private void validateAmount(Reservation reservation, int requestedAmount) {
    if (reservation.getAmount() != requestedAmount) {
        throw new EscapeRoomException(ErrorCode.PAYMENT_AMOUNT_MISMATCH, reservation.getAmount(), requestedAmount);
    }
}
```

이 검증은 Toss 승인 API 호출보다 먼저 일어난다.

이 순서가 중요하다.

```text
금액 불일치 발견
  -> 예외 발생
  -> paymentGateway.confirm(...) 호출 안 함
```

그래야 조작된 금액으로 외부 결제 승인을 시도하지 않는다.

#### 4. 이미 같은 paymentKey로 확정된 요청은 중복 성공으로 처리한다

```java
if (reservation.isConfirmedWith(request.paymentKey())) {
    return toConfirmResponse(reservation);
}
```

새로고침이나 네트워크 재시도로 같은 승인 요청이 다시 들어올 수 있다.

우리 DB가 이미 같은 `paymentKey`로 `CONFIRMED`라면 기존 성공 결과를 반환해도 된다.

#### 5. 아직 PENDING인지 확인한다

```java
private void validatePending(Reservation reservation) {
    if (!reservation.isPending()) {
        throw new EscapeRoomException(ErrorCode.PAYMENT_NOT_PENDING, reservation.getOrderId());
    }
}
```

이미 다른 결제로 확정됐거나 취소된 예약이면 승인하면 안 된다.

#### 6. Toss 승인 API를 호출한다

```java
PaymentResult result = paymentGateway.confirm(
        new PaymentConfirmation(request.paymentKey(), request.orderId(), request.amount())
);
```

`PaymentService`는 Toss를 직접 모른다. `PaymentGateway`라는 포트만 안다.

#### 7. Toss 응답도 검증한다

```java
private void validateGatewayResult(PaymentConfirmRequest request, PaymentResult result) {
    boolean matchesRequest = request.paymentKey().equals(result.paymentKey())
            && request.orderId().equals(result.orderId())
            && request.amount() == result.totalAmount()
            && "DONE".equals(result.status());

    if (!matchesRequest) {
        throw new EscapeRoomException(ErrorCode.PAYMENT_GATEWAY_ERROR);
    }
}
```

외부 API 응답도 무조건 믿지 않는다.

요청한 `paymentKey`, `orderId`, `amount`와 Toss 응답이 일치하고, 상태가 `DONE`일 때만 우리 예약을 확정한다.

#### 8. DB 상태를 원자적으로 CONFIRMED로 바꾼다

```java
boolean confirmed = reservationRepository.confirmPayment(
        reservation.getId(),
        reservation.getOrderId(),
        request.paymentKey()
);
```

실제 SQL은 다음 조건을 건다.

```sql
UPDATE reservation
SET status = 'CONFIRMED', payment_key = :paymentKey
WHERE id = :reservationId
AND order_id = :orderId
AND status = 'PENDING'
```

`status = 'PENDING'` 조건이 중요하다.

동시에 같은 주문을 승인하려는 요청이 들어오더라도, 이미 확정된 예약은 다시 업데이트되지 않는다.

리뷰 포인트:

- `PaymentService`는 1단계 비즈니스 규칙의 중심이다.
- 검증 순서가 중요하다.
- Toss 호출 전 검증과 Toss 호출 후 검증이 모두 있다.
- 최종 DB 업데이트는 `PENDING` 조건을 걸어 안전하게 수행한다.

### 14.9 TossPaymentConfig: 서버가 secretKey로 인증한다

Toss 승인 API는 서버에서 호출한다.

그래서 서버는 `secretKey`로 Basic 인증 헤더를 만든다.

```java
@Bean
public RestClient tossRestClient(TossPaymentProperties properties) {
    String credentials = properties.secretKey() + ":";
    String basicToken = Base64.getEncoder()
            .encodeToString(credentials.getBytes(StandardCharsets.UTF_8));

    return RestClient.builder()
            .baseUrl(properties.baseUrl())
            .defaultHeader(HttpHeaders.AUTHORIZATION, "Basic " + basicToken)
            .build();
}
```

설정값은 `application.yml`에서 외부화했다.

```yaml
payment:
  toss:
    base-url: https://api.tosspayments.com
    secret-key: ${TOSS_SECRET_KEY:test_sk_placeholder}
    client-key: ${TOSS_CLIENT_KEY:test_ck_placeholder}
```

왜 `secretKey + ":"`를 Base64로 인코딩할까?

Toss API 인증은 Basic 인증 방식을 사용한다. Basic 인증은 일반적으로 `username:password` 형태를 Base64로 인코딩한다. Toss에서는 시크릿 키를 username처럼 사용하고 password는 비워두므로 `secretKey + ":"` 형태가 된다.

리뷰 포인트:

- `secretKey`는 브라우저에 내려가지 않는다.
- Toss API 호출용 `RestClient`에 기본 Authorization 헤더가 설정된다.
- 환경 변수로 실제 키를 주입할 수 있다.

### 14.10 TossPaymentGateway: 외부 API 호출과 에러 번역

`TossPaymentGateway`는 `PaymentGateway` 포트의 Toss 구현체다.

```java
@Component
public class TossPaymentGateway implements PaymentGateway {

    @Override
    public PaymentResult confirm(PaymentConfirmation confirmation) {
        TossConfirmRequest request = new TossConfirmRequest(
                confirmation.paymentKey(),
                confirmation.orderId(),
                confirmation.amount()
        );

        TossPaymentResponse response = tossRestClient.post()
                .uri("/v1/payments/confirm")
                .contentType(MediaType.APPLICATION_JSON)
                .body(request)
                .retrieve()
                .onStatus(HttpStatusCode::isError, (req, res) -> {
                    TossErrorResponse error = parseError(res.getBody());
                    throw toDomainException(error);
                })
                .body(TossPaymentResponse.class);

        return toResult(response);
    }
}
```

이 코드에는 책임이 세 가지 있다.

#### 1. 우리 도메인 요청을 Toss 요청으로 바꾼다

```text
PaymentConfirmation
  -> TossConfirmRequest
```

`PaymentConfirmation`은 우리 서비스 안쪽 모델이다.

`TossConfirmRequest`는 Toss API에 맞춘 바깥쪽 DTO다.

#### 2. Toss 승인 API를 호출한다

```text
POST /v1/payments/confirm
Content-Type: application/json
Authorization: Basic ...
```

요청 바디는 세 필드다.

```java
public record TossConfirmRequest(
        String paymentKey,
        String orderId,
        int amount
) {
}
```

#### 3. Toss 응답을 우리 도메인 결과로 바꾼다

```java
private PaymentResult toResult(TossPaymentResponse response) {
    return new PaymentResult(
            response.paymentKey(),
            response.orderId(),
            response.status(),
            response.totalAmount()
    );
}
```

Toss 응답 DTO도 서비스 계층으로 직접 넘기지 않는다.

```text
TossPaymentResponse
  -> PaymentResult
```

리뷰 포인트:

- Toss 전용 DTO는 어댑터 밖으로 나가지 않는다.
- `PaymentService`는 `TossConfirmRequest`나 `TossPaymentResponse`를 모른다.
- 외부 API 포맷 변경의 영향 범위를 어댑터로 좁힌다.

---

### 14.11 ErrorCode와 에러 매핑: 외부 에러를 우리 언어로 바꾼다

Toss는 에러가 발생하면 대략 이런 형태의 응답을 준다.

```json
{
  "code": "REJECT_CARD_PAYMENT",
  "message": "카드 결제가 거절되었습니다."
}
```

우리 서비스는 이 문자열을 그대로 밖으로 흘리지 않고 `ErrorCode`로 바꾼다.

```java
private EscapeRoomException toDomainException(TossErrorResponse error) {
    ErrorCode errorCode = switch (error.code()) {
        case "ALREADY_PROCESSED_PAYMENT" -> ErrorCode.PAYMENT_ALREADY_PROCESSED;
        case "DUPLICATED_ORDER_ID", "NOT_FOUND_PAYMENT_SESSION", "INVALID_REQUEST" ->
                ErrorCode.PAYMENT_INVALID_REQUEST;
        case "UNAUTHORIZED_KEY", "INVALID_API_KEY" -> ErrorCode.PAYMENT_GATEWAY_CONFIG_ERROR;
        case "REJECT_CARD_PAYMENT" -> ErrorCode.PAYMENT_CARD_REJECTED;
        case "NOT_FOUND_PAYMENT" -> ErrorCode.PAYMENT_NOT_FOUND;
        case "FAILED_PAYMENT_INTERNAL_SYSTEM_PROCESSING" -> ErrorCode.PAYMENT_GATEWAY_RETRYABLE;
        default -> ErrorCode.PAYMENT_GATEWAY_ERROR;
    };

    return new EscapeRoomException(errorCode);
}
```

관련 `ErrorCode`는 다음과 같이 추가됐다.

```java
PAYMENT_AMOUNT_MISMATCH(...)
PAYMENT_NOT_PENDING(...)
PAYMENT_ALREADY_PROCESSED(...)
PAYMENT_INVALID_REQUEST(...)
PAYMENT_GATEWAY_CONFIG_ERROR(...)
PAYMENT_CARD_REJECTED(...)
PAYMENT_GATEWAY_RETRYABLE(...)
PAYMENT_GATEWAY_ERROR(...)
```

이렇게 하는 이유는 두 가지다.

첫째, Toss의 에러 코드를 우리 API 응답 정책에 맞게 바꾸기 위해서다.

둘째, 나중에 Toss가 아닌 다른 PG사로 바뀌어도 서비스 전체가 Toss 에러 코드에 오염되지 않게 하기 위해서다.

리뷰 포인트:

- 외부 에러 코드는 어댑터에서 해석한다.
- 사용자에게 보여줄 수 있는 에러와 운영자가 봐야 하는 에러를 구분한다.
- 모르는 Toss 에러는 기본 게이트웨이 에러로 떨어진다.

### 14.12 failUrl 처리: 실패한 결제 대기 예약을 정리한다

결제창을 열기 전에 서버는 이미 PENDING 예약을 만들어 둔다.

따라서 사용자가 결제를 취소하거나 실패하면 이 PENDING 예약을 정리해야 한다.

브라우저는 failUrl로 돌아오면 서버에 실패 처리를 요청한다.

```javascript
await api("/user/payments/fail", {
  method: "POST",
  body: JSON.stringify({
    code: params.get("code"),
    message: params.get("message"),
    orderId: params.get("orderId"),
  }),
});
```

결제창 자체를 열다가 실패한 경우에도 정리 요청을 보낸다.

```javascript
if (reservationOrder?.orderId) {
  await api("/user/payments/fail", {
    method: "POST",
    body: JSON.stringify({
      code: "PAYMENT_WINDOW_FAILED",
      message: error.message,
      orderId: reservationOrder.orderId,
    }),
  });
}
```

서버의 실패 처리 로직은 간단하다.

```java
@Transactional
public void handleFailure(PaymentFailRequest request, long memberId) {
    if (!StringUtils.hasText(request.orderId())) {
        return;
    }
    reservationRepository.deletePendingByOrderIdAndMemberId(request.orderId(), memberId);
}
```

`orderId`가 없으면 그냥 종료한다.

Toss의 사용자 취소 흐름에서는 `orderId`가 없을 수 있기 때문이다. 이때 무리하게 삭제하려고 하면 NPE나 잘못된 삭제가 발생할 수 있다.

삭제 SQL에는 사용자 ID와 PENDING 조건이 함께 들어간다.

```sql
DELETE FROM reservation
WHERE order_id = :orderId
AND member_id = :memberId
AND status = 'PENDING'
```

리뷰 포인트:

- 결제 실패는 예약 실패와 연결된다.
- 하지만 모든 failUrl에 `orderId`가 있다고 가정하면 안 된다.
- 다른 사용자의 PENDING 예약을 삭제하면 안 된다.
- 이미 확정된 예약은 failUrl로 삭제되면 안 된다.

### 14.13 테스트 코드: 무엇을 안전망으로 잡았는가

이번 1단계에서 추가된 테스트는 세 종류다.

```text
PaymentServiceTest
TossPaymentGatewayTest
PaymentApiIntegrationTest
```

#### PaymentServiceTest

서비스 규칙을 검증한다.

```java
@DisplayName("저장 금액과 요청 금액이 다르면 게이트웨이를 호출하기 전에 차단한다.")
void amount_mismatch_blocks_before_gateway_call() {
    ...
    verify(paymentGateway, never()).confirm(any());
    verify(reservationRepository, never()).confirmPayment(anyLong(), any(), any());
}
```

이 테스트가 중요한 이유는 “금액 검증이 Toss 호출보다 먼저”라는 요구사항을 증명하기 때문이다.

또 다른 핵심 테스트는 게이트웨이 응답 검증이다.

```java
@DisplayName("게이트웨이 승인 응답이 요청 값과 다르면 예약을 확정하지 않는다.")
void gateway_result_mismatch_fails() {
    ...
    verify(reservationRepository, never()).confirmPayment(anyLong(), any(), any());
}
```

외부 응답도 무조건 믿지 않는다는 규칙을 잡아준다.

#### TossPaymentGatewayTest

Toss API 호출 모양을 검증한다.

```java
server.expect(requestTo(BASE_URL + "/v1/payments/confirm"))
        .andExpect(method(HttpMethod.POST))
        .andExpect(header(HttpHeaders.AUTHORIZATION, "Basic " + BASIC_TOKEN))
        .andExpect(content().json("""
                {
                  "paymentKey": "payment-key",
                  "orderId": "order_123456",
                  "amount": 15000
                }
                """));
```

이 테스트는 다음을 보장한다.

- URL이 맞다.
- HTTP 메서드가 `POST`다.
- Basic 인증 헤더가 있다.
- JSON 바디에 `paymentKey`, `orderId`, `amount`가 있다.

또한 Toss 에러 코드 매핑도 검증한다.

```java
expectError("REJECT_CARD_PAYMENT", withForbiddenRequest());
assertMappedError(ErrorCode.PAYMENT_CARD_REJECTED);
```

#### PaymentApiIntegrationTest

사용자 흐름 전체를 검증한다.

```java
@DisplayName("예약 생성은 결제 대기 주문을 만들고 승인 성공 시 예약을 확정한다.")
void creates_pending_reservation_and_confirms_payment() {
    String accessToken = loginUserToken();
    Map<String, Object> order = createReservationOrder(accessToken);
    ...
    post("/api/user/payments/confirm")
    ...
    get("/api/user/reservations/me")
}
```

이 테스트는 “예약 생성 -> PENDING -> 결제 승인 -> CONFIRMED” 흐름을 실제 API 수준에서 확인한다.

보안상 중요한 테스트도 있다.

```java
@DisplayName("다른 사용자의 결제 대기 주문은 승인할 수 없다.")
void other_member_cannot_confirm_payment() { ... }

@DisplayName("다른 사용자의 결제 실패 콜백은 대기 주문을 삭제하지 않는다.")
void other_member_cannot_delete_pending_payment_on_failure() { ... }
```

이 두 테스트는 `orderId`를 알아도 남의 예약을 승인하거나 삭제할 수 없다는 점을 확인한다.

리뷰 포인트:

- 테스트는 단순히 성공 케이스만 보지 않는다.
- 금액 조작, 외부 응답 불일치, 중복 처리, 다른 사용자 접근, failUrl 예외 상황을 함께 본다.
- 외부 API 호출은 실제 Toss가 아니라 mock/stub으로 검증한다.

### 14.15 이 구현의 철학을 한 문장으로 정리하기

이번 1단계 구현의 철학은 다음이다.

> 브라우저와 외부 PG에서 돌아온 값은 검증 대상이고, 우리 서버가 결제 기준값과 최종 예약 상태를 책임진다.

그래서 코드가 다음 방향으로 구성됐다.

- 금액 기준값은 서버 DB에 저장한다.
- 브라우저는 결제 인증까지만 맡는다.
- 서버는 승인 전에 저장값과 요청값을 비교한다.
- 서버는 Toss 승인 응답도 다시 검증한다.
- Toss 전용 DTO와 에러 코드는 어댑터에 격리한다.
- 결제 실패나 취소로 남은 PENDING 예약을 정리한다.
- 테스트는 “외부 호출 전 차단”과 “상태 전이”를 중심으로 작성한다.

이 관점으로 PR을 보면 코드가 훨씬 덜 흩어져 보인다.

## 15. 더 생각해보기 답변 초안

### Q1. 같은 `paymentKey`로 승인을 두 번 호출하면 `ALREADY_PROCESSED_PAYMENT`가 온다. 이를 에러로 볼지, 이미 성공한 결제로 볼지?

상태에 따라 다르게 봐야 한다.

우리 DB에서 이미 같은 `paymentKey`로 `CONFIRMED`라면 중복 요청 또는 새로고침으로 볼 수 있다. 이때는 기존 성공 결과를 돌려주는 것이 자연스럽다.

하지만 우리 DB가 아직 `PENDING`인데 Toss가 이미 처리됐다고 하면 바로 성공 처리하기 어렵다. 우리 서비스 상태와 Toss 상태가 불일치하기 때문이다. 이 경우는 에러로 보고 운영자가 확인하거나 별도 보정 흐름을 두는 편이 안전하다.

### Q2. `onStatus`를 4xx와 5xx로 나눈다면, 5xx 중 어떤 예외를 재시도해도 안전할까?

결제 승인은 부수효과가 있는 요청이라 단순 재시도는 조심해야 한다.

재시도하려면 최소한 다음 조건이 필요하다.

- 같은 주문에 대해 같은 요청임을 식별할 수 있어야 한다.
- 중복 결제를 막을 장치가 있어야 한다.
- 응답을 받지 못한 이전 요청이 실제로 처리됐을 가능성을 고려해야 한다.

Toss 내부 일시 오류처럼 `FAILED_PAYMENT_INTERNAL_SYSTEM_PROCESSING` 계열은 재시도 후보가 될 수 있지만, 실제 구현에서는 멱등키 또는 결제 상태 조회 같은 안전장치가 함께 필요하다.

### Q3. 브라우저 결제창은 왜 클라이언트 키로, 서버 승인은 왜 시크릿 키로 호출할까?

브라우저 코드는 사용자에게 노출된다. 그래서 브라우저에는 공개 가능한 `clientKey`만 둔다.

반면 결제 승인은 실제 결제를 확정하는 민감한 작업이다. 아무나 호출하면 안 되므로 서버가 `secretKey`로 인증해서 Toss API를 호출한다.

즉 역할이 다르다.

```text
clientKey = 결제창을 열기 위한 공개 키
secretKey = 서버가 Toss API를 호출하기 위한 비밀 키
```

### Q4. Toss가 느리거나 연결되지 않으면 우리 서버는 어떻게 될까?

현재 1단계 구현은 외부 호출이 빠르게 응답한다고 가정한다.

만약 Toss가 느리거나 연결되지 않으면 서버 요청 스레드가 오래 붙잡힐 수 있다. 사용자는 응답을 늦게 받거나 실패를 볼 수 있고, 트래픽이 많으면 서버 처리량이 떨어질 수 있다.

그래서 이후 단계에서는 다음이 중요해진다.

- connect timeout
- read timeout
- 재시도 정책
- 멱등성
- 장애 시 사용자 안내
- 결제 상태 보정 또는 조회
