# 2단계 Timeout과 Idempotency-Key 이해 가이드

이 문서는 Toss Payments 미션 2단계를 이해하고, 이번 구현을 리뷰하기 위한 가이드다.

2단계의 핵심은 단순히 `RestClient`에 timeout 숫자를 넣는 것이 아니다. 외부 결제 API가 느리거나 응답을 주지 않을 때 **우리 서버가 멈추지 않게 하고**, 동시에 결제 결과가 애매해진 주문을 **성급히 실패로 단정하지 않는 것**이다.

---

## 1. 2단계에서 해결하려는 문제

1단계 구현은 정상 흐름을 중심으로 되어 있었다.

```text
브라우저 결제 인증 성공
  -> 서버가 Toss 승인 API 호출
  -> Toss가 빠르게 DONE 응답
  -> 예약 CONFIRMED
```

하지만 실제 외부 API는 항상 빠르게 응답하지 않는다.

2단계에서 다루는 위험은 두 가지다.

| 위험 | 무슨 일이 생기는가 | 왜 문제인가 |
| --- | --- | --- |
| Timeout 없음 | Toss 응답이 늦으면 서버 스레드가 계속 대기 | 결제와 무관한 요청까지 느려질 수 있음 |
| Read timeout 후 재시도 | 서버는 응답을 못 받았지만 Toss는 이미 승인했을 수 있음 | 같은 결제를 다시 승인하면 중복 결제 위험 |

그래서 이번 단계의 핵심 문장은 이것이다.

> 느린 외부 호출은 빨리 끊고, 결과가 불명확한 결제는 실패로 단정하지 않고 확인 가능한 상태로 남긴다.

---

## 2. Connect Timeout과 Read Timeout

timeout은 하나가 아니다.

### connect timeout

`connect timeout`은 TCP 연결을 맺기까지 기다리는 시간이다.

```text
우리 서버
  -> Toss 서버에 연결 시도
  -> 연결 자체가 안 됨
  -> connect timeout 또는 connect failure
```

이 경우는 요청이 Toss에 도달하지 못했을 가능성이 크다.

### read timeout

`read timeout`은 연결 후 응답을 읽기까지 기다리는 시간이다.

```text
우리 서버
  -> Toss 서버에 연결 성공
  -> 승인 요청 전송
  -> Toss 응답이 늦음
  -> read timeout
```

이 경우는 더 위험하다.

요청이 Toss에 도달했을 수 있고, Toss가 실제로 결제를 승인했을 수도 있다. 다만 우리 서버가 응답을 받지 못했을 뿐이다.

그래서 `read timeout`은 다음처럼 이해해야 한다.

```text
결제 실패가 아니다.
결제 성공도 아니다.
결과를 모르는 상태다.
```

이번 구현에서는 이 상태를 `PAYMENT_CHECK_REQUIRED`로 남긴다.

## 3. Idempotency-Key가 필요한 이유

결제 승인 API는 `POST`다.

`POST` 요청은 기본적으로 여러 번 보내도 같은 결과가 보장되지 않는다.

예를 들어 다음 상황을 생각해보자.

```text
1. 서버가 Toss 승인 API 호출
2. Toss는 실제로 승인 처리
3. 그런데 우리 서버는 read timeout으로 응답을 못 받음
4. 사용자가 새로고침하거나 서버가 재시도
5. 같은 결제 승인 요청이 다시 나감
```

이때 중복 결제를 막기 위해 `Idempotency-Key`를 사용한다.

Toss 공식 문서 기준으로 멱등키는 다음 성질을 가진다.

- 요청 헤더 이름은 `Idempotency-Key`다.
- 최대 길이는 300자다.
- 최초 요청일부터 15일간 유효하다.
- 같은 API 키, API 주소, HTTP 메서드, 멱등키 조합이면 첫 응답을 재사용한다.
- 같은 요청을 다시 보낼 수 있게 해주지만, 에러 원인을 모른 채 멱등키를 바꿔 재시도하는 것은 위험하다.

이번 구현의 선택은 다음이다.

```text
주문 생성 시 UUID 멱등키를 만든다.
reservation.idempotency_key에 저장한다.
같은 주문의 승인 요청에는 항상 같은 멱등키를 사용한다.
```

중요한 점은 **매 호출마다 새 UUID를 만들면 안 된다**는 것이다. 새 UUID를 만들면 Toss 입장에서는 매번 다른 요청이 되므로 멱등성이 사라진다.

## 4. 이번 구현의 의사결정

2단계에는 선택할 지점이 많다. 이번 구현에서는 다음 기준으로 결정했다.

### 4.1 timeout 설정 방식

선택지는 두 가지였다.

| 선택지 | 장점 | 단점 |
| --- | --- | --- |
| Spring Boot 프로퍼티 방식 | 설정만으로 적용 가능 | 현재 프로젝트는 Spring Boot 3.4.4라 요구사항의 Boot 4.0 복수형 prefix와 맞지 않음 |
| 코드 방식 | 현재 버전에서 명확하게 동작 | 설정 코드가 필요함 |

이번 구현은 **코드 방식**을 선택했다.

단, timeout 값은 코드에 박지 않고 `application.yml`에 둔다.

```yaml
payment:
  toss:
    connect-timeout: 500ms
    read-timeout: 2s
```

실제 적용은 `SimpleClientHttpRequestFactory`로 한다.

### 4.2 read timeout 상태 표현

선택지는 세 가지였다.

| 선택지 | 장점 | 문제 |
| --- | --- | --- |
| 그냥 실패 처리 | 구현이 단순 | 실제로 승인됐을 수 있어 위험 |
| 계속 PENDING 유지 | 기존 상태 재사용 | 사용자가 “무슨 일이 생겼는지” 알기 어려움 |
| 확인 필요 상태 추가 | 의미가 명확 | 상태와 UI, query 수정 필요 |

이번 구현은 `PAYMENT_CHECK_REQUIRED` 상태를 추가했다.

이 상태는 다음 의미다.

```text
Toss 승인 요청은 보냈다.
하지만 응답을 받지 못했다.
결제 결과를 확인하거나 같은 멱등키로 재시도해야 한다.
```

### 4.3 실패 내역을 남길 것인가, 삭제할 것인가

1단계에서는 결제 실패 시 PENDING 예약을 삭제했다.

2단계 요구사항은 주문/결제 내역에서 실패 상태도 확인하라고 한다. 그래서 삭제 대신 `PAYMENT_FAILED`로 남긴다.

단, 실패 주문이 슬롯을 계속 점유하면 안 된다. 그래서 활성 예약 판단은 상태 기준으로 바꿨다.

```text
활성 예약으로 보는 상태:
- PENDING
- CONFIRMED
- PAYMENT_CHECK_REQUIRED

활성 예약으로 보지 않는 상태:
- PAYMENT_FAILED
```

이 결정 덕분에 실패 내역은 남기면서, 사용자는 같은 슬롯을 다시 예약할 수 있다.

## 6. DB 변경: 멱등키와 결제 상태 저장

파일:

- `src/main/resources/schema.sql`

추가/변경된 핵심은 다음이다.

```sql
status VARCHAR(30) NOT NULL DEFAULT 'CONFIRMED',
order_id VARCHAR(64),
idempotency_key VARCHAR(300),
amount INT NOT NULL DEFAULT 10000,
payment_key VARCHAR(200),
CONSTRAINT uk_reservation_order UNIQUE (order_id),
CONSTRAINT uk_reservation_idempotency_key UNIQUE (idempotency_key)
```

### 왜 status 길이를 늘렸나?

새 상태 `PAYMENT_CHECK_REQUIRED`가 길기 때문이다. 기존 `VARCHAR(20)`으로는 저장할 수 없어 `VARCHAR(30)`으로 늘렸다.

### 왜 idempotency_key를 저장하나?

멱등키는 주문당 고정되어야 한다.

```text
예약 생성 시 한 번 생성
-> DB 저장
-> 같은 주문 confirm 요청마다 같은 키 사용
```

### 왜 reservation의 slot unique 제약을 제거했나?

실패 주문을 내역에 남기기 위해서다.

`PAYMENT_FAILED` 주문이 남아 있어도 그 슬롯은 다시 예약 가능해야 한다. 전체 `slot_id`에 unique 제약이 있으면 실패 내역 하나 때문에 같은 슬롯을 다시 예약할 수 없다.

대신 활성 예약 여부는 repository query에서 상태 기준으로 판단한다.

## 7. 주문 생성: orderId와 idempotencyKey를 함께 만든다

파일:

- `src/main/java/roomescape/reservation/application/OrderIdGenerator.java`
- `src/main/java/roomescape/reservation/application/ReservationService.java`
- `src/main/java/roomescape/reservation/domain/Reservation.java`

`OrderIdGenerator`는 이제 두 값을 만든다.

```java
public String generate() {
    return UUID.randomUUID().toString().replace("-", "");
}

public String generateIdempotencyKey() {
    return UUID.randomUUID().toString();
}
```

`ReservationService.save()`는 예약 생성 시 두 값을 모두 넣는다.

```java
Reservation reservation = reservationRepository.save(
        Reservation.createPending(
                memberId,
                slot,
                orderIdGenerator.generate(),
                orderIdGenerator.generateIdempotencyKey()
        )
);
```

`Reservation.createPending(...)`은 `PENDING` 상태와 결제 정보를 가진 주문을 만든다.

```java
public static Reservation createPending(long memberId, Slot slot, String orderId, String idempotencyKey) {
    return new Reservation(
            null,
            memberId,
            slot,
            ReservationStatus.PENDING,
            orderId,
            idempotencyKey,
            slot.getPrice(),
            null
    );
}
```

여기서 `paymentKey`는 아직 없다. `paymentKey`는 Toss 인증 성공 후 successUrl로 돌아올 때 생긴다.

## 8. Timeout 설정: SimpleClientHttpRequestFactory

파일:

- `src/main/java/roomescape/payment/config/TossPaymentProperties.java`
- `src/main/java/roomescape/payment/config/TossPaymentConfig.java`
- `src/main/resources/application.yml`

설정값은 `TossPaymentProperties`에 바인딩된다.

```java
@ConfigurationProperties(prefix = "payment.toss")
public record TossPaymentProperties(
        String baseUrl,
        String secretKey,
        String clientKey,
        Duration connectTimeout,
        Duration readTimeout
) {
}
```

`TossPaymentConfig`에서 request factory에 적용한다.

```java
SimpleClientHttpRequestFactory requestFactory = new SimpleClientHttpRequestFactory();
requestFactory.setConnectTimeout(properties.connectTimeout());
requestFactory.setReadTimeout(properties.readTimeout());

return RestClient.builder()
        .baseUrl(properties.baseUrl())
        .defaultHeader(HttpHeaders.AUTHORIZATION, "Basic " + basicToken)
        .requestFactory(requestFactory)
        .build();
```

이번 구현의 기본값:

```yaml
connect-timeout: 500ms
read-timeout: 2s
```

이 값은 정답이라기보다 시작점이다. 운영에서는 Toss 응답 시간, 사용자 경험, 서버 처리량을 보고 조정해야 한다.

## 9. Toss 승인 요청: Idempotency-Key 헤더 추가

파일:

- `src/main/java/roomescape/payment/domain/PaymentConfirmation.java`
- `src/main/java/roomescape/payment/adapter/out/toss/TossPaymentGateway.java`

`PaymentConfirmation`에 멱등키가 추가됐다.

```java
public record PaymentConfirmation(
        String paymentKey,
        String orderId,
        String idempotencyKey,
        int amount
) {
}
```

`TossPaymentGateway`는 이 값을 헤더로 보낸다.

```java
TossPaymentResponse response = tossRestClient.post()
        .uri("/v1/payments/confirm")
        .contentType(MediaType.APPLICATION_JSON)
        .header("Idempotency-Key", confirmation.idempotencyKey())
        .body(request)
        .retrieve()
        ...
        .body(TossPaymentResponse.class);
```

이제 같은 주문의 confirm 재요청은 같은 멱등키를 사용한다.

```text
orderId = A
idempotencyKey = K

1번째 confirm -> K
read timeout 발생
2번째 confirm -> K
```

Toss는 같은 요청으로 보고 첫 응답을 재사용할 수 있다.

## 10. Toss 에러와 네트워크 예외 분리

파일:

- `src/main/java/roomescape/payment/adapter/out/toss/TossPaymentGateway.java`
- `src/main/java/roomescape/exception/ErrorCode.java`

Toss가 HTTP 응답으로 내려주는 에러는 기존처럼 `onStatus`에서 처리한다.

```java
.onStatus(HttpStatusCode::isError, (req, res) -> {
    TossErrorResponse error = parseError(res.getBody());
    throw toDomainException(error);
})
```

반면 timeout, connect failure는 Toss JSON 에러가 아니다. `RestClientException` 계열로 온다.

```java
} catch (ResourceAccessException exception) {
    throw toConnectionException(exception);
} catch (RestClientException exception) {
    throw toConnectionException(exception);
}
```

root cause를 보고 분기한다.

```java
if (rootCause instanceof SocketTimeoutException) {
    return new EscapeRoomException(ErrorCode.PAYMENT_GATEWAY_TIMEOUT_UNKNOWN);
}
if (rootCause instanceof ConnectException) {
    return new EscapeRoomException(ErrorCode.PAYMENT_GATEWAY_UNAVAILABLE);
}
```

이번 구현에서 중요한 구분:

| 상황 | ErrorCode | 상태 변경 |
| --- | --- | --- |
| read timeout 등 timeout | `PAYMENT_GATEWAY_TIMEOUT_UNKNOWN` | `PAYMENT_CHECK_REQUIRED` |
| 연결 실패 | `PAYMENT_GATEWAY_UNAVAILABLE` | 상태 변경 없음 |
| 카드 거절 | `PAYMENT_CARD_REJECTED` | `PAYMENT_FAILED` |
| 잘못된 요청 | `PAYMENT_INVALID_REQUEST` | `PAYMENT_FAILED` |

## 11. PaymentService: 상태 전이의 중심

파일:

- `src/main/java/roomescape/payment/application/PaymentService.java`

`PaymentService.confirm(...)`은 여전히 결제 승인 흐름의 중심이다.

2단계에서 추가된 핵심은 gateway 예외 처리다.

```java
private PaymentResult confirmPayment(PaymentConfirmRequest request, Reservation reservation) {
    try {
        return paymentGateway.confirm(
                new PaymentConfirmation(
                        request.paymentKey(),
                        request.orderId(),
                        reservation.getIdempotencyKey(),
                        request.amount()
                )
        );
    } catch (EscapeRoomException exception) {
        handleGatewayException(request, reservation, exception);
        throw exception;
    }
}
```

타임아웃은 확인 필요 상태로 남긴다.

```java
if (errorCode == ErrorCode.PAYMENT_GATEWAY_TIMEOUT_UNKNOWN) {
    reservationRepository.markPaymentCheckRequired(
            reservation.getId(),
            reservation.getOrderId(),
            request.paymentKey()
    );
    return;
}
```

명확한 결제 실패는 실패 상태로 남긴다.

```java
if (FAILED_PAYMENT_ERRORS.contains(errorCode)) {
    reservationRepository.markPaymentFailed(
            reservation.getId(),
            reservation.getOrderId(),
            request.paymentKey()
    );
}
```

### 왜 `noRollbackFor = EscapeRoomException.class`가 필요한가?

`confirm()`은 예외를 응답으로 내려보내야 한다.

하지만 동시에 `PAYMENT_CHECK_REQUIRED`나 `PAYMENT_FAILED` 상태는 DB에 남아야 한다.

기본 트랜잭션에서는 런타임 예외가 발생하면 상태 변경도 롤백된다. 그래서 다음 설정이 필요했다.

```java
@Transactional(noRollbackFor = EscapeRoomException.class)
public PaymentConfirmResponse confirm(...)
```

이 결정은 중요하다.

```text
사용자에게는 에러 응답
DB에는 확인 필요/실패 상태 기록
```

두 가지를 동시에 만족시키기 위한 선택이다.

## 12. 실패 콜백: 삭제 대신 실패 상태로 남긴다

파일:

- `src/main/java/roomescape/payment/application/PaymentService.java`
- `src/main/java/roomescape/reservation/adapter/out/persistence/JdbcReservationRepository.java`

1단계에서는 failUrl을 받으면 PENDING 예약을 삭제했다.

2단계에서는 내역 확인 요구사항 때문에 실패 상태로 남긴다.

```java
public void handleFailure(PaymentFailRequest request, long memberId) {
    if (!StringUtils.hasText(request.orderId())) {
        return;
    }
    reservationRepository.markPendingPaymentFailedByOrderIdAndMemberId(request.orderId(), memberId);
}
```

repository는 로그인 사용자와 PENDING 상태를 조건으로 건다.

```sql
UPDATE reservation
SET status = 'PAYMENT_FAILED'
WHERE order_id = :orderId
AND member_id = :memberId
AND status = 'PENDING'
```

이 조건이 중요한 이유:

- 다른 사용자의 주문을 실패 처리하면 안 된다.
- 이미 확정된 예약을 failUrl로 실패 처리하면 안 된다.
- orderId가 없는 사용자 취소 흐름은 무시해야 한다.

## 13. 활성 예약 판단 변경

파일:

- `src/main/java/roomescape/reservation/adapter/out/persistence/JdbcReservationRepository.java`

실패 내역을 DB에 남기면서도 슬롯을 다시 예약 가능하게 하려면, “reservation row가 존재한다”와 “슬롯을 점유한다”를 분리해야 한다.

이번 구현에서는 활성 예약 상태를 이렇게 정의했다.

```java
private static final List<String> ACTIVE_PAYMENT_STATUSES = List.of(
        ReservationStatus.PENDING.name(),
        ReservationStatus.CONFIRMED.name(),
        ReservationStatus.PAYMENT_CHECK_REQUIRED.name()
);
```

`existsBySlotId(...)`는 이 상태들만 본다.

```sql
SELECT EXISTS (
    SELECT 1
    FROM reservation
    WHERE slot_id = :slotId
    AND status IN (:activeStatuses)
)
```

날짜/테마별 예약된 시간 조회도 active 상태만 본다.

```sql
LEFT JOIN reservation r
    ON s.id = r.slot_id
    AND r.status IN (:activeStatuses)
```

따라서 상태별 의미는 다음과 같다.

| 상태 | 내역 표시 | 슬롯 점유 | 의미 |
| --- | --- | --- | --- |
| `PENDING` | O | O | 결제 대기 |
| `CONFIRMED` | O | O | 예약 확정 |
| `PAYMENT_CHECK_REQUIRED` | O | O | 결제 결과 불명확, 확인 필요 |
| `PAYMENT_FAILED` | O | X | 결제 실패, 다시 예약 가능 |

파일:

```java

```

브라우저는 상태별 라벨을 만든다.

```javascript
function paymentStatusLabel(status) {
  if (status === "CONFIRMED") return "예약 확정";
  if (status === "PENDING") return "결제 대기";
  if (status === "PAYMENT_CHECK_REQUIRED") return "확인 필요";
  if (status === "PAYMENT_FAILED") return "결제 실패";
  if (status === "WAITING") return "대기";
  return status;
}
```

예약 카드에는 결제 정보도 표시한다.

```text
결제 상태
주문번호
결제 금액
결제키
```

`PAYMENT_CHECK_REQUIRED`와 `PAYMENT_FAILED`는 카드 메뉴에서 바로 취소하지 않게 했다.

특히 `PAYMENT_CHECK_REQUIRED`는 실제 승인됐을 수 있으므로, 단순 예약 취소로 슬롯을 풀면 위험하다.

## 16. 이번 단계에서 고민해야 하는 지점

### 16.1 timeout 값은 얼마가 적절한가?

이번 구현은 다음 값을 사용했다.

```yaml
connect-timeout: 500ms
read-timeout: 2s
```

하지만 이 값은 절대적인 정답이 아니다.

고민 기준:

- Toss 평균 응답 시간이 어느 정도인가?
- 사용자가 결제 버튼을 누른 뒤 몇 초까지 기다릴 수 있는가?
- 서버 스레드를 얼마나 빨리 보호해야 하는가?
- read timeout 후 재시도/조회 흐름이 충분히 준비되어 있는가?

### 16.2 connect 실패와 read timeout은 같은가?

같지 않다.

`connect` 실패는 요청이 도달하지 못했을 가능성이 크다.

`read timeout`은 요청이 도달했고 처리됐을 수도 있다.

그래서 read timeout은 더 보수적으로 다뤄야 한다.

### 16.3 확인 필요 상태는 어떻게 수렴시킬 것인가?

이번 단계에서는 `PAYMENT_CHECK_REQUIRED`로 남기고, 같은 멱등키로 재시도할 수 있게 했다.

하지만 더 완성도 높은 서비스라면 다음 중 하나가 필요하다.

- Toss 결제 조회 API로 상태 확인
- 백오피스에서 확인 필요 주문 처리
- 일정 시간 후 자동 재조회
- 사용자가 “결제 상태 다시 확인” 버튼을 누르는 흐름

즉 `PAYMENT_CHECK_REQUIRED`는 최종 상태라기보다 보정이 필요한 중간 상태다.

### 16.4 실패 내역을 reservation에 남기는 것이 최선인가?

이번 구현은 현재 구조를 크게 늘리지 않기 위해 reservation row에 실패/확인 필요 상태를 함께 남겼다.

대안은 별도 `payment_history` 또는 `order` 테이블을 만드는 것이다.

| 선택 | 장점 | 단점 |
| --- | --- | --- |
| reservation에 상태 추가 | 구현 범위가 작고 현재 화면과 연결 쉬움 | 예약과 결제 주문의 책임이 섞임 |
| 별도 payment/order 테이블 | 결제 이력 모델이 명확함 | 미션 범위 대비 변경량이 커짐 |

이번 단계에서는 요구사항 2의 범위와 기존 구조를 고려해 reservation 확장을 선택했다.

## 17. 더 생각해보기 답변 초안

### Q1. read timeout 값과 Toss 응답 시간이 거의 같은 경계값에서는 성공과 실패가 어떻게 갈릴까?

경계값에서는 작은 네트워크 지연, 스케줄링, 서버 처리 시간 차이로 성공과 실패가 갈릴 수 있다.

예를 들어 read timeout이 2초이고 Toss 응답도 2초 근처라면 어떤 요청은 성공하고 어떤 요청은 timeout이 날 수 있다. 그래서 timeout 값은 너무 공격적으로 짧게 잡으면 정상 결제를 확인 필요 상태로 많이 보낼 수 있고, 너무 길게 잡으면 서버 스레드를 오래 붙잡는다.

### Q2. connect와 read 중 어느 timeout을 더 짧게 두는 게 합리적일까?

일반적으로 connect timeout은 더 짧게 둔다.

연결 자체가 안 되는 상황은 빨리 포기하는 편이 낫다. read timeout은 Toss가 실제로 처리 중일 수 있으므로 사용자 경험과 외부 API 평균 응답 시간을 보고 조금 더 여유를 둔다.

이번 구현도 `connect-timeout: 500ms`, `read-timeout: 2s`로 두었다.

### Q3. read timeout을 실패로 단정하는 것과 확인 필요로 두는 것 중 무엇이 나은가?

확인 필요로 두는 것이 안전하다.

read timeout은 요청이 Toss에 도달했는지, 결제가 승인됐는지 모르는 상태다. 실패로 단정하고 슬롯을 풀면 실제로는 결제된 주문을 다시 판매할 수 있다.

### Q4. 확인 필요가 영구 상태로 방치되지 않으려면 무엇이 필요한가?

결제 상태를 다시 확인하는 수단이 필요하다.

후속 단계나 실제 서비스에서는 다음 중 하나가 필요하다.

- 결제 조회 API
- 같은 Idempotency-Key로 안전 재시도
- 관리자 확인 페이지
- 자동 재조회 배치
- 웹훅 수신

### Q5. read timeout과 connect 실패는 재시도 안전성이 다를까?

다르다.

connect 실패는 요청이 도달하지 못했을 가능성이 높아 재시도 위험이 상대적으로 낮다. read timeout은 요청이 이미 처리됐을 수 있어 멱등키 없이 재시도하면 위험하다.

그래서 승인 API에는 `Idempotency-Key`를 붙인다.
