# 3단계 Rate Limit 이해 가이드

## 이 문서의 목표

이 문서는 3단계 요구사항을 구현한 코드를 읽기 전에 필요한 배경 지식과, 이번 브랜치에 추가된 코드가 각각 어떤 요구사항을 만족하는지 함께 설명한다.

이 문서만 읽고 나면 다음을 설명할 수 있어야 한다.

1. 왜 결제 미션에서 Rate Limit이 필요한지
2. 토큰 버킷이 `capacity`와 `refillPerSecond`로 어떤 동작을 만드는지
3. 인바운드 Rate Limit과 아웃바운드 Rate Limit이 무엇이 다른지
4. 토스가 `429 Too Many Requests`를 반환했을 때 왜 `Retry-After`를 존중해야 하는지
5. 3단계에서 추가된 코드가 어떤 요구사항을 위해 존재하는지

## 3단계 요구사항 상황 이해

1단계에서는 예약을 만들 때 결제에 필요한 `orderId`, `amount`를 함께 만들었다. 2단계에서는 결제 승인 요청이 느리거나 응답 여부를 알 수 없을 때를 대비해서 타임아웃과 멱등키를 추가했다.

3단계의 관심사는 "호출이 너무 많을 때"다.

호출이 많아지는 방향은 두 가지다.

1. 사용자가 우리 서버로 너무 많이 요청한다.
2. 우리 서버가 토스 서버로 너무 많이 요청한다.

첫 번째는 우리 서버를 보호하는 문제다. 예약/결제 API에 요청이 몰리면 일정량까지만 통과시키고, 초과분은 컨트롤러까지 보내지 않고 바로 `429`로 거부해야 한다.

두 번째는 외부 API 사용자로서 지켜야 할 예의이자 안정성 문제다. 토스가 허용한 호출량보다 많이 보내면 어차피 토스에서 `429`로 거부한다. 그러므로 우리 서버가 토스 호출 직전에 먼저 한도를 확인하고, 초과하면 외부로 보내지 않는 편이 낫다.

즉 3단계의 핵심은 이것이다.

```text
들어오는 요청: 우리 서버 보호
나가는 요청: 토스 서버 한도 보호
공통 알고리즘: Token Bucket
```

## Rate Limit과 TPS

TPS는 초당 처리한 트랜잭션 수라는 "측정값"이다.

Rate Limit은 초당 몇 건까지 허용할지 정하는 "정책"이다.

예를 들어 `refillPerSecond = 10`이면 평균적으로 초당 10개 토큰이 채워진다. 요청 하나는 토큰 하나를 소비한다. 그래서 긴 시간으로 보면 초당 10건 정도만 통과한다.

하지만 실제 서비스에서는 순간적으로 요청이 몰릴 수 있다. 이때 `capacity`가 버스트를 허용한다. `capacity = 30`, `refillPerSecond = 10`이면 평소에는 최대 30개의 토큰을 모아둘 수 있고, 짧은 순간에는 30건이 바로 통과할 수 있다. 대신 이후에는 토큰이 다시 찰 때까지 초과 요청이 거부된다.

## 토큰 버킷 원리

이번 구현의 토큰 버킷은 다음 규칙을 따른다.

```text
현재 토큰 수 = min(capacity, 현재 토큰 수 + 경과 시간 * refillPerSecond)
요청 1개 통과 = 토큰 1개 소비
토큰이 없으면 = 거부
```

구현 파일은 `src/main/java/roomescape/ratelimit/TokenBucketRateLimiter.java`다.

핵심 코드는 다음과 같다.

```java
public synchronized boolean tryConsume() {
    refill();
    if (tokens < 1) {
        return false;
    }
    tokens -= 1;
    return true;
}
```

`tryConsume()`은 요청이 들어올 때마다 호출된다. 먼저 `refill()`로 지금까지 경과한 시간만큼 토큰을 채운다. 그 뒤 토큰이 1개 이상 있으면 1개를 소비하고 `true`를 반환한다. 토큰이 없으면 `false`를 반환한다.

`synchronized`를 붙인 이유는 동시 요청 때문이다. 여러 요청이 동시에 들어왔을 때 같은 토큰을 중복 소비하면 안 된다. 예를 들어 `capacity = 5`인데 20개 스레드가 동시에 접근해도 정확히 5개만 통과해야 한다.

`retryAfterSeconds()`는 거부된 클라이언트에게 "몇 초 뒤 다시 시도하면 좋은지" 알려주기 위한 값이다.

```java
public synchronized long retryAfterSeconds() {
    refill();
    if (tokens >= 1) {
        return 0;
    }
    double missingTokens = 1 - tokens;
    return Math.max(1, (long) Math.ceil(missingTokens / refillPerSecond));
}
```

`LongSupplier`로 시간을 주입받는 이유는 테스트 때문이다. 운영에서는 `System::nanoTime`을 쓰지만, 테스트에서는 가짜 시계를 직접 앞으로 이동시켜야 "0.5초 뒤 토큰 1개 보충" 같은 상황을 결정적으로 검증할 수 있다.

## 인바운드 Rate Limit

인바운드 Rate Limit은 사용자가 우리 서버에 보내는 요청을 제한한다.

이번 구현에서는 예약/결제 API에만 적용했다.

```java
registry.addInterceptor(rateLimitInterceptor)
        .addPathPatterns(
                "/api/user/reservations/**",
                "/api/user/payments/**"
        );
```

구현 파일은 `src/main/java/roomescape/ratelimit/RateLimitConfig.java`다.

컨트롤러 앞단에서 요청을 막기 위해 `HandlerInterceptor`를 사용했다. 구현 파일은 `src/main/java/roomescape/ratelimit/RateLimitInterceptor.java`다.

핵심 코드는 다음과 같다.

```java
if (rateLimiter.tryConsume()) {
    return true;
}

response.setStatus(HttpStatus.TOO_MANY_REQUESTS.value());
response.setHeader(HttpHeaders.RETRY_AFTER, String.valueOf(rateLimiter.retryAfterSeconds()));
objectMapper.writeValue(
        response.getWriter(),
        ApiResponse.fail(errorCode.getCode(), errorCode.getMessage())
);
return false;
```

`true`를 반환하면 컨트롤러로 요청이 계속 진행된다. `false`를 반환하면 컨트롤러는 호출되지 않는다.

요구사항에서 "컨트롤러 호출 없이 429로 거부"라고 한 부분을 이 코드가 만족한다.

응답에는 두 가지가 들어간다.

1. HTTP Status: `429 Too Many Requests`
2. Header: `Retry-After: n`

`Retry-After`는 클라이언트에게 다시 시도할 최소 대기 시간을 알려준다.

설정은 `src/main/resources/application.yml`에 있다.

```yaml
rate-limit:
  capacity: 50
  refill-per-second: 50
```

이 값을 바꾸면 코드 수정 없이 인바운드 요청 허용량을 조절할 수 있다.

## 아웃바운드 Rate Limit

아웃바운드 Rate Limit은 우리 서버가 토스 서버로 보내는 요청을 제한한다.

인바운드와 같은 `TokenBucketRateLimiter`를 쓰지만, 설정은 분리했다.

```yaml
outbound-rate-limit:
  capacity: 20
  refill-per-second: 20
  max-attempts: 3
  fallback-retry-after: 1s
```

분리한 이유는 두 한도의 기준이 다르기 때문이다.

인바운드 한도는 우리 서버가 견딜 수 있는 양이다. 아웃바운드 한도는 토스가 우리에게 허용한 양 또는 우리가 외부 API에 보내도 괜찮다고 정한 양이다. 두 값이 우연히 같을 수는 있어도, 같은 정책이라고 보면 안 된다.

아웃바운드 호출 전 토큰을 확인하는 구현은 `src/main/java/roomescape/ratelimit/OutboundRateLimitInterceptor.java`다.

```java
if (!rateLimiter.tryConsume()) {
    throw new OutboundRateLimitException(rateLimiter.retryAfterSeconds());
}
return execution.execute(request, body);
```

토큰이 없으면 `execution.execute(...)`를 호출하지 않는다. 즉 외부 요청 자체가 나가지 않는다.

이 방식은 fail-fast다. 토큰이 찰 때까지 서버 스레드를 붙잡고 기다리는 대신, 지금은 처리할 수 없다고 즉시 실패시킨다.

장점은 서버 자원을 오래 붙잡지 않는다는 것이다. 단점은 사용자가 바로 실패 응답을 받을 수 있다는 것이다. 이번 단계에서는 요구사항이 "외부로 보내지 않고 거부"이므로 fail-fast가 더 적절하다.

## 토스 429와 Retry-After 재시도

토스가 `429 Too Many Requests`를 반환하면, 토스 입장에서는 "지금 너무 많이 왔으니 조금 뒤 다시 보내라"는 의미다.

이때 응답 헤더에 `Retry-After`가 있으면 그 시간만큼 기다렸다가 재시도한다.

구현 파일은 `src/main/java/roomescape/ratelimit/RetryAfterInterceptor.java`다.

핵심 코드는 다음과 같다.

```java
ClientHttpResponse response = execution.execute(request, body);
if (response.getStatusCode() != HttpStatus.TOO_MANY_REQUESTS) {
    return response;
}
if (attempt >= maxAttempts) {
    response.close();
    throw new EscapeRoomException(ErrorCode.PAYMENT_GATEWAY_RATE_LIMITED);
}

Duration retryAfter = retryAfter(response.getHeaders());
response.close();
sleep(retryAfter);
attempt++;
```

`429`가 아니면 그대로 응답을 반환한다. `429`면 재시도 가능 횟수를 확인한다. 아직 남아 있으면 `Retry-After`만큼 대기한 뒤 같은 요청을 다시 보낸다.

`Retry-After`가 없거나 숫자가 아니면 `fallbackRetryAfter`를 사용한다.

```java
outbound-rate-limit:
  fallback-retry-after: 1s
```

무한 재시도를 막기 위해 `maxAttempts`를 둔다.

```java
outbound-rate-limit:
  max-attempts: 3
```

3번 시도했는데도 계속 `429`면 `PAYMENT_GATEWAY_RATE_LIMITED` 예외로 실패한다.

## RestClient 인터셉터 순서

토스 호출 `RestClient`에는 두 인터셉터가 등록된다.

구현 파일은 `src/main/java/roomescape/payment/config/TossPaymentConfig.java`다.

```java
return RestClient.builder()
        .baseUrl(properties.baseUrl())
        .defaultHeader(HttpHeaders.AUTHORIZATION, "Basic " + basicToken)
        .requestFactory(requestFactory)
        .requestInterceptor(retryAfterInterceptor)
        .requestInterceptor(outboundRateLimitInterceptor)
        .build();
```

순서가 중요하다.

`RetryAfterInterceptor`가 바깥쪽에 있고, 그 안쪽에 `OutboundRateLimitInterceptor`가 있다. 그래서 토스 429 때문에 재시도할 때마다 아웃바운드 토큰을 다시 소비한다.

만약 순서가 반대라면 최초 요청에서만 토큰을 소비하고, 재시도 요청은 토큰을 소비하지 않을 수 있다. 그러면 "나가는 호출마다 Rate Limit 적용"이라는 요구사항과 어긋난다.

## 2단계 멱등성과 3단계 재시도의 관계

2단계에서 예약마다 고정 `idempotencyKey`를 만들었다.

3단계의 429 재시도는 같은 HTTP 요청을 다시 보내는 구조다. 이때 결제 승인 요청의 `Idempotency-Key`가 바뀌면 안 된다.

`TossPaymentGateway`는 결제 승인 요청을 보낼 때 예약에 저장된 멱등키를 헤더에 담는다.

```java
.header("Idempotency-Key", confirmation.idempotencyKey())
```

그리고 `RetryAfterInterceptor`는 같은 `request`, 같은 `body`로 다시 `execution.execute(request, body)`를 호출한다. 따라서 재시도에서도 같은 멱등키가 유지된다.

테스트 `TossPaymentGatewayTest`에는 토스가 첫 번째 요청에 `429`를 반환하고 두 번째 요청에 성공을 반환하는 상황을 추가했다. 두 요청 모두 같은 `Idempotency-Key` 헤더를 기대하도록 해서, 2단계의 안전장치가 3단계 재시도에서도 유지되는지 검증한다.

## read timeout 재시도와 429 재시도의 차이

2단계의 read timeout은 위험하다. 우리 서버가 응답을 못 받았을 뿐, 토스에서는 결제가 이미 승인됐을 수도 있다. 그래서 read timeout에서는 무작정 같은 요청을 다시 보내는 대신 `PAYMENT_CHECK_REQUIRED` 상태로 바꾸고 확인이 필요하다고 표시했다.

반면 429는 "요청량이 많아서 지금 처리하지 않았다"는 의미에 가깝다. 따라서 `Retry-After` 이후 같은 요청을 다시 보내는 것이 자연스럽다.

그래도 결제 승인 요청이기 때문에 멱등키는 유지해야 한다. 네트워크나 외부 시스템에서 어떤 일이 있었는지 항상 100% 알 수 없기 때문이다.

정리하면 다음과 같다.

```text
read timeout:
  응답을 못 받음
  이미 처리됐을 수도 있음
  상태를 PAYMENT_CHECK_REQUIRED로 바꾸고 확인 필요

429:
  호출량 초과
  Retry-After 이후 재시도 가능
  같은 멱등키로 재시도
```

## 3단계에서 추가된 코드 리뷰

### `TokenBucketRateLimiter`

역할: 인바운드/아웃바운드에서 공통으로 사용하는 토큰 버킷 알고리즘.

필요한 요구사항:

- `capacity`, `refillPerSecond` 기반 직접 구현
- 가짜 시계 주입 가능
- `tryConsume()`
- `retryAfterSeconds()`
- 동시성 안전

테스트:

- `TokenBucketRateLimiterTest`
- 초기 capacity만 통과
- 경과 시간만큼 보충
- capacity 초과 보충 방지
- Retry-After 계산
- 동시 요청에서 정확히 capacity개만 통과

### `RateLimitInterceptor`

역할: 예약/결제 API로 들어오는 요청을 컨트롤러 전에 제한.

필요한 요구사항:

- 초과 요청은 컨트롤러 호출 없이 `429`
- `Retry-After` 헤더 포함
- 프로젝트 표준 `ApiResponse.fail(...)` 형식 사용

테스트:

- `RateLimitInterceptorTest`
- 첫 요청은 통과
- 두 번째 요청은 `false`
- 상태 코드 `429`
- `Retry-After` 포함
- 응답 본문에 `RATE_LIMIT_429` 포함

### `RateLimitConfig`

역할: 설정값으로 토큰 버킷을 만들고, 인터셉터를 등록.

필요한 요구사항:

- `rate-limit.*` 외부화
- `outbound-rate-limit.*` 외부화
- 예약/결제 엔드포인트에 인바운드 Rate Limit 적용
- 토스 RestClient에서 쓸 outbound 인터셉터 빈 생성

핵심 판단:

- 인바운드 버킷과 아웃바운드 버킷은 같은 클래스지만 다른 인스턴스로 둔다.
- 두 방향의 정책 기준이 다르므로 설정도 분리한다.

### `OutboundRateLimitInterceptor`

역할: 토스 호출 직전에 outbound 토큰을 확인하고, 없으면 외부 호출을 보내지 않음.

필요한 요구사항:

- 같은 토큰 버킷 재사용
- 한도 초과 시 외부로 보내지 않고 `OutboundRateLimitException`

테스트:

- `OutboundRateLimitInterceptorTest`
- 첫 요청은 실행
- 두 번째 요청은 예외
- 외부 실행 횟수는 1회로 유지

### `RetryAfterInterceptor`

역할: 토스가 `429`를 반환하면 `Retry-After`만큼 대기 후 재시도.

필요한 요구사항:

- `ClientHttpRequestInterceptor`
- `Retry-After` 초 단위 해석
- 헤더가 없으면 fallback 사용
- `maxAttempts` 이후 도메인 예외
- 무한 재시도 금지

테스트:

- `RetryAfterInterceptorTest`
- `Retry-After: 2`이면 2초 대기를 기록하고 재시도
- 헤더가 없으면 fallback 사용
- 재시도 횟수 초과 시 `PAYMENT_GATEWAY_RATE_LIMITED`

### `TossPaymentConfig`

역할: 토스 결제 승인용 `RestClient`에 타임아웃, 인증 헤더, Rate Limit 인터셉터를 함께 설정.

필요한 요구사항:

- 기존 2단계 타임아웃 유지
- 토스 429 재시도 인터셉터 등록
- outbound Rate Limit 인터셉터 등록

핵심 판단:

- `retryAfterInterceptor`를 먼저 등록하고 `outboundRateLimitInterceptor`를 나중에 등록했다.
- 이 순서여야 429 재시도 요청마다 outbound 토큰을 다시 소비한다.

### `ErrorCode`

추가된 에러는 세 가지다.

```text
RATE_LIMIT_EXCEEDED
PAYMENT_GATEWAY_RATE_LIMITED
PAYMENT_OUTBOUND_RATE_LIMITED
```

`RATE_LIMIT_EXCEEDED`는 사용자가 우리 서버 인바운드 한도를 초과했을 때 사용한다.

`PAYMENT_GATEWAY_RATE_LIMITED`는 토스가 계속 429를 반환해서 재시도를 포기했을 때 사용한다.

`PAYMENT_OUTBOUND_RATE_LIMITED`는 우리 서버가 토스 호출을 보내기 전 자체 outbound 한도를 초과했을 때 사용한다.

## 실행 흐름으로 따라가기

예약 생성 흐름은 다음 순서로 보면 된다.

```text
브라우저
  -> /api/user/reservations
  -> RateLimitInterceptor
  -> UserReservationController
  -> ReservationService.save()
  -> ReservationRepository.save()
  -> 브라우저에 orderId, amount 반환
```

결제 승인 흐름은 다음 순서로 보면 된다.

```text
브라우저
  -> /api/user/payments/confirm
  -> RateLimitInterceptor
  -> PaymentController
  -> PaymentService.confirm()
  -> TossPaymentGateway.confirm()
  -> RestClient
      -> RetryAfterInterceptor
      -> OutboundRateLimitInterceptor
      -> Toss Payments API
```

Rate Limit 관점에서 읽을 때는 다음 질문을 던지면 좋다.

1. 이 요청은 들어오는 요청인가, 나가는 요청인가?
2. 어느 설정값을 사용하는가?
3. 토큰이 없으면 컨트롤러 호출 전에 막히는가, 외부 호출 전에 막히는가?
4. 거부 응답에 `Retry-After`가 있는가?
5. 토스 429 재시도에서도 멱등키가 유지되는가?

## 더 생각해보기 답변

### `capacity`를 키우면 순간 버스트가, `refillPerSecond`를 키우면 평균 처리량이 어떻게 변하는가?

`capacity`를 키우면 한순간에 몰린 요청을 더 많이 받아줄 수 있다. 대신 너무 크게 잡으면 순간적으로 서버 부하가 커질 수 있다.

`refillPerSecond`를 키우면 장기적으로 통과 가능한 평균 요청 수가 늘어난다. 초당 10개가 채워지면 평균 10 TPS에 가깝고, 초당 50개가 채워지면 평균 50 TPS에 가까워진다.

### 들어오는 한도와 나가는 한도를 늘 같은 값으로 두면 안 되는 이유는?

기준이 다르기 때문이다.

들어오는 한도는 우리 서버가 견딜 수 있는 처리량이다. 나가는 한도는 토스가 허용한 처리량 또는 우리가 외부 API에 보내도 된다고 정한 처리량이다.

우리 서버가 초당 100건을 받을 수 있어도 토스 호출은 초당 20건만 허용될 수 있다. 반대로 토스 한도가 높아도 우리 서버 DB나 스레드 풀이 더 낮은 처리량만 견딜 수 있다.

### 나가는 쪽을 즉시 거부하지 않고 토큰이 찰 때까지 대기시키면 어떤 장단점이 있는가?

대기 방식은 사용자가 실패를 덜 볼 수 있다는 장점이 있다. 토큰이 금방 찬다면 요청이 자연스럽게 처리된다.

하지만 서버 스레드를 대기 상태로 붙잡는다. 요청이 많아질수록 대기 중인 스레드가 쌓이고, 전체 서버가 느려질 수 있다. 테스트도 실제 시간에 의존하기 쉬워진다.

이번 구현은 fail-fast를 선택했다. 외부 호출을 보내지 않고 빠르게 실패시키면 서버 자원을 오래 잡지 않고, 사용자에게 잠시 후 다시 시도하라는 명확한 응답을 줄 수 있다.

### `read timeout` 재시도와 429 재시도는 무엇이 다른가?

read timeout은 결과를 모른다. 토스가 처리했지만 응답만 못 받았을 수 있다. 그래서 상태를 `PAYMENT_CHECK_REQUIRED`로 남기고 확인이 필요하다고 처리했다.

429는 호출량 초과다. 일반적으로 요청이 처리되지 않았으니 `Retry-After` 이후 재시도할 수 있다. 다만 결제 승인 요청은 중요한 요청이므로 같은 멱등키를 유지해야 한다.

### Rate Limit과 Circuit Breaker는 무엇이 다른가?

Rate Limit은 호출량을 제한한다. 정상적인 요청이라도 너무 많으면 막는다.

Circuit Breaker는 외부 시스템이 계속 실패할 때 호출 자체를 잠시 끊는다. 요청량이 적어도 외부 API가 계속 5xx나 타임아웃을 내면 더 보내지 않는 것이 낫다.

둘은 보완 관계다. Rate Limit은 "너무 많이 보내지 않기"이고, Circuit Breaker는 "상대가 아픈 동안 더 찌르지 않기"다.
