# 사이클2 설계 고민 기록

## 고민 포인트 1. 예약 취소 시 대기 기능 구현에서 '예약 전환'을 구현하기 위한 데이터 구조

예약 취소 시 대기 1번을 예약으로 전환하는 기능을 구현하려고 한다.

현재 구조에서는 `ReservationSlot`이 코드 레벨의 값 객체로만 존재한다. DB 테이블에는 별도의 슬롯 테이블이 없고, 예약은 `date`, `theme_id`, `time_id` 조합으로 슬롯을 표현한다. 대기는 현재 `reservation_id`를 FK로 가지고 있어 특정 예약 row에 연결된다.

이 구조에서 예약 취소와 대기 전환을 구현할 때 두 가지 선택지를 고민했다.

### 선택지 1: 현재 구조 유지

현재 구조를 유지하면 대기는 계속 `reservation_id`를 바라본다. 예약 취소 후 대기 1번을 예약으로 전환할 때는 기존 예약을 삭제하고 새 예약을 생성한 뒤, 남은 대기들의 FK를 새 예약으로 갱신해야 한다.

순번은 DB에 저장하지 않고 조회 시 계산하고 있으므로, 대기 1번이 삭제되면 남은 대기들의 순번은 다음 조회에서 자연스럽게 다시 계산된다. 따라서 별도의 순번 재정렬 UPDATE는 필요하지 않다.

#### 특징

- 기존 테이블 구조를 유지할 수 있다.
- 예약, 대기, 히스토리 조회 쿼리의 변경 범위가 작다.
- 대기 수를 50개 미만으로 제한한다면 남은 대기 FK 갱신 비용은 크지 않을 가능성이 높다.
- 예약 취소가 아주 빈번한 작업이 아니라면 DB 비용도 정책적으로 감당 가능한 수준으로 볼 수 있다.
- 다만 대기가 "슬롯"이 아니라 "기존 예약"을 바라보므로, 예약 row가 바뀔 때 남은 대기들의 참조도 함께 바뀐다.
- 슬롯 자체를 대표하는 DB row가 없기 때문에 비관적 락을 슬롯 단위로 명확하게 적용하기 어렵다.

### 선택지 2: 슬롯 테이블 도입

슬롯을 별도 테이블로 만들면 예약과 대기가 모두 같은 `slot_id`를 바라볼 수 있다.

이 경우 예약이 취소되고 대기 1번이 새 예약으로 전환되어도 남은 대기들은 계속 같은 슬롯을 참조한다. 
따라서 남은 대기들의 FK를 새 예약으로 갱신할 필요가 없다. 예약으로 전환된 대기 row만 삭제하면 된다.

#### 특징

- 예약과 대기를 모두 "특정 슬롯에 대한 상태"로 표현할 수 있어 도메인 관점으로 해석할 때 더 자연스럽다.
- 기존 예약이 삭제되고 새 예약이 생성되어도 남은 대기들의 참조가 수정될 일이 없다.
- 슬롯 row를 기준으로 비관적 락을 적용할 수 있어 같은 슬롯의 예약 취소, 대기 승인, 대기 신청 흐름을 더 확실하게 직렬화할 수 있다.
- 동시성 문제가 많이 발생하는 도메인(방탈출 예약)임을 고려하면 더 설득력 있는 구조다.
- 대신 테이블 구조 변경이 필요하다.
- 예약 생성, 대기 생성, 예약 취소, 대기 전환, 히스토리 조회 쿼리와 테스트 전반의 수정 비용이 커진다.


### 현재 판단

현재 미션 범위와 구현 비용을 우선하면 기존 구조를 유지해도 합리적이다. 대기 수를 50개 미만으로 제한하고, 예약 취소가 빈번하지 않다는 정책을 둔다면 남은 대기 FK 갱신 비용은 감당 가능한 수준으로 보인다.

하지만 도메인 모델의 안정성과 확장성을 우선하면 슬롯 테이블을 도입하는 쪽이 더 이상적이다. 

대기는 특정 예약 row를 기다리는 것보다 특정 날짜, 테마, 시간 슬롯을 기다리는 것으로 보는 편이 도메인 적으로도 자연스럽다. 

또한 동시성 문제가 많이 발생하는 도메인이라고 가정하면 슬롯 row를 기준으로 비관적 락을 적용할 수 있는 구조가 더 명확하다.

#### 선택 기준
- 미션 범위 안에서 구현 비용을 줄이고 정책적 제약으로 충분히 설명한다면 현재 구조를 유지
- 대기 승인 기능을 핵심 도메인 로직으로 보고, 동시성 제어와 장기적인 확장성을 더 중요하게 본다면 슬롯 테이블을 도입

## 고민 포인트 2. 슬롯을 DB 테이블로 승격할 때의 수정 비용

슬롯을 테이블로 만들면 예약과 대기가 모두 같은 슬롯을 바라볼 수 있어 모델은 더 자연스러워진다. 하지만 현재 구조에서는 변경 비용이 꽤 클 것으로 예상된다.

### 수정 비용이 클 것으로 예상되는 부분

#### DB 스키마

현재는 `reservation`이 `date`, `theme_id`, `time_id`를 직접 가지고 있고, `reservation_waiting`은 `reservation_id`를 FK로 가진다.

슬롯 테이블을 도입하면 `reservation_slot` 같은 테이블이 추가되고, 예약과 대기는 모두 `slot_id`를 바라보도록 바뀐다. 이때 기존 `reservation.date`, `reservation.theme_id`, `reservation.time_id`의 역할이 `reservation_slot`로 이동한다.

수정 비용이 큰 이유는 단순히 컬럼 하나를 추가하는 변경이 아니라, 예약과 대기의 참조 방향 자체가 바뀌기 때문이다.

#### Repository SQL

현재 Repository SQL은 대부분 `reservation`을 중심으로 조인한다. 특히 대기 조회는 `reservation_waiting -> reservation -> theme/reservation_time` 흐름으로 필요한 정보를 가져온다.

슬롯이 테이블로 분리되면 조회 흐름이 `reservation_waiting -> reservation_slot -> theme/reservation_time` 또는 `reservation -> reservation_slot -> theme/reservation_time` 형태로 바뀐다. 히스토리 조회처럼 예약과 대기를 합쳐 보여주는 SQL도 함께 수정해야 한다.

수정 비용이 큰 이유는 슬롯 정보가 여러 SQL에 암묵적으로 퍼져 있기 때문이다.

#### 예약 생성과 대기 생성 흐름

현재 예약 생성은 `date`, `theme`, `time` 조합으로 중복 예약을 확인한 뒤 예약 row를 만든다. 대기 생성은 같은 조합의 예약 row를 찾고, 그 예약에 대기 row를 연결한다.

슬롯을 도입하면 예약과 대기 모두 먼저 슬롯을 찾거나 만들어야 한다. 대기는 더 이상 예약 row를 기다리는 것이 아니라 슬롯을 기다리는 형태가 된다.

수정 비용이 큰 이유는 현재 서비스 흐름이 "슬롯"보다 "예약 row"를 중심으로 구성되어 있기 때문이다.

#### 예약 취소와 대기 전환 흐름

현재 구조를 유지한다면 예약 취소 후 대기 1번을 새 예약으로 전환할 때, 기존 예약을 삭제하고 새 예약을 생성한 뒤 남은 대기들의 `reservation_id`를 새 예약으로 옮겨야 한다.

슬롯을 도입하면 남은 대기들은 계속 같은 `slot_id`를 바라보므로 FK 갱신이 필요 없다. 예약으로 전환된 대기 row만 삭제하면 된다.

이 지점은 슬롯 테이블 도입의 장점이지만, 동시에 현재 구조의 변경 비용이 드러나는 부분이다. 현재 대기는 실제로 슬롯을 기다리는 요구사항을 가지고 있지만, DB 구조에서는 예약 row를 기다리고 있다.

#### 히스토리 조회

현재 내 예약/대기 조회는 예약 테이블과 대기 테이블을 `UNION ALL`로 합치고, 대기 쪽에서는 `reservation_id`를 통해 예약의 날짜, 테마, 시간을 가져온다.

슬롯을 도입하면 대기 이력은 예약 row가 아니라 슬롯을 통해 날짜, 테마, 시간을 가져와야 한다. 예약 이력도 슬롯을 통해 같은 정보를 가져오게 된다.

수정 비용이 큰 이유는 히스토리 조회가 현재 테이블 구조에 강하게 의존하고 있기 때문이다.

#### 테스트 전반

Repository 테스트, API 테스트, 서비스 테스트 fixture는 현재 `reservation`과 `reservation_waiting.reservation_id` 구조를 직접 사용한다. 테스트 데이터 생성 SQL도 이 구조에 맞춰져 있다.

슬롯 테이블을 도입하면 테스트 fixture에서 먼저 슬롯을 만들고, 예약과 대기를 그 슬롯에 연결해야 한다.

수정 비용이 큰 이유는 테스트가 실제 DB 구조를 보호하는 역할을 하고 있기 때문이다. 이 비용은 나쁜 비용이라기보다, 영속성 구조 변경에 따라 자연스럽게 발생하는 비용이다.

### 변경 비용의 원인

이번 변경 비용의 핵심 원인은 **슬롯이라는 개념이 코드 일부에는 있지만, DB 관계와 서비스 흐름에서는 아직 중심 개념이 아니기 때문이다.**

현재 코드에는 `ReservationSlot` 값 객체가 있어 도메인 코드 일부에서는 날짜, 테마, 시간을 묶어 표현한다. 하지만 DB에서는 슬롯이 별도 row로 존재하지 않고, 대기도 슬롯이 아니라 예약 row를 바라본다.

그래서 슬롯을 테이블로 승격하려는 순간 다음 차이가 드러난다.

- 코드 레벨에서는 슬롯을 값으로 표현한다.
- DB 레벨에서는 슬롯이 독립된 참조 대상이 아니다.
- 서비스 흐름에서는 대기 생성과 대기 전환이 예약 row 중심으로 흘러간다.

즉 변경 비용이 큰 이유는 단순히 코드가 많아서가 아니라, 도메인의 중심 개념을 "예약"에서 "슬롯"으로 옮기는 변경이기 때문이다.

### 설계 연습 관점에서 기억할 점

손볼 곳이 많아지는 것 자체가 곧 설계 실패를 의미하지는 않는다. 요구사항이 도메인의 중심 개념을 바꾸는 수준이라면 큰 변경은 자연스럽다.

다만 손볼 곳이 많은 이유를 구분해야 한다.

- 테이블 구조가 바뀌어서 자연스럽게 수정되는 비용인가?
- 아니면 중요한 도메인 개념이 여러 계층에 흩어져 있었기 때문에 생긴 비용인가?

이번 경우에는 둘 다 존재한다. 스키마와 SQL 수정은 자연스러운 비용이다. 하지만 대기가 실제로는 슬롯을 기다리는데 코드와 DB에서는 예약 row를 기다리는 형태였다는 점은 다음 설계에서 기억해야 할 신호다.

다음부터 비슷한 도메인을 설계할 때는 다음 질문을 먼저 던져본다.

- 사용자가 실제로 기다리는 대상은 무엇인가?
- 여러 객체가 함께 바라보는 기준점이 있는가?
- 그 기준점은 단순 값 객체로 충분한가, 아니면 DB에서 독립된 row로 존재해야 하는가?
- 동시성 제어가 필요하다면 어떤 row를 잠글 수 있는가?
- 특정 row가 삭제되거나 새로 생성될 때, 다른 데이터들의 참조가 흔들리는가?

이번 기록의 결론은 슬롯 테이블을 반드시 도입해야 한다는 것이 아니다. 현재 미션 범위와 정책 제약을 고려하면 기존 구조를 유지하는 선택도 가능하다. 다만 슬롯을 테이블로 승격하면 수정 비용은 커지지만, 예약과 대기를 같은 슬롯에 대한 상태로 다룰 수 있고, 동시성 제어의 기준점도 더 명확해진다.
