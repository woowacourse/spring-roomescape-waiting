# 사이클2 설계 고민 기록

## 고민 포인트 1. 예약 취소 시 대기 기능 구현에서 '예약 전환'을 구현하기 위한 데이터 구조

예약 취소 시 대기 1번을 예약으로 전환하는 기능을 구현하려고 한다.

현재 구조에서는 `ReservationSlot`이 코드 레벨의 값 객체로만 존재한다. DB 테이블에는 별도의 슬롯 테이블이 없고, 예약은 `date`, `theme_id`, `time_id` 조합으로 슬롯을 표현한다. 대기는 현재 `reservation_id`를 FK로 가지고 있어 특정 예약 row에 연결된다.

이 구조에서 예약 취소와 대기 전환을 구현할 때 두 가지 선택지를 고민했다.

### 선택지 1: 현재 구조 유지

현재 구조를 유지하면 대기는 계속 `reservation_id`를 바라본다. 예약 취소 후 대기 1번을 예약으로 전환할 때는 기존 예약을 삭제하고 새 예약을 생성한 뒤, 남은 대기들의 FK를 새 예약으로 갱신해야 한다.

순번은 DB에 저장하지 않고 조회 시 계산하고 있으므로, 대기 1번이 삭제되면 남은 대기들의 순번은 다음 조회에서 자연스럽게 다시 계산된다. 따라서 별도의 순번 재정렬 UPDATE는 필요하지 않다.

#### 특징

- 기존 테이블 구조를 유지할 수 있다.
- 예약, 대기, 히스토리 조회 쿼리의 변경 범위가 작다.
- 대기 수를 50개 미만으로 제한한다면 남은 대기 FK 갱신 비용은 크지 않을 가능성이 높다.
- 예약 취소가 아주 빈번한 작업이 아니라면 DB 비용도 정책적으로 감당 가능한 수준으로 볼 수 있다.
- 다만 대기가 "슬롯"이 아니라 "기존 예약"을 바라보므로, 예약 row가 바뀔 때 남은 대기들의 참조도 함께 바뀐다.
- 슬롯 자체를 대표하는 DB row가 없기 때문에 비관적 락을 슬롯 단위로 명확하게 적용하기 어렵다.

### 선택지 2: 슬롯 테이블 도입

슬롯을 별도 테이블로 만들면 예약과 대기가 모두 같은 `slot_id`를 바라볼 수 있다.

이 경우 예약이 취소되고 대기 1번이 새 예약으로 전환되어도 남은 대기들은 계속 같은 슬롯을 참조한다. 
따라서 남은 대기들의 FK를 새 예약으로 갱신할 필요가 없다. 예약으로 전환된 대기 row만 삭제하면 된다.

#### 특징

- 예약과 대기를 모두 "특정 슬롯에 대한 상태"로 표현할 수 있어 도메인 관점으로 해석할 때 더 자연스럽다.
- 기존 예약이 삭제되고 새 예약이 생성되어도 남은 대기들의 참조가 수정될 일이 없다.
- 슬롯 row를 기준으로 비관적 락을 적용할 수 있어 같은 슬롯의 예약 취소, 대기 승인, 대기 신청 흐름을 더 확실하게 직렬화할 수 있다.
- 동시성 문제가 많이 발생하는 도메인(방탈출 예약)임을 고려하면 더 설득력 있는 구조다.
- 대신 테이블 구조 변경이 필요하다.
- 예약 생성, 대기 생성, 예약 취소, 대기 전환, 히스토리 조회 쿼리와 테스트 전반의 수정 비용이 커진다.


### 현재 판단

현재 미션 범위와 구현 비용을 우선하면 기존 구조를 유지해도 합리적이다. 대기 수를 50개 미만으로 제한하고, 예약 취소가 빈번하지 않다는 정책을 둔다면 남은 대기 FK 갱신 비용은 감당 가능한 수준으로 보인다.

하지만 도메인 모델의 안정성과 확장성을 우선하면 슬롯 테이블을 도입하는 쪽이 더 이상적이다. 

대기는 특정 예약 row를 기다리는 것보다 특정 날짜, 테마, 시간 슬롯을 기다리는 것으로 보는 편이 도메인 적으로도 자연스럽다. 

또한 동시성 문제가 많이 발생하는 도메인이라고 가정하면 슬롯 row를 기준으로 비관적 락을 적용할 수 있는 구조가 더 명확하다.

#### 선택 기준
- 미션 범위 안에서 구현 비용을 줄이고 정책적 제약으로 충분히 설명한다면 현재 구조를 유지
- 대기 승인 기능을 핵심 도메인 로직으로 보고, 동시성 제어와 장기적인 확장성을 더 중요하게 본다면 슬롯 테이블을 도입

## 고민 포인트 2. 슬롯을 DB 테이블로 승격할 때의 수정 비용

슬롯을 테이블로 만들면 예약과 대기가 모두 같은 슬롯을 바라볼 수 있어 모델은 더 자연스러워진다. 하지만 현재 구조에서는 변경 비용이 꽤 클 것으로 예상된다.

### 수정 비용이 클 것으로 예상되는 부분

#### DB 스키마

현재는 `reservation`이 `date`, `theme_id`, `time_id`를 직접 가지고 있고, `reservation_waiting`은 `reservation_id`를 FK로 가진다.

슬롯 테이블을 도입하면 `reservation_slot` 같은 테이블이 추가되고, 예약과 대기는 모두 `slot_id`를 바라보도록 바뀐다. 이때 기존 `reservation.date`, `reservation.theme_id`, `reservation.time_id`의 역할이 `reservation_slot`로 이동한다.

수정 비용이 큰 이유는 단순히 컬럼 하나를 추가하는 변경이 아니라, 예약과 대기의 참조 방향 자체가 바뀌기 때문이다.

#### Repository SQL

현재 Repository SQL은 대부분 `reservation`을 중심으로 조인한다. 특히 대기 조회는 `reservation_waiting -> reservation -> theme/reservation_time` 흐름으로 필요한 정보를 가져온다.

슬롯이 테이블로 분리되면 조회 흐름이 `reservation_waiting -> reservation_slot -> theme/reservation_time` 또는 `reservation -> reservation_slot -> theme/reservation_time` 형태로 바뀐다. 히스토리 조회처럼 예약과 대기를 합쳐 보여주는 SQL도 함께 수정해야 한다.

수정 비용이 큰 이유는 슬롯 정보가 여러 SQL에 암묵적으로 퍼져 있기 때문이다.

#### 예약 생성과 대기 생성 흐름

현재 예약 생성은 `date`, `theme`, `time` 조합으로 중복 예약을 확인한 뒤 예약 row를 만든다. 대기 생성은 같은 조합의 예약 row를 찾고, 그 예약에 대기 row를 연결한다.

슬롯을 도입하면 예약과 대기 모두 먼저 슬롯을 찾거나 만들어야 한다. 대기는 더 이상 예약 row를 기다리는 것이 아니라 슬롯을 기다리는 형태가 된다.

수정 비용이 큰 이유는 현재 서비스 흐름이 "슬롯"보다 "예약 row"를 중심으로 구성되어 있기 때문이다.

#### 예약 취소와 대기 전환 흐름

현재 구조를 유지한다면 예약 취소 후 대기 1번을 새 예약으로 전환할 때, 기존 예약을 삭제하고 새 예약을 생성한 뒤 남은 대기들의 `reservation_id`를 새 예약으로 옮겨야 한다.

슬롯을 도입하면 남은 대기들은 계속 같은 `slot_id`를 바라보므로 FK 갱신이 필요 없다. 예약으로 전환된 대기 row만 삭제하면 된다.

이 지점은 슬롯 테이블 도입의 장점이지만, 동시에 현재 구조의 변경 비용이 드러나는 부분이다. 현재 대기는 실제로 슬롯을 기다리는 요구사항을 가지고 있지만, DB 구조에서는 예약 row를 기다리고 있다.

#### 히스토리 조회

현재 내 예약/대기 조회는 예약 테이블과 대기 테이블을 `UNION ALL`로 합치고, 대기 쪽에서는 `reservation_id`를 통해 예약의 날짜, 테마, 시간을 가져온다.

슬롯을 도입하면 대기 이력은 예약 row가 아니라 슬롯을 통해 날짜, 테마, 시간을 가져와야 한다. 예약 이력도 슬롯을 통해 같은 정보를 가져오게 된다.

수정 비용이 큰 이유는 히스토리 조회가 현재 테이블 구조에 강하게 의존하고 있기 때문이다.

#### 테스트 전반

Repository 테스트, API 테스트, 서비스 테스트 fixture는 현재 `reservation`과 `reservation_waiting.reservation_id` 구조를 직접 사용한다. 테스트 데이터 생성 SQL도 이 구조에 맞춰져 있다.

슬롯 테이블을 도입하면 테스트 fixture에서 먼저 슬롯을 만들고, 예약과 대기를 그 슬롯에 연결해야 한다.

수정 비용이 큰 이유는 테스트가 실제 DB 구조를 보호하는 역할을 하고 있기 때문이다. 이 비용은 나쁜 비용이라기보다, 영속성 구조 변경에 따라 자연스럽게 발생하는 비용이다.

### 변경 비용의 원인

이번 변경 비용의 핵심 원인은 **슬롯이라는 개념이 코드 일부에는 있지만, DB 관계와 서비스 흐름에서는 아직 중심 개념이 아니기 때문이다.**

현재 코드에는 `ReservationSlot` 값 객체가 있어 도메인 코드 일부에서는 날짜, 테마, 시간을 묶어 표현한다. 하지만 DB에서는 슬롯이 별도 row로 존재하지 않고, 대기도 슬롯이 아니라 예약 row를 바라본다.

그래서 슬롯을 테이블로 승격하려는 순간 다음 차이가 드러난다.

- 코드 레벨에서는 슬롯을 값으로 표현한다.
- DB 레벨에서는 슬롯이 독립된 참조 대상이 아니다.
- 서비스 흐름에서는 대기 생성과 대기 전환이 예약 row 중심으로 흘러간다.

즉 변경 비용이 큰 이유는 단순히 코드가 많아서가 아니라, 도메인의 중심 개념을 "예약"에서 "슬롯"으로 옮기는 변경이기 때문이다.

### 설계 연습 관점에서 기억할 점

손볼 곳이 많아지는 것 자체가 곧 설계 실패를 의미하지는 않는다. 요구사항이 도메인의 중심 개념을 바꾸는 수준이라면 큰 변경은 자연스럽다.

다만 손볼 곳이 많은 이유를 구분해야 한다.

- 테이블 구조가 바뀌어서 자연스럽게 수정되는 비용인가?
- 아니면 중요한 도메인 개념이 여러 계층에 흩어져 있었기 때문에 생긴 비용인가?

이번 경우에는 둘 다 존재한다. 스키마와 SQL 수정은 자연스러운 비용이다. 하지만 대기가 실제로는 슬롯을 기다리는데 코드와 DB에서는 예약 row를 기다리는 형태였다는 점은 다음 설계에서 기억해야 할 신호다.

다음부터 비슷한 도메인을 설계할 때는 다음 질문을 먼저 던져본다.

- 사용자가 실제로 기다리는 대상은 무엇인가?
- 여러 객체가 함께 바라보는 기준점이 있는가?
- 그 기준점은 단순 값 객체로 충분한가, 아니면 DB에서 독립된 row로 존재해야 하는가?
- 동시성 제어가 필요하다면 어떤 row를 잠글 수 있는가?
- 특정 row가 삭제되거나 새로 생성될 때, 다른 데이터들의 참조가 흔들리는가?

이번 기록의 결론은 슬롯 테이블을 반드시 도입해야 한다는 것이 아니다. 현재 미션 범위와 정책 제약을 고려하면 기존 구조를 유지하는 선택도 가능하다. 다만 슬롯을 테이블로 승격하면 수정 비용은 커지지만, 예약과 대기를 같은 슬롯에 대한 상태로 다룰 수 있고, 동시성 제어의 기준점도 더 명확해진다.

## 고민 포인트 3. 예약 생성과 대기 전환의 동시성 처리 경계

슬롯 테이블을 도입하고 비관적 락을 사용한다면, 같은 슬롯에 대한 모든 요청을 직렬화할 수도 있다. 예를 들어 예약 생성, 대기 신청, 예약 취소 후 대기 전환을 모두 슬롯 row 기준으로 잠그면 각 요청은 항상 최신 슬롯 상태를 보고 판단할 수 있다.

하지만 이 방식은 모든 요청을 같은 락 규칙 안에 넣기 때문에 처리 흐름은 명확해지는 대신 성능 비용이 생긴다. 같은 슬롯에 요청이 몰리면 예약 생성까지 대기해야 하고, 락을 잡는 트랜잭션이 많아질수록 대기 시간과 타임아웃 가능성도 커진다.

### 예약과 대기는 요청 의도가 다르다

고민하면서 예약과 대기는 서로 다른 성격의 데이터라는 점을 다시 보게 되었다.

예약을 시도한 사용자는 "이 슬롯이 예약 가능한 상태"라고 보고 예약을 요청한다. 반면 대기를 시도한 사용자는 "이미 예약된 슬롯이고, 예약이 취소되면 들어갈 수 있다"는 것을 알고 대기 요청을 한다.

즉 두 요청은 원하는 조건 자체가 다르다.

- 예약 요청: 지금 예약 가능한 슬롯을 차지하고 싶다.
- 대기 요청: 지금은 예약되어 있으므로, 취소될 때 순번에 따라 들어가고 싶다.

따라서 예약 생성이 실패했다고 해서 그 요청을 자동으로 대기 데이터 생성으로 이어가는 것은 부자연스럽다. 예약 실패와 대기 신청은 같은 사용자의 연속 행동일 수는 있지만, 도메인적으로는 서로 다른 의도를 가진 별도 요청으로 보는 편이 자연스럽다.

### 선택지 비교

#### 선택지 1: 예약 생성은 유니크 제약으로 방어하고, 대기 전환은 트랜잭션으로 처리

예약 생성은 `reservation.slot_id`에 UNIQUE 제약을 두고, 동시에 같은 슬롯에 예약 생성 요청이 들어오면 하나만 성공하고 나머지는 충돌로 처리한다.

예약 취소 후 대기 1번을 예약으로 전환하는 흐름은 여러 데이터가 함께 바뀌므로 하나의 트랜잭션으로 묶는다.

이 방식에서는 예약 생성 요청이 실패했을 때 자동으로 대기로 전환하지 않는다. 사용자는 예약 실패 안내를 보고, 대기를 원한다면 별도로 대기 신청을 해야 한다.

장점은 예약 생성 흐름이 단순하고, 동시성 제어를 실제로 빈번하게 발생할 수 있는 요청에 맞게 나눌 수 있다는 점이다. 예약 생성은 유니크 제약으로, 대기 신청은 유니크 제약과 조회 시 순번 계산으로 처리한다. 예약 취소 후 대기 전환은 원자성이 중요하므로 트랜잭션으로 묶는다.

#### 선택지 2: 예약 생성과 대기 전환을 모두 슬롯 락으로 직렬화

예약 생성도 슬롯 row를 먼저 잠근 뒤 현재 예약 존재 여부를 확인하고 예약을 생성한다. 예약 취소 후 대기 전환도 같은 슬롯 row를 먼저 잠근다.

이 방식은 같은 슬롯의 상태 판단을 모두 직렬화하므로 흐름은 가장 명확하다. 하지만 예약 생성처럼 단순한 요청까지 락 대기 대상이 되고, 같은 슬롯에 요청이 몰릴 때 처리량이 떨어질 수 있다.

또한 여러 흐름에서 비관적 락을 사용하게 되므로 모든 트랜잭션이 같은 순서로 락을 잡도록 규칙을 강하게 지켜야 한다. 예를 들어 항상 `slot -> reservation -> reservation_waiting` 순서로 접근해야 한다. 이 규칙이 깨지면 데드락 가능성이 커진다.

### 결정

현재 판단은 선택지 1에 가깝다.

예약 생성은 UNIQUE 제약으로 최종 중복을 방어하고, 제약 위반이 발생하면 예약 불가능 상태로 안내한다. 예약 생성이 실패했다고 해서 자동으로 대기 데이터를 만들지는 않는다.

예약 취소 후 대기 1번을 예약으로 전환하는 흐름은 하나의 작업 단위로 본다. 이 흐름에서는 기존 예약 삭제, 새 예약 생성, 승인된 대기 삭제가 함께 성립해야 하므로 트랜잭션으로 묶는다.

다만 현재 기능 범위에서는 비관적 락을 우선 도입하지 않는다.

현재 도메인에서 빈번하게 발생할 가능성이 높은 동시성은 예약 생성과 대기 신청이다. 예약 생성은 `reservation.slot_id` UNIQUE 제약으로 하나만 성공하도록 방어할 수 있고, 대기 신청은 `reservation_waiting(slot_id, name)` UNIQUE 제약으로 같은 이름의 중복 대기를 막을 수 있다. 대기 순번은 저장하지 않고 조회 시 계산하므로, 대기 신청 시점에 순번 재정렬을 위한 락도 필요하지 않다.

예약 취소 후 대기 승격은 기존 예약 삭제, 새 예약 생성, 승인된 대기 삭제가 함께 일어나야 하므로 트랜잭션으로 원자성은 보장한다. 하지만 같은 예약을 동시에 취소하는 상황은 현재 사용자 흐름에서 빈번하다고 보기 어렵다. 따라서 현재 단계에서는 비관적 락을 적용하지 않고, 트랜잭션과 DB 제약으로 우선 정합성을 관리한다.

추후 수동 승인, 관리자 강제 승격, 다중 취소 요청 처리처럼 같은 슬롯의 상태를 동시에 변경하는 유스케이스가 늘어나면 슬롯 row를 기준으로 비관적 락을 도입한다.

이렇게 결정한 이유는 다음과 같다.

- 예약과 대기는 요청 의도가 다르다.
- 예약 실패를 대기 생성으로 자동 연결하는 것은 도메인 흐름상 부자연스럽다.
- 예약 생성은 UNIQUE 제약으로 중복 생성 문제를 막을 수 있다.
- 대기 전환은 여러 데이터를 함께 바꾸는 상태 전이이므로 트랜잭션이 필요하다.
- 현재 기능 범위에서 같은 예약 취소가 동시에 발생할 가능성은 낮으므로 비관적 락은 우선 도입하지 않는다.
- 비관적 락은 수동 승인, 관리자 강제 승격처럼 같은 슬롯을 변경하는 유스케이스가 늘어났을 때 도입한다.

따라서 이번 기능에서는 모든 슬롯 관련 요청을 무조건 직렬화하지 않는다. 빈번한 동시성은 UNIQUE 제약으로 방어하고, 복잡한 상태 전이인 "예약 취소 후 대기 전환"은 트랜잭션으로 묶어 원자성을 보장한다.

### 데드락 우려와 대응 방향

비관적 락을 여러 흐름에 넓게 적용하면 데드락 가능성이 생길 수 있다. 특히 서로 다른 트랜잭션이 같은 슬롯의 데이터를 다루면서 락을 잡는 순서가 다르면 문제가 된다.

예를 들어 예약 생성 흐름이 `reservation`을 먼저 건드린 뒤 `slot`에 접근하고, 예약 취소 후 대기 전환 흐름이 `slot`을 먼저 잠근 뒤 `reservation`을 삭제하거나 생성한다면 서로가 가진 락을 기다리는 상황이 생길 수 있다.

이번 결정에서는 예약 생성 흐름을 비관적 락 대상에서 제외한다. 예약 생성은 `reservation.slot_id` UNIQUE 제약으로 중복 생성을 막고, 제약 위반이 발생하면 예약 불가능 상태로 안내한다. 따라서 예약 생성이 슬롯 row를 잠그고 대기하는 흐름에 들어가지 않으므로 데드락 경우의 수를 줄일 수 있다.

또한 현재 단계에서는 예약 취소 후 대기 전환 흐름에도 비관적 락을 적용하지 않는다. 이 흐름은 트랜잭션으로 묶어 중간 실패를 막되, 같은 예약을 동시에 취소하는 상황의 발생 가능성이 낮다고 판단한다.

나중에 같은 슬롯을 동시에 변경하는 유스케이스가 늘어나 비관적 락을 도입한다면 다음 규칙을 지킨다.

- 항상 `slot` row를 먼저 잠근다.
- 이후 접근 순서는 `slot -> reservation -> reservation_waiting`으로 고정한다.
- 트랜잭션 안에서는 DB 상태 변경만 처리하고, 알림이나 로그 같은 부수 효과는 분리한다.

현재 단계에서는 트랜잭션과 DB 제약으로 필요한 정합성을 먼저 확보한다. 이후 같은 슬롯을 동시에 변경하는 유스케이스가 늘어나 비관적 락을 도입한다면, 위 규칙으로 락 순서를 고정해 데드락 가능성을 줄인다.

### 예약 취소와 대기 신청이 동시에 일어나는 경우

예약 취소로 첫 번째 대기가 예약으로 승격되는 순간에, 다른 사용자가 같은 슬롯에 새로 대기를 신청하는 상황도 생각할 수 있다.

현재 정책에서는 대기 신청은 "이미 예약된 슬롯에 대해 취소를 기다리기 위한 요청"이다. 따라서 대기 신청 시점에 해당 슬롯에 예약이 존재해야 한다.

예약 취소와 대기 신청이 동시에 들어오면 요청 처리 순서에 따라 결과가 달라질 수 있다.

- 대기 신청이 먼저 예약 존재를 확인하면 대기 신청은 성공할 수 있다.
- 예약 취소와 첫 대기 승격이 먼저 끝나면 새 예약자가 생긴 상태이므로 이후 대기 신청도 다시 "예약된 슬롯에 대한 대기"로 성공할 수 있다.
- 아주 짧은 순간에 기존 예약 삭제 후 새 예약 생성 전 상태를 읽는다면 예약이 없다고 보고 대기 신청이 실패할 수 있다.

이 문제를 완전히 없애려면 예약 취소 후 대기 전환과 대기 신청을 같은 슬롯 row 기준으로 직렬화해야 한다. 하지만 현재 단계에서는 이 상황이 핵심 사용자 흐름에서 빈번하다고 보지 않는다. 또한 대기 신청이 실패하더라도 사용자는 다시 조회한 뒤 대기 신청을 재시도할 수 있다.

따라서 현재 구현에서는 예약 취소 후 대기 전환은 트랜잭션으로 묶어 중간 실패를 막고, 대기 신청은 예약 존재 여부와 `reservation_waiting(slot_id, name)` UNIQUE 제약으로 처리한다. 예약 취소와 대기 신청의 완전한 직렬화는 수동 승인, 관리자 강제 승격, 대량 동시 요청처럼 같은 슬롯 상태를 동시에 변경하는 요구가 커졌을 때 비관적 락과 함께 도입한다.

### 트랜잭션 롤백 테스트를 보류한 이유

예약 취소 후 대기 전환은 다음 세 변경이 함께 성립해야 한다.

- 기존 예약 삭제
- 첫 번째 대기 이름으로 같은 슬롯에 새 예약 생성
- 승인된 대기 row 삭제

따라서 이 흐름은 하나의 트랜잭션으로 묶었다.

다만 현재 단계에서는 중간 실패를 일부러 발생시켜 롤백을 검증하는 테스트는 작성하지 않았다. 이유는 현재 구현에서 자연스럽게 실패를 유도할 수 있는 지점이 많지 않기 때문이다.

기존 예약을 삭제한 뒤 같은 슬롯에 새 예약을 생성하므로 `reservation.slot_id` UNIQUE 제약은 보통 실패 지점이 되지 않는다. 승인된 대기 row 삭제도 단순 `DELETE`이고, 해당 row를 참조하는 다른 테이블이 없어 DB 제약 위반을 자연스럽게 만들기 어렵다.

테스트 전용 Fake나 Mock으로 Repository 중간 호출에서 예외를 던지게 만들면 실패 주입은 가능하다. 하지만 Fake는 실제 DB 트랜잭션 롤백을 검증하지 못하고, Repository Mock은 이번 테스트 전략에서 피하기로 한 방향이다. Spring 테스트에서 Spy나 테스트 전용 Bean으로 실패를 주입하는 방법도 있지만, 현재 미션 범위에서는 테스트 장치의 복잡도가 검증 이득보다 커질 수 있다고 판단했다.

그래서 이번 단계에서는 정상 전환 통합 테스트로 결과 상태를 검증하고, 트랜잭션 경계는 코드에 명시하는 수준으로 둔다. 추후 외부 부수효과, 실패 가능한 협력자, 또는 더 복잡한 상태 전이가 추가되어 자연스러운 실패 지점이 생기면 롤백 테스트를 보강한다.

## N+1 쿼리 문제를 알게 된 지점

이번 리뷰에서 `findAvailableTimes()`가 쿼리를 몇 번 실행하는지 확인해보라는 코멘트를 받았다.

처음에는 단순히 슬롯 목록을 가져온 뒤 각 슬롯마다 예약이 있는지 확인하는 흐름이라고 생각했다. 하지만 이 구조가 N+1 쿼리 문제가 될 수 있다는 것을 이번에 알게 되었다.

N+1 쿼리는 먼저 목록을 조회하는 쿼리 1번을 실행한 뒤, 그 목록의 각 항목마다 추가 쿼리 N번이 반복되는 상황을 말한다.

현재 코드에서는 다음 흐름이 그 예시였다.

```java
reservationSlotRepository.findByDateAndTheme(date, themeService.getById(themeId))
        .stream()
        .filter(slot -> reservationRepository.findBySlot(slot).isEmpty())
        .filter(slot -> !slot.isPast(requestedAt))
        .map(ReservationSlot::getTime)
        .toList();
```

여기서 `findByDateAndTheme()`가 해당 날짜와 테마의 슬롯 목록을 한 번 조회한다. 이후 각 슬롯마다 `reservationRepository.findBySlot(slot)`을 호출한다.

따라서 슬롯이 10개라면 슬롯 목록 조회 1번 + 예약 여부 확인 10번으로 총 11번의 쿼리가 발생할 수 있다. 슬롯이 많아질수록 쿼리 수도 같이 늘어난다.

이번 문제의 핵심은 예약 가능 시간 조회의 목적이 "특정 날짜와 테마에서 아직 예약되지 않은 슬롯을 찾는 것"이라는 점이다.

처음에는 Repository에서 예약이 없는 슬롯을 한 번에 조회하는 방법을 생각했다. 하지만 다시 보면 "예약이 없는 슬롯"은 곧 "예약 가능한 슬롯"이라는 유스케이스 판단에 가깝다. 이 판단을 Repository에 넣으면 Repository가 단순 컬렉션 조회를 넘어 예약 가능 정책을 알게 된다.

지금까지 Repository를 컬렉션처럼 쓰려고 했던 방향을 고려하면, Repository에는 필요한 컬렉션을 조회하는 책임만 두고 예약 가능 여부 판단은 서비스에서 하는 편이 더 자연스럽다.

해결 방향은 다음과 같다.

- `ReservationSlotRepository`는 특정 날짜와 테마의 슬롯 목록을 한 번 조회한다.
- `ReservationRepository`는 같은 날짜와 테마의 예약 목록을 한 번 조회한다.
- 서비스는 예약 목록에서 예약된 `slotId`를 `Set`으로 만든 뒤, 슬롯 목록에서 해당 id를 제외한다.

이렇게 하면 쿼리는 슬롯 목록 조회 1번 + 예약 목록 조회 1번으로 고정된다. 동시에 "예약 가능한 슬롯"이라는 판단은 서비스에 남고, Repository는 조건에 맞는 컬렉션을 반환하는 역할을 유지할 수 있다.

## 트랜잭션을 바라보는 기준 변화

처음에는 트랜잭션을 주로 동시성 처리를 위한 도구로 이해하고 있었다. 그래서 예약 생성 흐름에 `@Transactional`을 붙일지 판단할 때도, "동시 예약 생성 경합을 막을 수 있는가"를 중심으로 생각했다.

그 기준에서는 예약 생성에 트랜잭션을 붙이는 효과가 크지 않다고 판단했다. 예약 생성은 최종적으로 `reservation` row 하나를 추가하는 흐름이고, 같은 슬롯에 대한 동시 예약 생성은 `reservation.slot_id` UNIQUE 제약이 최종적으로 막아주기 때문이다. 또한 기본 격리 수준의 트랜잭션만으로는 조회 후 저장 사이에 다른 요청이 먼저 저장하는 경쟁 상태를 막을 수 없다고 보았다.

하지만 리뷰를 통해 트랜잭션은 동시성 제어만을 위한 장치가 아니라, 하나의 유스케이스를 일관된 작업 단위로 선언하는 역할도 한다는 점을 알게 되었다.

예약 생성은 단순히 `INSERT` 한 번을 호출하는 메서드가 아니라 다음 흐름을 하나로 수행한다.

- 테마 조회
- 예약 시간 조회
- 관리자가 열어둔 슬롯 조회
- 같은 슬롯의 기존 예약 여부 확인
- 예약 도메인 생성
- 예약 저장

현재 데이터 변경이 하나뿐이더라도, 이 메서드는 "예약 생성"이라는 하나의 작업 단위다. 이후 검증이나 부가 데이터 변경이 추가될 가능성도 있다. 이때 트랜잭션 경계가 명시되어 있지 않으면, 새로운 변경이 추가되었을 때 일부만 반영되는 구조가 될 수 있다.

따라서 예약 생성 메서드에도 `@Transactional`을 추가했다. 이 결정은 동시 예약 생성 문제를 트랜잭션으로 해결하겠다는 의미가 아니다. 동시 중복 예약의 최종 방어는 여전히 DB UNIQUE 제약이 담당한다. 여기서의 트랜잭션은 예약 생성 유스케이스를 하나의 작업 단위로 명시하기 위한 경계다.

이번 변경을 통해 트랜잭션 경계를 판단할 때 다음 기준을 추가로 가지게 되었다.

- 동시성 제어가 필요한가?
- 여러 데이터 변경이 함께 성립해야 하는가?
- 하나의 유스케이스로서 앞으로 변경이 추가되어도 같은 작업 단위로 유지되어야 하는가?

앞으로는 트랜잭션을 "동시성 처리 도구"로만 보지 않고, 유스케이스의 작업 단위를 드러내는 설계 요소로도 함께 판단한다.

## 요청 시각의 기준 위치 변경

기존에는 서비스 내부에서 `LocalDateTime.now()`를 호출해 예약 생성, 예약 취소, 예약 변경, 대기 생성, 예약 가능 시간 조회의 기준 시각을 만들었다.

처음에는 서비스가 도메인 검증을 수행하므로 서비스 안에서 현재 시각을 만들어도 자연스럽다고 생각했다. 하지만 리뷰를 통해 `요청 시각`이 정확히 무엇을 의미하는지 다시 보게 되었다.

요청 시각은 "서비스 메서드 어느 지점을 지나는 시각"이라기보다, 사용자의 요청을 처리하기 시작한 기준 시각에 가깝다. 그런데 서비스 내부에서 `now()`를 직접 호출하면 다음 문제가 생긴다.

- 같은 요청 안에서도 메서드마다 기준 시각이 조금씩 달라질 수 있다.
- 서비스 테스트가 실제 현재 시각에 의존해 불안정해질 수 있다.
- 과거/미래 판단을 테스트하려면 시간을 직접 제어하기 어렵다.
- 서비스가 현재 시각을 생성하는 책임까지 가지게 되어, 입력을 받아 유스케이스를 수행하는 책임이 흐려진다.

그래서 요청을 받는 경계인 컨트롤러에서 `LocalDateTime.now()`를 한 번 만들고, 서비스에는 `requestedAt`을 파라미터로 전달하도록 변경했다.

변경 후 서비스는 더 이상 현재 시각을 직접 생성하지 않는다. 대신 컨트롤러가 넘긴 요청 기준 시각을 바탕으로 다음 판단을 수행한다.

- 신규 예약이 과거 슬롯인지 확인
- 예약 취소/수정 가능 여부 확인
- 대기 생성 가능 여부 확인
- 예약 가능 시간 조회에서 이미 지난 슬롯 제외

이 변경으로 서비스 테스트에서도 기준 시각을 직접 넘길 수 있게 되었다. 예를 들어 `2026-08-05T12:00:00`을 요청 시각으로 고정하면, 현재 실제 날짜와 무관하게 예약 가능/불가능 판단을 검증할 수 있다.

앞으로 시간에 의존하는 로직을 만들 때는 다음 기준을 따른다.

- "현재 시각"이 필요한 위치가 어디인지 먼저 정한다.
- 사용자 요청을 기준으로 판단하는 시각은 컨트롤러 같은 요청 경계에서 생성한다.
- 서비스와 도메인은 전달받은 기준 시각으로 판단한다.
- 테스트에서는 기준 시각을 직접 넘겨 시간 의존성을 제어한다.
