### 예약 대기 자동 승인 기능

- [x] 예약을 취소하면 해당 슬롯의 1순위 예약 대기가 예약으로 자동 전환된다.
- [x] 자동 전환된 예약 대기는 대기 목록에서 제거된다.
- [x] 남은 예약 대기의 순번은 재계산된다.
- [x] 해당 슬롯에 대기가 없으면 예약만 취소된다.
- [x] 예약 시작 시각 기준 24시간 미만으로 남은 예약은 취소할 수 없다.
- [x] 존재하지 않는 예약은 취소할 수 없다.
- [x] 자동 전환 중 예약 생성 또는 대기 삭제가 실패하면 예약 취소도 롤백된다.

### 정책 결정

1. 예약 대기는 자동 승인 방식을 선택한다.

2. 현재 서비스에는 결제, 본인 인증, 관리자 승인과 같은 추가 확정 절차가 존재하지 않는다.
따라서 사용자가 예약 대기를 신청하는 행위는 해당 슬롯에 빈자리가 생길 경우 예약으로 전환될 수 있음에 동의한 것으로 본다.

3. 자동 승인은 관리자 개입 없이 대기 흐름을 완결할 수 있고, 가장 오래 기다린 사용자에게 우선권을 제공한다.

4. 예약은 시작 시각 기준 24시간 전까지만 취소할 수 있다.

5. 예약 취소와 동시에 대기자가 자동으로 예약 확정되기 때문에, 너무 임박한 시점의 취소는 승격된 대기자에게 준비 시간을 충분히 제공하지 못할 수 있다.
따라서 예약자에게는 취소 가능 시간을 명확히 제한하고(24시간 전), 대기자에게는 자동 승격 이후 예약을 인지하고 준비할 수 있는 최소 시간을 보장한다.

### 트랜잭션 경계

1. 예약 취소와 1순위 대기 자동 승격은 하나의 슬롯 소유권 이전 유스케이스로 본다.

2. 예약 삭제, 대기 기반 예약 생성, 승격된 대기 삭제 중 일부만 성공하면 사용자가 잘못된 중간 상태를 보게 된다.
따라서 이 작업들은 하나의 트랜잭션으로 묶는다.

3. 해당 슬롯에 대기가 없는 경우에는 예약 삭제만 수행한다.
이 경우 승격은 필요한 후행 작업이 아니므로 예약 취소는 정상적으로 완료된다.

### 테스트 전략

1. 컨트롤러 테스트는 HTTP 요청과 응답 계약을 검증한다.
요청 매핑, 요청 값 검증, 응답 상태 코드, 응답 포맷만 확인하고 비즈니스 로직과 DB 동작은 검증하지 않는다.

2. 서비스 테스트는 실제 DAO와 DB를 사용하는 통합 테스트로 작성한다.
예약 취소, 대기 자동 승격, 승격된 대기 삭제가 하나의 유스케이스로 처리되는지 검증한다.

3. 성공 케이스보다 예외 케이스와 경계 케이스를 더 많이 작성한다.
존재하지 않는 예약 취소, 취소 제한 시간이 지난 예약 취소, 대기 없는 예약 취소, 자동 승격 실패 상황을 우선 검증한다.

4. 트랜잭션 테스트는 `@Transactional` 선언 여부를 확인하지 않는다.
확인하려는 것은 Spring의 롤백 기능이 아니라, 내가 의도한 유스케이스 경계에 핵심 변경들을 함께 배치했는지이다.

5. 자동 승격 과정에서 일부러 실패를 발생시켜 두 번째 핵심 변경이 실패했을 때 첫 번째 핵심 변경이 롤백되는지 확인한다.
예를 들어 예약 삭제 후 대기 기반 예약 생성 또는 승격된 대기 삭제가 실패하는 상황을 만들고, 기존 예약과 대기 상태가 잘못된 중간 상태로 커밋되지 않았는지 검증한다.
