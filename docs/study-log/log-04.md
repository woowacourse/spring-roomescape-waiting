## 학습 로그 #4 — 외부 결제 API 연동

**학습 범위**: Toss Payments 승인 API, `RestClient`, 포트와 어댑터, 외부 오류 변환

### 이번 단계의 핵심 흐름

1. 서버가 예약을 `PENDING`으로 만들고 `orderId`, `amount`를 먼저 저장한다.
2. 브라우저는 공개 가능한 클라이언트 키로 Toss 결제 인증을 진행한다.
3. 성공 URL의 금액은 신뢰하지 않고 DB 주문 금액과 비교한다.
4. 금액이 일치할 때만 `PaymentGateway`를 통해 승인 API를 호출한다.
5. 승인 결과가 `DONE`이면 `paymentKey`를 저장하고 예약을 `CONFIRMED`로 전환한다.

`PaymentService`는 `PaymentGateway`, `PaymentConfirmation`, `PaymentResult`만 알고 있다. Toss 요청·응답 DTO와 `{code, message}` 오류 형식은 `TossPaymentGateway` 뒤에 숨겼다. 외부 시스템의 언어를 서비스 내부로 그대로 가져오지 않는 것이 부패 방지 계층의 역할이라는 점을 코드로 확인했다.

### 더 생각해보기

#### 1. 같은 `paymentKey`로 승인을 두 번 호출하면 에러인가, 이미 성공한 결제인가?

로컬 주문이 이미 `DONE`이고 저장된 `paymentKey`도 같다면 새 오류로 취급하지 않고 기존 성공 결과를 반환하는 것이 사용자 새로고침에 안전하다. 현재 구현도 이 경우 외부 API를 다시 호출하지 않는다.

반면 로컬 주문은 `READY`인데 Toss가 `ALREADY_PROCESSED_PAYMENT`를 반환했다면 무조건 성공으로 간주하기 어렵다. Toss에서는 처리됐지만 우리 DB 반영이 실패한 상황일 수 있고, 승인 API 응답만으로는 그 결제가 현재 주문과 같은 금액으로 성공했는지 다시 확인할 수 없기 때문이다. 현재 미션은 승인 API만 호출하므로 이 경우 `PAYMENT_ALREADY_PROCESSED`로 드러내고 운영 확인 대상으로 남긴다. 결제 조회 API까지 허용되는 단계라면 Toss 상태를 조회해 로컬 상태를 복구하는 보상 흐름을 만들 수 있다.

#### 2. 어떤 5xx를 재시도해도 안전한가?

명시적으로 일시 장애를 뜻하는 `FAILED_PAYMENT_INTERNAL_SYSTEM_PROCESSING`은 재시도 후보로 분류했다. 하지만 HTTP 5xx나 연결 타임아웃을 만났다고 즉시 같은 요청을 자동 재시도하는 것은 조심해야 한다. 서버가 응답을 보내기 전에 결제 승인을 끝냈을 수도 있기 때문이다.

따라서 현재 구현은 재시도 가능한 도메인 예외로만 구분하고 자동 재시도하지 않는다. 자동 재시도를 도입한다면 같은 `paymentKey/orderId/amount`를 유지하고, 횟수와 간격을 제한하며, 처리 결과가 불명확한 타임아웃은 조회·대사로 확인하는 절차가 함께 필요하다.

#### 3. 브라우저는 왜 클라이언트 키이고 서버 승인은 왜 시크릿 키인가?

브라우저에 들어간 값은 사용자가 개발자 도구로 볼 수밖에 없다. 그래서 결제창 초기화에는 상점을 식별하지만 승인 권한은 없는 클라이언트 키를 사용한다. 카드 정보와 본인 인증도 Toss 및 카드사 화면에서 처리되어 우리 서버로 들어오지 않는다.

시크릿 키는 실제 승인 권한을 가지므로 서버 환경변수에만 둔다. 브라우저가 시크릿 키를 가지면 사용자가 임의의 금액과 주문번호로 승인 API를 호출할 수 있다. 클라이언트 키와 시크릿 키의 분리는 공개 영역과 신뢰 영역의 경계를 만든다.

#### 4. Toss가 느리거나 연결되지 않으면 우리 서버는 어떻게 되는가?

현재 승인 호출은 동기식이므로 응답을 기다리는 동안 요청 처리 스레드와 DB 트랜잭션이 유지된다. 지연 요청이 쌓이면 스레드·DB 연결 풀이 고갈되고 다른 API까지 느려질 수 있다. 주문행 잠금도 오래 유지되어 같은 결제의 후속 요청이 대기한다.

다음 단계에서는 연결/응답 타임아웃을 명시하고, 외부 호출 전후의 짧은 DB 트랜잭션과 `PROCESSING` 같은 중간 상태를 검토해야 한다. 외부 승인은 성공했지만 DB 반영이 실패하는 구간도 존재하므로, 단순 롤백만 믿지 않고 재처리·대사 가능한 구조가 필요하다.

### 이번 단계에서 얻은 기준

- 외부 응답은 곧바로 도메인 모델이 되지 않는다. 어댑터가 번역한다.
- 외부에서 돌아온 금액은 우리 DB의 주문 금액과 반드시 비교한다.
- “재시도 가능”과 “즉시 자동 재시도”는 같은 의미가 아니다.
- 네트워크 호출과 DB 트랜잭션은 하나처럼 원자적으로 묶이지 않는다.
- 정상 흐름뿐 아니라 중복 콜백, 실패 URL의 누락값, 알 수 없는 외부 코드도 유스케이스의 일부다.
