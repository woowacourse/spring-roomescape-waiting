```
# 실제 학습 과정
- 예약 대기 기능을 구현하면서 조회와 변경의 책임을 분리하고 싶었다. 처음에는 CommandService와 QueryService를 나누는 정도로 접근했다.
- 그런데 리뷰에서 ReservationQueryService가 Reservation 도메인 객체를 그대로 반환하고 있다는 피드백을 받았다.
  Reservation은 cancelBy(), isOwnedBy() 같은 쓰기 행위를 가지고 있기 때문에,
  읽기 경로에서 그대로 반환하면 읽기 모델과 쓰기 모델을 완전히 분리한 것은 아니라는 내용이었다.

- 이 피드백을 계기로 CQRS를 더 찾아보았다.
  CQRS는 단순히 메서드나 서비스를 command/query로 나누는 것이 아니라,
  필요하다면 쓰기 모델과 읽기 모델 자체를 다르게 가져가는 구조라는 것을 알게 되었다.
- 쓰기 모델은 상태 변경과 도메인 규칙을 지키는 데 적합해야 하고,
  읽기 모델은 화면이나 응답에 필요한 데이터를 빠르고 명확하게 전달하는 데 적합해야 한다.
- 예약 대기 목록에서는 대기 순서(order)가 필요했다.
  하지만 이 order는 예약 대기를 저장할 때 의미 있는 값이 아니라,
  조회 시점에 같은 예약에 걸린 대기들을 정렬해서 계산해야 의미가 생기는 값이었다.
- 그래서 ReservationWaiting 도메인에 order를 억지로 넣기보다,
  ReservationWaitingWithOrder라는 읽기 모델을 따로 만들었다.
- waiting_order를 계산하는 SQL은 ReservationWaitingQueryRepository / ReservationWaitingQueryJdbcRepository에 두었다.
  저장용 Repository가 아니라 조회용 Repository에서 projection을 만들어 반환하도록 분리했다.
- ReservationTimeAvailability도 시간 도메인 자체가 아니라,
  특정 날짜와 테마에서 예약 가능 여부를 함께 보여주기 위한 읽기 모델로 바라보았다.

- 이 과정에서 projection, read model, write model, thin read layer, event sourcing 같은 키워드를 만났다.
- 특히 event sourcing이나 별도 read DB까지 찾아보면서,
  CQRS를 제대로 크게 적용하면 app server와 DB까지 command/query로 나누는 구조도 가능하다는 것을 알게 되었다.
- 하지만 현재 프로젝트는 하나의 DB와 하나의 app server를 사용하는 작은 단계의 미션이다.
  따라서 CQRS를 끝까지 적용하는 것은 지금 문제에 비해 과한 구조라고 판단했다.
- 이번에는 다음 정도만 가져가기로 했다.
  1. 쓰기 작업과 읽기 작업을 서비스 수준에서 구분한다.
  2. 조회에 특화된 데이터가 필요하면 별도 읽기 모델을 만든다.
  3. 읽기 모델을 만들기 위한 SQL은 QueryRepository에 둔다.
  4. 별도 DB, 이벤트 소싱, 완전한 read/write 모델 분리는 지금은 적용하지 않는다.

# 후기
- 처음에는 CQRS라는 키워드를 만나자마자 더 세련된 구조를 끝까지 적용해보고 싶었다.
- 하지만 찾아볼수록 현재 구조에서는 CQRS의 큰 이득을 모두 얻기 어렵다는 것을 알게 되었다.
- 특히 하나의 DB를 쓰는 상황에서는 읽기 DB와 쓰기 DB를 분리했을 때의 성능, 확장성, 동기화 전략 같은 장점을 제대로 가져가기 어렵다.

- 그래도 이 학습이 의미 없지는 않았다.
- "저장할 때 의미 없는 값이 읽을 때 의미 있어진다면, 같은 도메인 모델에 억지로 넣지 않아도 된다"는 기준을 얻었다.
- 또한 읽기 모델은 도메인을 대체하는 것이 아니라, 특정 조회 목적에 맞춰 만든 결과 모델이라는 것도 이해했다.
- 지금은 완전한 CQRS가 아니라 CQRS의 일부 아이디어를 현재 문제에 맞게 가져온 상태라고 보는 것이 더 정확하다.
```

```
# 효과적이었던 것
- 리뷰에서 받은 키워드를 바로 코드에 전부 적용하기보다, 현재 문제와 연결해서 본 것
```

### 4. AI 피드백

- 자신의 학습 전략에 대해 AI 학습 전문가에게 피드백을 요청하고, 유용했던 제안 1가지 이상 기록

```
# 나의 전략 분석
- 이번에는 리뷰에서 받은 키워드를 계기로 CQRS를 찾아보았다.
- 처음에는 구조 자체가 멋있어 보여서 더 크게 적용하고 싶어졌다.
- 하지만 현재 시스템의 규모와 요구사항을 다시 보니,
  완전한 CQRS보다 읽기 모델과 query repository 정도만 가져오는 것이 맞다고 판단했다.

# AI 학습 전문가의 피드백
- CQRS는 "적용했는가, 안 했는가"보다 "어느 문제를 해결하기 위해 어느 깊이까지 분리했는가"가 중요하다.
- 작은 프로젝트에서는 command/query 이름을 붙이는 것보다,
  읽기와 쓰기에서 모델의 책임이 실제로 달라지는 지점을 찾는 것이 더 실용적이다.
- 읽기 모델이 필요한 대표적인 신호는 다음과 같다.
  1. 조회 화면에 필요한 값이 도메인 상태와 다르다.
  2. 조회 시점에 계산되는 값이 있다.
  3. 여러 테이블을 조합한 결과가 응답에 바로 필요하다.
  4. 쓰기 도메인의 행위가 읽기 경로에서는 불필요하거나 오히려 의미를 흐린다.

# 유용했던 제안
- CQRS를 적용할 때 다음 순서로 판단한다.
  1. 지금 읽기와 쓰기의 목적이 실제로 다른가?
  2. 같은 모델을 쓰면 의미가 흐려지는 값이 있는가?
  3. 조회 전용 SQL이나 projection을 두면 코드가 더 명확해지는가?
  4. 별도 DB나 이벤트 기반 동기화가 필요한 정도의 규모인가?
- 이번 프로젝트는 1~3번은 해당하지만, 4번은 해당하지 않는다고 판단했다.

# 적용 결과
- ReservationWaiting 도메인에는 저장과 규칙에 필요한 값만 남겼다.
- 조회에 필요한 order는 ReservationWaitingWithOrder에서 표현했다.
- order 계산은 QueryRepository의 SQL에서 처리했다.
- 완전한 CQRS가 아니라, 현재 문제에 맞는 얇은 읽기 계층을 만든 것으로 정리했다.
```
