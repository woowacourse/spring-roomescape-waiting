# 검증과 검사

> 각각은 무엇을 확인하는가

## 배경

> `SlotRepository` 의 findAll()  
> 실제 `JDBCRepository` 에선 도메인 구조와 불일치하는 단순 조회 쿼리 적용  
> 테스트는 FakeRepository 사용해서 통과, 실제 로직에선 에러 발생  
> 정의된 `계약` 인 `SlotRepository` 인터페이스의 구현이 일치하지 않는 상황
>
> 테스트가 무엇을 `검사`하고 있는가?  
> 무엇을 `검사`하지 않고(누락하고) 있는가?  
> 테스트가 `검사`할 것과 도메인이 스스로 `검증`할 것은 어떻게 다른가?

### `Domain Object`

- 도메인 객체는 스스로 상태의 일관성과 유효성을 보장하는 `무결성` 을 보호한다
- `무결성` : 객체의 상태가 모순 없이 항상 올바르고 일관된 상태를 유지하는 것
- 도메인 객체는 내부에 `validate` 로직을 가진다
- 외부에서 들어온 데이터의 유효성을 검증 `validate` 한다.
- 유효성 : 데이터(상태)가 비즈니스 규칙이나 형식에 부합하는 것

### `Test`

- 테스트는 `도메인의 무결성`과 `기술적 정합성`, `시스템의 유효성`을 검사한다
- 단위 테스트로 `도메인 객체의 무결성`을 검사
- 통합 테스트로 도메인 모델 간/외부 인프라와 협력의 `기술적 정합성`을 검사
- E2E 테스트로 비즈니스 로직 준수, 전체 `시스템의 유효성`을 검사

> `도메인 객체`는 `스스로`, `자신의 상태 무결성`을 검증/보호
>
> `테스트`는 `외부`에서,
>
> - 단위 테스트로 `도메인의 무결성` 을
> - 통합 테스트로 `기술적인 정합성` 을
> - E2E 테스트로 `시스템의 유효성` 을
>
> 검사

### 누락된 검사

FakeRepository 의 로직으로 다른 기능의 동작만 검사,  
FakeRepository 와 JdbcRepository 의 동일한 동작(계약)을 검증하지 않음.

Repository 구현체의 findAll 이 올바르게 값을 꺼내주고 있는지,  
도메인 모델과 인프라간의 `정합성` 을 검증하는 통합 테스트가 누락.
