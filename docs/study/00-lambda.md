# 람다식과 상수

## 배경

> rowMapper 를 메서드로 두면 실행시마다 새로운 인스턴스를 생성하지 않나?
>
> static 상수로 두면 더 효율적인거 아닌가?
>
> 람다식이 싱글톤으로 재사용될 수 있다?

### 시도 1 - 정적 상수로 분리

```java
private static final RowMapper<Waiting> WAITING_ROW_MAPPER = (rs, rowNum) -> new Waiting(...);
```

장점?

- 클래스가 로드될 때 단 한 번만 생성되고, 재사용됨
- GC 호출을 최소화

단점?

- `WAITING_ROW_MAPPER` 내부 호출 메서드는 static 이어야 한다

> 재호출이 많고 유연하지 않은 구조에선 적극 활용 가능

추가

- 구구: 이게 상수인가요? 딱 보고 이게 왜 여기있는지 의문이 생길 것 같은데요?

--- 

### 시도 2 - `인스턴스 필드` 로 변경

기존의 호출될 때마다 람다 인스턴스를 생성해 반환한다.  
배경에서 언급한 바와 같이 람다식이 싱글톤 형식으로 재사용될 수도 있지만(JVM)  
람다식 내부에 어떠한 외부 참조(this. 포함)가 없어햐 한다는 명확한 조건이 있다.

그렇다면 메서드가 아닌 필드로 둔다면?  
스프링 빈에 등록된 `Repository` 는 싱글톤으로 재사용된다.  
그렇다면 아예 내부에 `상수` 가 아닌 `필드` 로 두고 재사용하면  
그게 제일 깔끔한 수준, 효율적인 방식 아닌가?

상수가 아닌, 클래스에 귀속된 필드로서  
클래스가 생성될 때 종속되어 하나만 생성되고,  
내부 메서드에서 재사용되다 클래스가 사라질 때 자연스럽게 소멸.

```java
private final RowMapper<Reservation> reservationrowMapper = (rs, rowNum) -> new Reservation(
```

> 근데뭐 상수랑 다를바없지않나?  
> ..  
> 연계된 모든 메서드에 static 을 박을 필요도 없고 정적 상수랑 인스턴스 필드가 결코 같진 않을거같은데

추가

- 생성되는 영역과 생명주기가 다르고
- 다형성과 유연성의 수준이 다르다

또 추가

> 지금같이 객체 기반 참조에선  
> 저장소의 매핑 로직이 중복되는데  
> 코드의 중복은 제거하면서 로직은 공유하게 하는 효과적인 방식은?

- 매퍼 클래스?
    - 결국 스태틱 메서드를 선택하는거에다가 모든 레포가 매퍼에 의존, 강결합
- 추상 클래스?
    - 공통 매핑을 부모 클래스에 넣고 상속받아서 개별 매핑을 구현하기?
    - 개별 매핑이 생길때마다 부모 클래스로 넘길거? 너무 확장성이 떨어지는데
- 컴포지션?
    - 모르던 개념(정확히 모르고 사용하던),
    - 서비스에 레포를 주입해 활용하듯 매퍼를 주입해서 공통 매핑을 재활용하기
    - 추상 클래스랑 비슷한 결이 느껴지지만 `상속보단 조합` 측면에서 바람직

또또 추가

매퍼도 추상화해서 매핑 로직 교체가 원활하도록 해야되는데  
지금처럼 ResultSet 을 가지고 매핑하는 이상 매퍼가 JDBC에 강결

차피 JPA되면 알아서 해결해주지않나?
