# Entity

### 왜 엔티티는 `NoArgsConstructor` 를 필요로 하는가

하이버네이트는 해당 객체의 생성자가 어떤 순서로 필드와 매핑되는지 알 수 없다.  
순서도 파라미터도 구조도 사용자가 만든 생성자를 파악하고 분석해서 사용하지 않는다.

그렇기에 하나의 패턴으로 생성 방식을 단순화한다.  
그저 리플렉션(런타임?)으로

- NoArg 생성자를 찾아서 호출
- 생성된 객체의 각 필드를 리플렉션으로 직접 세팅 (setter / 생성자 실행 X)

리플렉션이 접근 제어자를 우회하기에 필드의 접근 제어가 private 이어도 상관 없다.  
마찬가지로 setter 나 생성자 흐름을 타지 않기에 setter 도 생성자도 필요 없다.  
하지만 객체 자체를 생성하기 위한 생성자 호출은 필요하기에,  
**NoArgsConstructor 가 필요하다.** 추가로 해당 생성자의 접근 제어자는 protected 권장.

#### 왜 protected?

Lombok 의 `@NoArgsConstructor` 사용 시 기본값인 public 으로 생성된다.  
빈 도메인을 생성하는 것은 도메인의 무결성 측면에서 매우 지양해야 할 작업.  
protected면 그래도 도메인간의 로직에서만 호출될 테니 상대적으로 안전하니까..?
