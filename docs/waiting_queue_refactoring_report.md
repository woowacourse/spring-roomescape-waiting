# 대기열 순번 조회 아키텍처 개선 및 성능 실험 보고서

본 보고서는 예약(Reservation) 및 대기(Waiting) 도메인의 아키텍처적 경계를 명확히 하고, 대기열 우선순위 정렬 정책의 유연한 확장을 도모하기 위해 진행한 리팩터링 및 성능 실험의 결과를 담고 있습니다.

---

## 1. 아키텍처 개선 배경

### 기존 구조의 문제점 (1-Query 방식)
기존에는 사용자의 예약/대기 목록을 한 번에 보여주면서 대기 순번을 계산하기 위해 **SQL Window Function (`ROW_NUMBER() OVER`)**을 활용한 단일 복잡 쿼리를 실행했습니다.
* **비즈니스 로직의 DB 결합**: 대기 순번 정렬 정책(예: 선착순, VIP 우선 대기 등)은 엄연히 자바 애플리케이션 영역의 비즈니스 규칙이지만, SQL 내부의 `ORDER BY`에 하드코딩되어 강결합을 형성했습니다.
* **애그리거트 경계 위반**: 예약 도메인의 리포지토리나 DAO가 대기 도메인의 테이블과 직접 결합하여 조인 연산을 함으로써, 도메인 간 격리가 불가능했습니다.

### 개선 구조 (3-Query 방식 + 전략 패턴)
* **정렬 정책 분리 (`WaitingQueuePolicy`)**: 정렬 규칙을 `Comparator<ReservationWaiting>`를 구현한 전략 패턴으로 이관했습니다. 정책이 변경되어도 기존 코드는 변경되지 않는 **OCP(개방-폐쇄 원칙)**를 준수합니다.
* **도메인 결합 해제**: 예약 및 대기 리포지토리는 오직 자신 도메인의 데이터만 조회(SELECT)하고, 조율(Orchestration) 및 순번 매기기는 서비스 레이어(`ReservationService`)에서 수행하도록 역할을 명확히 분리했습니다.
* **Batch Query 도입 (`findAllBySlots`)**: 쿼리가 쪼개지면서 발생할 수 있는 N+1 문제를 방지하기 위해 유저가 대기 중인 모든 슬롯의 대기자 목록을 단 1번의 쿼리로 모아서 긁어옵니다.

## 2. 성능 실험 설계

가상 인메모리 DB인 H2 환경에서 실제 상용 클라우드 환경(예: AWS RDS)의 물리적인 네트워크 레이턴시를 반영하기 위해, **DB 쿼리당 1ms의 인위적인 네트워크 IO 지연(RTT)**을 추가하여 비교 실험을 진행했습니다.

### 데이터 모델링 조건
* **전체 대기열 규모**: 100,000건 (대기 테이블 전체 크기)
* **슬롯 분포**: 20,000개의 서로 다른 시간대/테마 슬롯에 분산 적재 (슬롯당 약 5명의 대기자)
* **조회 유저 ("브라운")**: 2개의 슬롯에 대기를 걸어놓은 상태
* **인덱스 적용**: 
  * `(reservation_date, time_id, theme_id)` 복합 인덱스 적용
  * `(name)` 인덱스 적용

## 4. 핵심 결과 분석 (왜 30배 차이가 발생하는가?)

### A. Window Function의 한계 (1-Query)
기존 SQL 구조는 서브쿼리 내부에서 `ROW_NUMBER() OVER(PARTITION BY ... ORDER BY id)` 연산을 계산합니다.
* 데이터베이스는 외부 `WHERE name = ?` 조건(이름 필터링)을 적용하기 전에, **전체 10만 건 데이터를 슬롯별로 그룹핑하고 정렬하여 순번을 먼저 부여**해야만 합니다.
* 10만 건이라는 대량의 데이터에 대해 정렬 및 테이블 전체 스캔 연산이 요청 시마다 발생하므로 극심한 성능 병목을 초래합니다.

### B. 인덱스 기반 Batch 조회 & 정렬 (3-Query)
3-Query 방식은 필요한 생 데이터만 정확히 타겟팅하여 가져온 뒤 가벼운 인메모리 연산을 수행합니다.
* **1단계 (이름 검색)**: `WHERE name = '브라운'` 조건으로 인덱스를 타서 브라운의 대기 목록만 즉시 가져옵니다. (단 2건)
* **2단계 (슬롯 전체 대기자 검색)**: 브라운이 예약 대기 중인 2개의 특정 슬롯 데이터를 복합 인덱스를 통해 딱 1번의 쿼리로 정합성 높게 가져옵니다. (슬롯당 5명씩 총 10건)
* **3단계 (인메모리 연산)**: 자바 컬렉션에서 단 10건에 대해서만 정렬(`WaitingQueuePolicy.compare`)하고 순번을 매깁니다.
* **결과**: 테이블의 총 크기가 10만 건이든 100만 건이든 관계없이 인덱스를 타고 항상 최소한의 데이터(몇십 건)만 다루기 때문에 **시간 복잡도 O(1) 수준의 압도적인 성능 안정성**을 갖게 됩니다.

## 5. 결론 및 제언

* **성능과 구조의 동시 개선**: 흔히 쿼리 수가 많아지면 무조건 성능이 불리할 것이라 직관적으로 판단하기 쉽지만, 본 실험을 통해 **적절한 인덱스 기반의 쪼개진 쿼리(3-Query)가 대량의 데이터를 풀스캔 정렬하는 무거운 단일 쿼리(1-Query)보다 30배 이상 압도적으로 우수함**이 입증되었습니다.
* **유지보수성 극대화**: 정렬 정책이 자바의 `WaitingQueuePolicy` 전략 인터페이스로 독립되었기 때문에, 향후 "VIP 회원 우선 순번 부여", "마일리지 보유량 순 부여" 등의 정책이 유연하게 추가되더라도 **기존 조회 SQL은 단 한 줄도 수정하지 않고 새로운 Comparator 전략 구현체만 주입하여 대응이 가능**합니다.
