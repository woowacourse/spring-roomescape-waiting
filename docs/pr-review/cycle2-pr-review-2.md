## ✅ 리팩토링 할 것 목록

### 값 객체와 도메인 불변식

- [x] **1. `ReservationName` 값 객체의 동등성 기준을 정의한다.**
  - 받은 리뷰: `ReservationName`이 값 객체라면 `equals`, `hashCode`를 적용할 수 있는지 확인하라.
  - 판단 기준: 값 객체는 식별자가 아니라 값 자체로 동등성을 판단하는 객체다. 현재 `Reservation.hasName()`처럼 내부 문자열 값을 직접 꺼내 비교하는 흐름이 있다면, 값 객체의 동등성 책임이 충분히 표현되지 않은 상태일 수 있다.
  - 해결 방향: `ReservationName`에 `equals`, `hashCode`를 구현하고, 이름 비교가 필요한 도메인 로직에서는 문자열 비교 대신 값 객체 비교를 사용하도록 정리한다.
  - 리뷰 코멘트: 말씀해주신 대로 값 객체라면 값 자체로 동등성을 판단하는 것이 자연스럽다고 생각해 `ReservationName`에 `equals`, `hashCode`를 추가했습니다. `Reservation.hasName()`도 문자열 값을 직접 꺼내 비교하지 않고 `ReservationName`끼리 비교하도록 수정했습니다.

- [x] **2. `Reservation.PAST_RESERVATION_MESSAGE`가 public 상수로 열려 있어야 하는지 검토한다.**
  - 받은 리뷰: `PAST_RESERVATION_MESSAGE`가 public으로 열려 있는 이유를 확인하라. 이 문자열만 필드로 정의한 이유와 테스트 때문에 프로덕션 코드가 바뀐 것은 아닌지 확인하라.
  - 판단 기준: 도메인은 외부 테스트나 응답 메시지 편의를 위해 내부 메시지를 공개할 필요가 없다. 예외 메시지 자체를 테스트하기 위해 프로덕션 코드의 접근 범위를 넓혔다면 테스트가 구현 세부사항에 과하게 의존하고 있는 신호일 수 있다.
  - 해결 방향: 메시지 상수의 접근 범위를 낮추거나 상수 자체를 제거한다. 테스트는 정확한 문자열보다 예외 타입이나 도메인 행위 실패 여부를 검증하도록 조정한다.
  - 리뷰 코멘트: public 상수로 열어둔 이유는 서비스에서 과거 예약 예외를 구분하고 테스트에서 메시지를 검증하기 위해서였는데, 말씀해주신 것처럼 도메인 내부 메시지를 외부 편의를 위해 공개한 형태였던 것 같습니다. 문자열 비교 대신 `PastReservationException` 타입으로 의미를 구분하도록 바꾸고, 메시지 상수는 private으로 숨겼습니다. 테스트도 구체 문자열에 의존하지 않도록 수정했습니다.

- [x] **3. `Reservation`의 생성 경로별 검증 차이를 불변식 관점에서 다시 정리한다.**
  - 받은 리뷰: 어떤 생성자는 `id`를 검증하고, 어떤 생성자는 예약 가능 여부만 검증한다. `Reservation` 객체의 불변식이 무엇인지, 어떤 생성 경로에서도 일관되게 지켜지는지 확인하라.
  - 판단 기준: `Reservation`에는 두 종류의 규칙이 섞여 있었다. 첫째, 모든 예약이 항상 만족해야 하는 공통 불변식이다. 이름, 슬롯, 생성 시각은 비어 있으면 안 되고, 저장소에서 복원된 예약이라면 id도 있어야 한다. 둘째, "사용자가 새 예약을 만든다"는 유스케이스에서만 필요한 정책이다. 새 예약은 과거 슬롯에 만들 수 없다.
  - 문제점: 기존에는 id가 있는 생성자와 id가 없는 생성자의 검증 차이가 생성자 오버로딩에 숨어 있었다. 그래서 이 차이가 "저장소에서 복원한 예약"과 "신규 예약 생성"이라는 의도된 맥락 차이인지, 단순히 검증이 빠진 것인지 코드만 보고 알기 어려웠다. 특히 대기 승급에서 새 예약을 만들 때 과거 슬롯이면 막히는 것이 의도된 정책인지 우연히 발생한 결과인지도 명확하지 않았다.
  - 해결 방향: 저장소에서 복원된 예약은 `new Reservation(id, ...)` 생성자로 공통 불변식만 검증한다. 반면 신규 예약 생성은 `Reservation.reserve(...)`라는 도메인 행위 이름을 가진 생성 경로로 분리하고, 여기에서 과거 슬롯 예약 불가 정책을 검증한다. 대기 승급도 새 예약을 만드는 행위이므로 `ReservationWaiting.toReservation()`에서 `Reservation.reserve(...)`를 사용하게 한다.
  - 리뷰 코멘트: 말씀해주신 내용을 보고 `Reservation`의 불변식과 신규 예약 생성 정책이 생성자 오버로딩 안에 섞여 있어 의도가 흐려졌다고 판단했습니다. 그래서 모든 예약이 공통으로 가져야 하는 값 검증은 저장소 복원용 생성자에서 보장하고, 사용자가 새 예약을 만드는 정책은 `Reservation.reserve(...)`라는 이름의 생성 경로로 분리했습니다. 대기 승급 역시 새 예약을 만드는 흐름이므로 같은 `reserve` 경로를 타도록 정리했습니다. 이를 통해 생성 경로마다 검증이 다른 이유가 저장소 복원과 신규 예약 생성의 맥락 차이로 드러나도록 수정했습니다.

- [x] **4. `ReservationName`을 받는 생성자를 둘지 검토한다.**
  - 받은 리뷰: `ReservationName`을 받는 부생성자를 만들어도 좋겠다.
  - 판단 기준: `Reservation` 내부 필드가 `ReservationName`이라면 생성자에서 매번 `String`을 받아 값 객체로 변환하는 방식은 불필요한 변환을 반복하게 만든다. 특히 다른 도메인 객체가 이미 검증된 `ReservationName`을 가지고 있다면 이를 그대로 전달할 수 있어야 한다.
  - 해결 방향: 외부 입력을 받는 경로는 `String` 생성자를 유지하되, 도메인 내부 협력에서는 `ReservationName`을 직접 받는 생성자 또는 정적 생성 경로를 제공할 수 있는지 검토한다.
  - 리뷰 코멘트: `Reservation`이 내부적으로 `ReservationName`을 필드로 가지고 있으면서도 생성 시에는 항상 `String`으로만 받다 보니, 이미 검증된 값 객체를 다시 문자열로 풀어 전달하는 흐름이 생길 수 있었습니다. `ReservationName`을 받는 생성자를 추가하고 기존 `String` 생성자는 값 객체 생성 후 위임하도록 정리했습니다.

### 슬롯 도메인과 테이블 승격

- [x] **5. `ReservationSlot` 테이블 승격이 요구사항으로부터 나온 설계인지 다시 검토한다.**
  - 받은 리뷰: `theme`, `time`은 이미 관리자가 설정하는 요구사항이 있다. 슬롯 테이블이 요구사항으로부터 파생된 것인지, 테이블 생성을 정당화하기 위해 요구사항을 만든 것인지 구분하라.
  - 판단 기준: `date-time-theme` 조합이 예약과 대기에서 공유된다는 사실만으로 별도 테이블 분리의 근거가 되지는 않는다. 테이블 분리는 중복 제거보다 생명주기, 정책, 변경 가능성, 독립적인 식별 필요성이 있을 때 더 설득력이 생긴다.
  - 해결 방향: `ReservationSlot`을 유지한다면 "관리자가 특정 날짜/테마/시간을 예약 가능 단위로 연다"는 생명주기가 실제 요구사항에 자연스럽게 연결되는지 설명한다. 그렇지 않다면 슬롯 테이블 없이 `date-theme-time` 조합을 기준으로 예약과 대기를 묶는 설계가 더 단순한지 비교한다.

- [x] **6. `ReservationWaiting`이 `Reservation`이 아니라 `ReservationSlot`을 참조하도록 바꿀지 검토한다.**
  - 받은 리뷰: 테이블을 별도로 분리했는데 `Waiting`이 `Slot`을 보지 않고 `Reservation`을 보는 이유를 확인하라. 실제로는 `reservation.getSlot()`을 꺼내 쓰고 있다.
  - 판단 기준: 대기가 기다리는 대상이 "특정 예약 row"인지 "특정 슬롯"인지가 핵심이다. 예약 취소 후 새 예약이 생성되는 구조라면 대기가 기존 예약 row를 참조하는 모델은 참조 갱신 문제를 만든다.
  - 해결 방향: 대기의 대상이 슬롯이라면 `ReservationWaiting`이 `ReservationSlot`을 직접 참조하도록 모델을 조정한다. 이 경우 예약 존재 여부 검증은 서비스나 더 큰 도메인 객체가 담당하고, 예약 취소/승급 시 대기 row의 FK를 옮기는 문제를 제거할 수 있는지 확인한다.
  - 리뷰 코멘트: 말씀해주신 것처럼 슬롯을 별도 테이블로 둔 현재 구조에서는 대기가 특정 예약 row를 바라보는 것보다 슬롯을 직접 바라보는 편이 더 자연스럽다고 판단했습니다. `ReservationWaiting`이 `ReservationSlot`을 필드로 가지도록 변경했고, Repository 조회도 예약 테이블 조인 없이 슬롯 정보만 복원하도록 정리했습니다. 다만 "예약이 있어야 대기할 수 있다", "이미 예약한 사람은 대기할 수 없다"는 유스케이스 검증은 예약 조회가 필요하므로 서비스에서 확인하도록 유지했습니다.

### 트랜잭션과 시각 의존성

- [x] **7. 예약 생성 흐름의 트랜잭션 경계를 명시할지 다시 검토한다.**
  - 받은 리뷰: 현재는 단일 저장처럼 보여도 향후 검증 로직이나 부가 작업이 추가될 수 있다. `save`가 하나의 예약 생성 작업이라면 트랜잭션 경계를 명시하는 것이 더 안전하지 않은지 확인하라.
  - 판단 기준: 트랜잭션은 단순히 INSERT 원자성만을 위한 장치가 아니라 하나의 유스케이스를 일관된 작업 단위로 드러내는 역할도 한다. 현재 코드가 단일 변경만 수행하더라도, 예약 생성이라는 유스케이스 경계를 명시하는 편이 이후 변경에 더 안전할 수 있다.
  - 해결 내용: 예약 생성 서비스 메서드에 `@Transactional`을 추가했다. 중복 예약 동시성은 여전히 UNIQUE 제약이 최종 방어선이라는 점을 유지하되, 트랜잭션 도입 이유는 "현재 동시성 해결"이 아니라 "예약 생성 유스케이스의 작업 단위 명시"로 정리했다.
  - 리뷰 코멘트: 처음에는 트랜잭션을 주로 동시성 제어 관점에서 생각해서, 예약 생성은 UNIQUE 제약으로 충분하다고 판단했습니다. 그런데 말씀해주신 것처럼 트랜잭션은 단순히 동시성 처리나 INSERT 원자성만이 아니라 하나의 유스케이스를 일관된 작업 단위로 드러내는 의미도 있다고 이해했습니다. 그래서 예약 생성 흐름에도 `@Transactional`을 명시하도록 수정했습니다. 중복 예약에 대한 최종 방어선은 여전히 DB UNIQUE 제약으로 유지했습니다.

- [x] **8. 요청 시각 생성 책임을 서비스 밖으로 옮길지 검토한다.**
  - 받은 리뷰: `요청 시각`은 해당 메서드를 지날 때인지, 컨트롤러에서 요청을 받았을 때인지 확인하라. 서비스에서 시각을 생성하면 테스트에 용이한 구조인지 확인하라.
  - 판단 기준: 서비스 내부에서 `LocalDateTime.now()`를 직접 호출하면 테스트가 현재 시각에 의존한다. 또한 요청 시각의 의미가 "HTTP 요청이 들어온 시점"인지 "서비스 로직이 실행된 시점"인지 모호해진다.
  - 해결 방향: 컨트롤러에서 요청 시각을 생성해 서비스로 전달하거나, `Clock`을 주입해 서비스가 현재 시각을 얻도록 변경하는 방식을 비교한다. 테스트 용이성과 계층 책임을 기준으로 선택한다.
  - 리뷰 코멘트: 말씀해주신 내용을 보고 `요청 시각`은 서비스 메서드를 지나는 순간보다 요청을 처리하기 시작한 경계의 시각에 가깝다고 판단했습니다. 그래서 서비스 내부의 `LocalDateTime.now()` 호출을 제거하고, 컨트롤러와 페이지 조립 경계에서 `requestedAt`을 생성해 서비스로 전달하도록 수정했습니다. 덕분에 서비스 테스트에서도 기준 시각을 직접 넘길 수 있어 시간 의존성을 더 명확히 제어할 수 있게 되었습니다.

### 히스토리 순번 계산과 대기 줄 표현

- [x] **9. `MyWaitingLines`의 책임을 표현 계층과 도메인 계층 사이에서 다시 정리한다.**
  - 받은 리뷰: `MyWaitingLines`의 책임이 무엇인지 확인하라. 표현 계층이라면 정렬 로직이 필요한지, 도메인 계층이라면 `ReservationWaitingLine`과 무엇이 다른지 확인하라. 대기 순서가 바뀌면 코드가 몇 군데 변경되는지도 점검하라.
  - 판단 기준: `MyWaitingLines`는 히스토리 응답을 만들기 위해 대기 순번을 계산하는 객체다. 그러나 대기 순서 규칙을 직접 가지고 있다면 도메인 규칙이 히스토리 조회 전용 객체에 중복될 수 있다.
  - 해결 방향: 대기 순서 규칙은 `ReservationWaitingLine` 또는 더 큰 도메인 객체 한 곳에 모은다. `MyWaitingLines`는 히스토리 조회 결과를 응답에 필요한 형태로 조립하는 책임만 남기거나, 히스토리 전용 계산 객체로 유지한다면 도메인 순서 규칙과 중복되지 않도록 의존 방향을 조정한다.
  - 리뷰 코멘트: 기존에는 `ReservationWaitingLine`에서 1-based 순번을 제거한 뒤 `MyWaitingLines`를 만들었지만, 다시 보니 `MyWaitingLines`가 정렬 규칙까지 직접 알고 있어 대기 순서 규칙이 두 곳에 생긴 상태였습니다. 대기 순서 자체는 첫 번째 대기 승급에도 쓰이는 도메인 규칙이라고 판단해 `ReservationWaitingLine`에 모으고, `MyWaitingLines`는 `ReservationWaitingLine.indexOf()`에 위임한 뒤 히스토리 응답에 필요한 표시 순번으로 `+1`만 하도록 정리했습니다.

### 더 큰 도메인 모델 검토

- [x] **10. 예약 취소 가능 여부와 대기 승급 흐름을 더 큰 도메인 객체로 내릴 수 있는지 검토한다.**
  - 받은 리뷰: 예약 취소 가능 여부와 대기 승급은 도메인에서 지켜야 하는 내용일 수 있다. 서비스에 private 메서드로 남아 있으면 다른 응용 계층에서 같은 로직이 복제될 수 있다.
  - 판단 기준: 리뷰의 핵심은 단순히 private 메서드를 줄이라는 것이 아니라, "예약 취소 시 첫 대기가 승급된다"는 요구사항 규칙이 특정 서비스의 절차 코드에만 머무르고 있다는 점이었다. 저장소 조회와 영속화는 서비스 책임이지만, 확정 예약과 대기 줄을 함께 보고 취소 결과를 결정하는 일은 슬롯 기준 도메인 규칙으로 볼 수 있다.
  - 해결 내용: `ReservationSlotBooking`을 추가해 하나의 슬롯에 대한 확정 예약과 대기 줄을 함께 다루도록 했다. 이 객체는 예약과 대기 줄이 같은 슬롯인지 검증하고, 취소 시 대기가 없으면 예약만 취소하고 대기가 있으면 첫 대기를 예약으로 승급시키는 `ReservationCancellation` 결과를 만든다. `ReservationService`는 이 결과를 받아 저장소에 반영하는 역할만 수행하도록 정리했다.
  - 리뷰 코멘트: 말씀해주신 내용을 보고 서비스의 private 메서드가 단순 분리가 아니라, "예약 취소 시 첫 대기를 승급한다"는 도메인 규칙을 서비스가 직접 쥐고 있는 신호라고 이해했습니다. 그래서 저장소 조회와 반영은 서비스에 남기되, 확정 예약과 대기 줄을 함께 보고 취소 결과를 결정하는 `ReservationSlotBooking`을 추가했습니다. 이제 서비스는 `ReservationCancellation` 결과에 따라 기존 예약 삭제, 승급 예약 저장, 승급된 대기 삭제를 반영하고, 취소/승급 판단은 도메인 객체가 담당하도록 정리했습니다.

### 입력 타입과 검증 위치

- [x] **11. `ReservationSlotService.open()`의 식별자 타입을 primitive로 바꿀 수 있는지 검토한다.**
  - 받은 리뷰: `open`의 인자가 primitive type이었다면 `validateReferenceIds`는 불필요하지 않았을지 확인하라.
  - 판단 기준: 컨트롤러 요청 DTO에서 `@NotNull`로 검증된 식별자가 서비스로 넘어온다면, 서비스에서 null 검증을 반복하는 것은 뷰 계층 검증과 중복될 수 있다. 서비스 메서드가 `long`을 받으면 null 가능성 자체가 사라진다.
  - 해결 내용: `ReservationSlotService.open()`의 `themeId`, `timeId`를 `long`으로 변경하고 서비스의 `validateReferenceIds`를 제거했다. API 요청은 DTO의 `@NotNull` 검증을 통과한 뒤 서비스로 전달되고, 관리자 페이지 폼 요청처럼 `@RequestParam(required = false)`로 null이 들어올 수 있는 경로는 컨트롤러에서 검증하도록 옮겼다.
  - 리뷰 코멘트: 말씀해주신 대로 서비스까지 `Long`을 받다 보니 null 검증 책임이 서비스에 남아 있었고, 이 검증이 비즈니스 규칙인지 요청 값 검증인지 모호해졌던 것 같습니다. 외부 입력의 null 검증은 DTO 또는 컨트롤러 경계에서 처리하고, 서비스는 `long themeId`, `long timeId`를 받아 참조 조회와 슬롯 생성 규칙에 집중하도록 수정했습니다.
