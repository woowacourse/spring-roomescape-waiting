# 2단계 모바일 인증 설계

## 선택 도구

세션 방식 (웹/모바일 동일)

- **웹**: 브라우저가 `Set-Cookie` 헤더를 자동으로 저장하고 이후 요청에 `Cookie: JSESSIONID=...`를 자동 첨부
- **모바일**: 응답의 `Set-Cookie` 헤더에서 `JSESSIONID`를 직접 파싱하여 이후 요청에 수동으로 전달

## 다른 후보

JWT 토큰 방식

- 로그인 후 토큰을 응답 바디로 전달, 이후 요청마다 `Authorization: Bearer ...` 헤더에 포함
- 무상태(stateless)라 서버 확장에 유리하지만, 탈취 시 즉시 무효화가 어렵고 구현 복잡도 높음

## 선택 이유

- 기존 세션 인증 로직을 그대로 재사용 가능 (서버 추가 구현 없음)
- 모바일도 `Set-Cookie` 헤더를 직접 파싱해 관리하면 되므로 서버 변경 불필요
- 웹과 모바일 인증 흐름이 동일해 중복 로직 없음

## 불편하거나 아쉬운 점

- 서버가 다중 인스턴스로 확장될 경우 세션 공유 문제 발생 (Redis 등 별도 저장소 필요)
- 모바일 클라이언트가 쿠키를 직접 관리해야 해서 브라우저 대비 구현 부담 있음
