# PR 리뷰 요청: 동시성 처리

## 리뷰 포인트

### 1. 동시성 테스트를 어떻게 작성하는 것이 가장 올바른 방법인가요?

현재 두 가지 방식으로 작성했습니다.

**H2 + CountDownLatch** (`ConcurrencyTest`)
- 모든 스레드를 `startLatch.await()`에서 대기시킨 뒤 동시에 출발시킵니다.
- H2의 락 모델이 MySQL과 달라 FOR UPDATE 동작이 다를 수 있습니다.

**MySQL(Testcontainers) + CyclicBarrier** (`ConcurrencyWithMySqlTest`)
- 모든 스레드가 `barrier.await()`에 모이면 자동으로 동시 출발합니다.
- 실제 MySQL 락 모델 위에서 동작하므로 운영 환경에 더 가깝습니다.
- 컨테이너 시작 비용이 있어 느립니다.

두 방식 모두 "진짜 동시"를 보장하지 않고 확률적으로 경쟁 조건을 유발합니다.
`@RepeatedTest`로 반복 횟수를 늘려 신뢰도를 보완하고 있습니다.

**궁금한 점:** 동시성 테스트를 어떻게 작성하는 것이 가장 올바른 방법인지 궁금합니다.
H2와 MySQL Testcontainers 중 어느 쪽이 적절한지, 또는 둘 다 필요한지,
그리고 `CountDownLatch`와 `CyclicBarrier` 중 어느 쪽을 선호하시는지 의견이 궁금합니다.

### 2. 비관적 락으로 충분한가, 어느 수준까지 동시성 제어를 고려해야 하는가?

현재 `findById`에 `FOR UPDATE`를 추가해 조회 시점에 행을 잠급니다.
`saveReservation`과 `cancelReservation`의 경쟁 조건은 이것으로 막힙니다.

비관적 락 외에 고려할 수 있는 방법들이 있습니다.

| 방법 | 특징 |
|---|---|
| 비관적 락 (`FOR UPDATE`) | 조회 시점에 잠금, 대기 발생, 현재 적용 |
| 낙관적 락 (`version` 컬럼) | 커밋 시점에 충돌 감지, 재시도 필요 |
| DB 유니크 제약 | INSERT 시 DB가 중복 차단, 단순하지만 적용 범위 제한적 |

지금 수준의 서비스에서는 비관적 락으로 충분해 보입니다.

**궁금한 점:** 어느 수준까지 동시성 제어를 고려하는 것이 적절한지 궁금합니다.
비관적 락으로 충분한 상황과, 낙관적 락이나 다른 전략으로 전환을 고려해야 하는 상황의 기준이 있는지 여쭤보고 싶습니다.
