## 관찰: 1차캐시

### 시도한 코드

### 예측한 SQL/동작

```
save()
->  아직 INSERT 안 나감
->  flush()
->  INSERT 나감

```

### 실제 SQL/동작

```
save()
->  INSERT 바로 나감
->  id 채워짐
->  flush()
->  추가 INSERT 없음
```

### 왜 그런가

JPA의 `save()`는 엔티티를 영속성 컨텍스트가 관리하도록 만드는 동작이다.  
일반적으로 영속성 컨텍스트는 INSERT SQL을 바로 실행하지 않고 flush 시점까지 모아둘 수 있다.

하지만 현재 엔티티의 id 생성 전략은 `GenerationType.IDENTITY`이다.  
IDENTITY 전략에서는 id를 애플리케이션이 미리 만드는 것이 아니라 DB가 INSERT 시점에 생성한다.

따라서 Hibernate는 새 엔티티를 영속성 컨텍스트에서 식별하기 위해 DB에 INSERT를 먼저 보내고, DB가 생성한 id를 받아와 엔티티의 id 필드에 채운다.

그래서 `save()` 직후 INSERT가 바로 실행되었고, `flush()` 시점에는 이미 INSERT가 끝난 상태이므로 추가 INSERT가 발생하지 않았다.

즉, 쓰기 지연이 동작하지 않는 것이 아니라, IDENTITY 전략에서는 id 확보를 위해 INSERT 지연이 제한된다.
