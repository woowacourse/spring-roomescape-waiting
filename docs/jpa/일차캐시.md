## 관찰: 1차캐시

### 시도한 코드

Theme 객체를 만들고, themeRepository에서 두 번 연속적으로 가져옴.

### 예측한 SQL/동작

SQL 문이 하나만 나올 것이다.
왜냐하면 기존에 하나 조회해서 1차 캐시에 넣어놓을 것이기 때문이다.

### 실제 SQL/동작

```sql
select t1_0.id,
       t1_0.description,
       t1_0.name,
       t1_0.thumbnail_image_url
from theme t1_0
where t1_0.id = ?
```

### 왜 그런가

Theme을 조회하면서 영속성 컨텍스트에 미리 캐싱을 위해 첫 번째 조회순간에 넣어놓고,
추가적으로 두 번째 조회 순간에는 영속성 컨텍스트에 있는 기존의 캐싱을 사용해
그것을 DB까지 접근하지 않고 그대로 가져온다. 정리해보자면

영속성 컨텍스트는 엔티티를 `Entity 타입 + id` 기준으로 관리한다.
첫 번째 findById에서 DB 조회 결과로 만들어진 Theme 엔티티가 영속성 컨텍스트에 저장된다.
두 번째 findById에서는 같은 Theme 타입과 같은 id의 엔티티가 이미 영속성 컨텍스트에 있으므로 DB 조회 없이 그 인스턴스를 반환한다.
따라서 같은 트랜잭션 안에서는 반복 조회 시 SELECT가 생략되고, 엔티티 동일성도 보장된다.
