# 방탈출 예약/대기 리팩토링 계획

## 목표

사이클2 요구사항을 구현하기 전에 현재 코드를 학습 가능한 구조로 개선한다.

이번 리팩토링의 목적은 단순히 코드를 옮기는 것이 아니라, 다음 기준을 만족하는 방향으로 점진적으로 개선하는 것이다.

- 도메인이 단순 데이터 보관을 넘어 핵심 규칙과 행위를 표현한다.
- 애플리케이션 서비스는 유스케이스 흐름을 조율하고, 세부 규칙 판단을 도메인에 위임한다.
- 헥사고날 아키텍처를 도입하여 의존 방향을 명확히 한다.
- 리팩토링 중 기존 외부 동작이 깨지지 않도록 테스트 안전망을 먼저 보강한다.
- 사이클2의 대기 승인/자동 전환과 트랜잭션 일관성 요구사항을 수용할 수 있는 구조를 만든다.

## 현재 코드에서 관찰한 문제

### 1. 빈약한 도메인

`Reservation`, `Waiting`, `Slot`은 대부분 식별자와 외래키를 보관하는 역할만 한다.

현재 많은 비즈니스 규칙은 `ReservationService`, `WaitingService`, `SlotService`에 흩어져 있다.

예시:

- 예약/대기 소유자 검증
- 과거 날짜/시간 검증
- 슬롯 예약 가능 여부 판단
- 대기 가능 여부 판단
- 대기 순번 계산

### 2. 서비스 책임 과다

서비스가 다음 책임을 동시에 가진다.

- 요청 DTO를 도메인으로 변환
- 저장소 조회
- 검증
- 비즈니스 규칙 판단
- 응답 DTO 생성
- 유스케이스 흐름 조율

이 상태에서는 기능이 늘어날수록 서비스가 커지고, 규칙 변경의 영향 범위를 예측하기 어려워진다.

### 3. 아키텍처 의존 방향이 불명확함

현재 저장소 인터페이스가 `infrastructure` 패키지에 있다.

애플리케이션 계층은 추상화에 의존하고 있지만, 패키지 구조만 보면 도메인/애플리케이션이 인프라를 향하는 것처럼 보인다.

헥사고날 아키텍처로 전환할 때는 저장소 인터페이스를 `application.port.out`으로 옮기고, JDBC 구현체만 `adapter.out.persistence`에 두는 방향이 적절하다.

### 4. DTO와 도메인 생성 책임이 섞임

`ReservationSaveRequest`, `WaitingRequest`가 `toDomain()`을 가진다.

입력 DTO가 도메인 생성 책임을 가지면 웹 계층의 요청 구조와 도메인 생성 규칙이 결합된다.

### 5. 대기 순번 정책이 암묵적임

현재 대기 순번은 별도 컬럼이 아니라 `waiting.id` 기준으로 계산된다.

즉, 대기 삭제 후 순번은 저장된 값이 갱신되는 것이 아니라 조회 시 다시 계산된다.

사이클2의 "순번 재정렬" 요구사항을 만족하려면 이 정책을 명시적으로 다루거나, 별도 순번 컬럼을 도입할지 결정해야 한다.

## 전체 진행 순서

### 1단계: 테스트 안전망 보강

리팩토링 전에 기존 외부 동작을 테스트로 고정한다.

한 번에 모든 테스트를 추가하지 않고, 리팩토링할 범위 주변의 테스트를 작게 추가한다.

첫 번째 대상은 대기 생성 정책이다.

추가할 테스트 후보:

- 예약된 슬롯에는 대기를 신청할 수 있다.
- 대기가 이미 있는 슬롯에는 다음 순번으로 대기를 신청할 수 있다.
- 예약도 대기도 없는 슬롯에는 대기를 신청할 수 없다.
- 이미 예약한 사용자는 같은 슬롯에 대기할 수 없다.
- 이미 대기한 사용자는 같은 슬롯에 중복 대기할 수 없다.
- 본인 대기가 아니면 취소할 수 없다.
- 없는 대기를 취소하면 성공 처리한다.

완료 기준:

- 현재 정책이 테스트 이름으로 드러난다.
- `./gradlew test`가 통과한다.
- 아직 구조 변경은 하지 않는다.

### 2단계: 도메인 규칙 식별 및 도메인 테스트 추가

서비스에 숨어 있는 규칙을 도메인 객체가 표현할 수 있는 언어로 정리한다.

우선 도입할 수 있는 도메인 행위:

- `Reservation.isOwnedBy(memberId)`
- `Waiting.isOwnedBy(memberId)`
- `Slot.isPast(clock, reservationTime)`
- `WaitingLine.first()`
- `WaitingLine.orderOf(waiting)`
- `WaitingLine.remove(waiting)`

완료 기준:

- 도메인 단위 테스트가 생긴다.
- 서비스의 private 검증 로직 일부가 도메인 행위로 이동할 준비가 된다.

### 3단계: 작은 도메인 행위부터 서비스에서 이동

가장 위험이 낮은 소유자 검증부터 이동한다.

예상 순서:

1. `Reservation`에 본인 예약 여부 판단 추가
2. `Waiting`에 본인 대기 여부 판단 추가
3. 서비스에서 `Objects.equals(...)` 직접 비교 제거
4. 테스트 통과 확인

완료 기준:

- 서비스는 "소유자 검증을 수행한다"는 흐름만 남긴다.
- 구체적인 소유 여부 판단은 도메인이 담당한다.

### 4단계: 대기 정책을 도메인 개념으로 분리

대기 가능 여부와 순번 계산은 사이클2의 핵심 기반이다.

현재 정책:

- 예약이 있는 슬롯은 대기 가능
- 대기가 이미 있는 슬롯은 대기 가능
- 예약도 대기도 없는 슬롯은 대기 불가
- 같은 사용자는 같은 슬롯에 중복 대기 불가
- 이미 예약한 사용자는 같은 슬롯에 대기 불가

후보 도메인:

- `WaitingLine`
- `ReservationSlot`
- `SlotOccupancy`

완료 기준:

- 대기 가능 여부 판단이 서비스의 단순 if 조합이 아니라 명시적인 도메인 개념으로 표현된다.
- 대기 순번 계산 정책이 코드와 테스트에 드러난다.

### 5단계: 헥사고날 패키지 구조로 점진 전환

기능별로 한 번에 하나씩 옮긴다.

추천 구조:

```text
roomescape
  reservation
    domain
    application
      port
        in
        out
    adapter
      in
        web
      out
        persistence
  waiting
    domain
    application
      port
        in
        out
    adapter
      in
        web
      out
        persistence
  slot
    domain
    application
      port
        out
    adapter
      out
        persistence
```

완료 기준:

- 애플리케이션 계층은 포트 인터페이스에만 의존한다.
- JDBC 구현은 어댑터 바깥쪽에 위치한다.
- 컨트롤러는 inbound adapter로 분리된다.

### 6단계: 애플리케이션 서비스를 유스케이스 중심으로 정리

현재 CRUD 중심 서비스 메서드를 유스케이스 단위로 재구성한다.

후보:

- `CreateReservationUseCase`
- `CancelReservationUseCase`
- `ChangeReservationUseCase`
- `CreateWaitingUseCase`
- `CancelWaitingUseCase`
- `FindMyReservationsUseCase`

완료 기준:

- 각 유스케이스의 트랜잭션 경계가 명확해진다.
- 서비스가 모든 규칙을 직접 판단하지 않고 도메인과 포트를 조율한다.

### 7단계: 사이클2 요구사항 구현

리팩토링 기반 위에서 대기 승인/전환 흐름을 구현한다.

추천 선택지는 자동 전환이다.

자동 전환 흐름:

```text
예약 취소
→ 취소한 예약의 슬롯 확인
→ 해당 슬롯의 첫 번째 대기 조회
→ 첫 번째 대기를 예약으로 전환
→ 전환된 대기 삭제
→ 나머지 대기 순번은 조회 시 재계산
```

필수 검증:

- 예약 취소와 대기 전환은 하나의 트랜잭션으로 묶인다.
- 중간 실패 시 예약 삭제만 반영되는 상태가 생기지 않는다.
- 전환 후 같은 슬롯에는 예약이 하나만 존재한다.
- 전환 후 기존 대기 2번은 조회 시 1번으로 보인다.

## 테스트 전략

### 서비스 테스트에 적합한 것

- 특정 조건에서 예외가 발생하는지
- 저장소 호출 여부
- 소유자 검증
- 단순 유스케이스 흐름

### 통합 테스트에 적합한 것

- 실제 DB 기반 대기 순번 계산
- 예약과 대기 조회 병합
- 취소 후 순번 재계산
- 트랜잭션 롤백
- 유니크 제약과 애플리케이션 검증의 관계

### 도메인 테스트에 적합한 것

- 소유자 판단
- 대기열 순번 계산
- 첫 번째 대기자 선택
- 과거 슬롯 판단
- 예약/대기 가능 상태 판단
