### Review 02

> ### 의존성 불일치
>
> Slot이라는 도메인이 추가되면서 Reservation과 Waiting 도메인이 조금 모호해진 것 같습니다.  
> Reservation이 Waiting은 Repository로 참고하고, Slot은 Service로 참조하도록 설정하신 이유가 있을까요?

### Feedback 02

먼저, `Slot` 개념을 도입하며 명확한 생명주기를 표현할 기회라 생각했어요.  
`Time` / `Theme` 데이터를 기반으로 슬롯을 먼저 생성하고, 그 슬롯을 통해  
예약과 대기라는 하위 엔티티를 표현/관리한다면 적절한 애그리거트 루트라는 생각이었습니다.  
마지막 리뷰에서 말씀해 주셨다시피 생명주기를 거꾸로 표현해 버렸지만..;

슬롯을 스키마로만 두고 예약과 대기의 필드로 꺼내오기보다  
도메인으로 구현 후 객체를 필드로 참조하도록 하면 OneToOne, ManyToOne 구조로  
알맞게 표현되지 않을까, 예약과 대기의 책임을 슬롯에 적절히 분산할 수 있지 않을까 생각했어요.  
아마 이 지점이  
기존 필드 구조를 붙잡고 가려다 슬롯이 단순히 date / Time / Theme 를 묶어주는 역할이 되어  
하위 엔티티의 필드로 들어가버리는 거꾸로된 구조가 만들어진 원인이었지 않나 싶네요.

다음으로, 기존 구조를 붙잡은 채로 `SlotService`를 구현하다 보니  
`ReservationService` / `WaitingService` 는 `SlotService` 만 참조하면 되는 구조가 만들어졌고,  
이 상황에서 일관성을 준수하기 위해 `SlotService` 를 굳이 분해할 필요가 있는가? 하는 의문에

정말 필요하지 않은 행위를 그저 원칙이라는 허상에 매여서 행하는,  
사이클 1에서 단어에 매몰돼 해야할 일을 놓치는 것이라 생각해 과감히 일관성을 버렸어요.

배경 설명이 좀 길어졌지만 결론적으론 `Slot` -> `Session` 으로 보다 직관적인 네이밍을 적용하고,  
세션이 애그리거트 루트 역할을 명확히 수행할 수 있도록 구조 리팩토링을 진행해 보았습니다.

### Review 06

> ### 혼재된 개념과 역할, 구조
>
> 지금 제 생각에는 Reservation, Waiting, Slot간의 역할이 조금 모호합니다.  
> 말씀해주신 것처럼 Slot은 Reservation의 저장, 삭제에 따라 생명주기를 가져갑니다.  
> Slot의 생명주기가 Reservation에 종속된다면, Slot은 독립 도메인이 아니라 Reservation에 딸린 개념 아닌가요?  
> 그렇다면 왜 별도 테이블과 별도 서비스로 분리했나요?
>
> 정답이 있는 영역은 아니고 비즈니스를 어떻게 이해하느냐에 따라서 다른 설계가 나올 수 있습니다.  
> 각각의 트레이드 오프를 고민해보시고  
> Slot을 별도의 VO로 생각하고 테이블을 없애던가,    
> Slot을 예약 가능 시간대 개념(?) 독립적인 객체로 가져가려면 애그리거트 루트로 만들던가 해야합니다.
>
> 지금 구조에서는 두 방향을 모두 채택하려고 하고 있어서  
> 별도의 테이블을 가지면서 생명주기도 공유하고 이 때문에 로직 자체가 복잡하게 보여집니다.

### Feedback 06

> 두 방향을 모두 채택하려고 하고 있어서

가 문제점을 정확히 짚고 깨달음을 준 것 같아요.  
구구와 질의응답 시간에서 관련된 내용을 여쭤보니

- 슬롯이란 명칭과 현재 수행중인 역할의 개념이 직관적으로 와닿지 않는다
- 지금의 구조라면 오히려 예약이 애그리거트 루트인데 왜 슬롯을 루트라 생각하는가

등의 답변을 받아 조금 더 방향성을 잡을 수 있었어요.

`왜 기존의 구조에서 슬롯을 테이블로 분리했는가`

- 예약과 대기에서 중복되는 데이터, 날짜/시간/테마 를 묶어서 관리하는 것이(정규화) 바람직하기에

`왜 세션(슬롯)을 도메인 객체로 구현했는가`

- 다른 도메인에서 중복되는 상태와 행위를 하나의 책임으로 묶을 필요성이 있었기에

`왜 스키마와 도메인 객체를 1:1로 대응했는가`

- `리소스/도메인/스키마`는 1:1 대응 관계일 필요도, 1:1 대응 관계가 아닐 필요도 없기에,  
  그저 도메인의 비즈니스 로직을 어떻게 표현할 것인지, 해석과 기준에 따른 자신(팀)만의 선택에 불과하기에.
