### Review 01

> ### 검증과 검사
>
> 전체 조회 쿼리에서 Slot을 가져오는 RowMapper에는 mapTimeSlot() 과 mapTheme() 이 있습니다.

> 테스트 코드에서는 Fake 객체를 쓰고 있어서 잡지 못하고 있는데 확인 부탁드립니다.

### Feedback 01

자동 삭제 테스트만 성공하고 실제 구현체에 잘못된 쿼리를 남겨놨었네요. 수정했습니다!  
이런 상황을 뭐라 표현해야 할 지 공부하다 관련된 내용을 학습 로그로 정리해 보았습니다.  
추가로 이런 상황을 방지하기 위해 성공 테스트도 전반적으로 추가했습니다!

### Review 02

> ### 의존성 불일치
>
> Slot이라는 도메인이 추가되면서 Reservation과 Waiting 도메인이 조금 모호해진 것 같습니다.  
> Reservation이 Waiting은 Repository로 참고하고, Slot은 Service로 참조하도록 설정하신 이유가 있을까요?

### Feedback 02

먼저, `Slot` 개념을 도입하며 명확한 생명주기를 표현할 기회라 생각했어요.  
`Time` / `Theme` 데이터를 기반으로 슬롯을 먼저 생성하고, 그 슬롯을 통해  
예약과 대기라는 하위 엔티티를 표현/관리한다면 적절한 애그리거트 루트라는 생각이었습니다.  
마지막 리뷰에서 말씀해 주셨다시피 생명주기를 거꾸로 표현해 버렸지만..;

슬롯을 스키마로만 두고 예약과 대기의 필드로 꺼내오기보다  
도메인으로 구현 후 객체를 필드로 참조하도록 하면 OneToOne, ManyToOne 구조로  
알맞게 표현되지 않을까, 예약과 대기의 책임을 슬롯에 적절히 분산할 수 있지 않을까 생각했어요.  
아마 이 지점이  
기존 필드 구조를 붙잡고 가려다 슬롯이 단순히 date / Time / Theme 를 묶어주는 역할이 되어  
하위 엔티티의 필드로 들어가버리는 거꾸로된 구조가 만들어진 원인이었지 않나 싶네요.

다음으로, 기존 구조를 붙잡은 채로 `SlotService`를 구현하다 보니  
`ReservationService` / `WaitingService` 는 `SlotService` 만 참조하면 되는 구조가 만들어졌고,  
이 상황에서 일관성을 준수하기 위해 `SlotService` 를 굳이 분해할 필요가 있는가? 하는 의문에

정말 필요하지 않은 행위를 그저 원칙이라는 허상에 매여서 행하는,  
사이클 1에서 단어에 매몰돼 해야할 일을 놓치는 것이라 생각해 과감히 일관성을 버렸어요.

배경 설명이 좀 길어졌지만 결론적으론 `Slot` -> `Session` 으로 보다 직관적인 네이밍을 적용하고,  
세션이 애그리거트 루트 역할을 명확히 수행할 수 있도록 구조 리팩토링을 진행해 보았습니다.
