# 42. 사이클 끊기는 도구가 다르다 + 책임을 제자리에

**날짜**: 2026-06-18
**학습 범위**: 양방향 의존(패키지/빈) 사이클을 *어떻게* 끊는가 — 결합의 **성격(동기 vs 비동기)**에 따라 도구가 다름. 그리고 "남의 살림(상태 전이)을 하는 서비스"는 책임이 잘못 놓인 신호. (#41 가드의 *코드 적용* 연장 — payment↔reservation 사이클을 실제로 끊음.)

분류: 아키텍처 / 의존성·책임 — 코드 적용 (재방문: #28 아웃박스, #37 포트, #40 사이클·아웃박스)

## 0. 계기
package-by-feature가 드러낸 사이클 중 `payment↔reservation`을 끊기로(#40 미뤄둔 것). 출발은 *증상* — "PaymentService가 자꾸 PendingReservation을 취소/확정한다"는 위화감. 책임이 잘못 놓인 듯.

## 1. 증상의 루트는 사이클
PaymentService가 예약 상태전이(confirm/cancelPending/orphan 만료)를 *직접* 함 = 예약 생명주기를 payment가 소유. "그럼 ReservationService로 옮기고 호출하면?" → **빈 사이클**(ReservationService가 이미 PaymentService를 주입 → 생성자 양방향 = 스프링 기동 실패). 즉 책임 오배치는 사이클의 *증상*.

## 2. 핵심: 사이클마다 끊는 도구가 다르다
- `reservation↔waiting`(#40): 승격은 **비동기**(취소→나중에 워커가 승격) → **아웃박스**로 끊음(#28).
- `payment↔reservation`: 결제는 **동기 요청 흐름**(예약 만들면 주문 생성 / 결제되면 예약 확정) → 아웃박스가 안 맞음. 대신 **책임을 중립 제3자로 재배치**해서 끊음.
- → 한 줄: *비동기면 이벤트/아웃박스, 동기면 의존을 제3자로 라우팅(또는 포트로 역전).*

## 3. 끊은 방법 — 중립 `order` 패키지
- `Order` 도메인을 독립 `order` 패키지로 분리 + `OrderService` 신설: 주문 생명주기만 소유, **예약·결제를 모름**(중립).
- `ReservationService.create`가 주문 생성을 `OrderService`로 호출 → **`reservation→payment` 변 제거**.
- 그 결과 `payment→reservation`(확정/취소 위임)이 **단방향**이 됨. 예약 상태전이는 `ReservationService`로 이관, PaymentService는 게이트웨이+조율만.
- 검증: `reservation→payment` 0건, order 중립 → 사이클 소멸.

## 4. 화살표로 본 직관 (내가 인출한 것)
끊기 전: `ReservationService →(createOrder) PaymentService →(confirm) reservation` = 양방향.
끊은 후: createOrder가 **OrderService로 빠지면서** ReservationService→Payment 변이 사라짐 → Payment→Reservation 단방향. *한 변을 제3자로 우회시키면 반대 변이 단방향이 된다.*

## 5. 한 문장 봉인
> 양방향 사이클은 "누가 누굴 부르나"의 문제 — **비동기**면 이벤트/아웃박스로, **동기**면 한 변을 *중립 제3자*로 재배치(또는 포트로 역전)해 단방향으로 편다. 그리고 *상태 전이는 그 상태를 가진 쪽이 소유*한다(조율자는 위임만).
