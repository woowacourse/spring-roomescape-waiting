# 학습 로그 #36

**날짜**: 2026-06-15
**학습 범위**: confirm 호출의 멱등성 — 네트워크 재시도 안전성. paymentKey 중복감지 vs Idempotency-Key, "멱등성 키는 왜 항상 클라이언트가 발급하나".

분류: 외부 API 연동 / 결제 (재방문: #31 재시도·멱등성, #35 paymentKey·confirm)

## 1. 계기

log_31에서 saga 재발명 때 `"...→재시도→멱등성"`으로 **이름만 흐릿하게** 댔던 키워드. 오늘 그걸 *confirm 호출*이라는 구체적 자리에서 못박음. (재방문 = 같은 키워드, 더 깊이.)

## 2. 문제 — 응답 유실의 모호함

- 서버가 `POST /confirm`을 **한 번** 호출 → 토스가 출금 성공 → 그런데 **응답이 유실**(타임아웃).
- 서버 입장: **"성공인지 실패인지 알 수 없다."** "응답 못 받음" ≠ "안 일어남".
- **딜레마**: 재시도 안 하면 → (사실 미출금이면) 결제 영영 유실 / 재시도 하면 → (사실 출금됐으면) **이중 출금**.
- → **멱등성 = "재시도해도 안전하다"** 를 보장해 이 딜레마를 푼다. 마음 놓고 재시도하게.

## 3. 두 층위

### (a) paymentKey 중복감지 (자연 멱등)
- 토스가 **paymentKey**로 "이미 DONE인 결제네" 알아챔 → **돈은 두 번 안 빠짐**(side effect 반복 금지 = 멱등성 1차 목표).
- 단 재시도 응답이 **에러**(`ALREADY_PROCESSED_PAYMENT`) → 서버가 "이 에러 = 사실 성공"으로 **분기 처리**해야 함. 돌아가지만 *에러 채널에 의존*.

### (b) Idempotency-Key (투명 멱등)
- 우리가 *시도마다* 고유 키 발급 → 헤더로 첨부 → **토스가 그 키로 응답을 캐시** → 같은 키 재시도엔 **처음 성공응답(DONE)을 그대로** 반환.
- 재시도가 **투명**해짐: 서버는 분기 없이 그냥 "성공이네". 출금은 당연히 1회.

## 4. 핵심 — 멱등성 키는 왜 *클라이언트*가 발급하나 (후보 탈락법)

- "토스가 만든다" 가정 → 그 키는 토스의 **응답**에 실려 와야 함 → 근데 재시도 상황은 그 응답이 **유실**된 상황 → 우리 손에 안 들어옴 → 재시도에 못 붙임 → 토스가 못 알아봄. **탈락.**
- 결론: **멱등성 키는 "응답 유실에도 살아남아야" 제 역할.** 유실에 살아남는 유일한 길 = *재시도하는 쪽이 보내기 전에 직접 만들어 쥠.* → **"누가 살아남게 보장하냐"가 발급 주인을 정한다.**

## 5. 두 키 대조 (오늘 핵심 그림)

| | paymentKey | Idempotency-Key |
|---|---|---|
| 누가 발급 | **토스** | **우리(클라이언트)** |
| 식별 대상 | 거래(payment) | (재)**시도** |
| 우리 손에 언제 | successUrl로 받음 | **보내기 전에 직접 생성** |
| 재시도 응답 | (중복) 에러 → 성공으로 분기 | **처음 성공응답 그대로(투명)** |

- 둘 다 "고유 식별자"지만 **주인·목적이 정반대.** [[35_토스결제-인증vs승인]]
