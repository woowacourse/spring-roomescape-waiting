## 2. 이번 타임의 학습 전략

### 학습 과정

#### IllegalArgumentException의 문제

라이브러리도 `IllegalArgumentException`을 던진다. GlobalExceptionHandler가 이를 통째로 잡아서 400을 반환하면, 프로그래밍 버그가 사용자 오류(400)로 위장될 수 있다.

#### BadRequestException의 문제

`BadRequestException`은 `HttpStatus.BAD_REQUEST`를 포함한다. 도메인이 이걸 던지면 도메인이 HTTP를 알게 된다. 레이어 오염이다.

#### 도메인 예외 분리

- 도메인은 비즈니스 규칙 위반을 표현하는 것이 책임
- HTTP 매핑은 GlobalExceptionHandler의 책임
- 도메인이 물어봐야 할 것: "규칙이 위반됐는가?" (HTTP 코드가 아님)

#### 트레이드오프

| | 도메인 순수성 | 다형성 |
|---|---|---|
| enum + switch | ✅ | ❌ (조건 분기) |
| `toHttpStatus()` 추상 메서드 | ❌ | ✅ |

진짜 다형성을 쓰려면 예외가 HttpStatus를 알아야 한다. 두 가지를 동시에 가질 수 없다.

#### 핵심 깨달음

"Domain 예외는 400으로 고정"이라는 아이디어를 냈다가, 그 생각 자체가 **상태코드에 종속**된 설계임을 스스로 발견했다. 도메인 설계 시 HTTP 코드를 먼저 생각하는 순간 레이어 분리가 무너진다.

## 3. 전략 평가

- 효과적이었던 것: 코드 리뷰 피드백을 계기로 설계 원칙 탐구까지 자연스럽게 연결됐다.
- 비효과적이었던 것: 딱히 없었다.

## 4. AI 피드백

오늘 핵심은 "이것도 좀 이상하긴 하네"라고 스스로 느꼈을 때다. '400으로 고정'이라는 아이디어 자체가 잘못됐다는 걸 스스로 발견한 순간 — 상태코드에 종속된 사고방식을 스스로 인식했다. 이게 설계 감각이 생기는 과정이다.

## 5. 다음 타임에 바꿀 것

- 바꿀 것: 딱히 없음
