# 학습 로그 #23

**학습 범위**: 객체지향 리팩토링 — Slot 값 객체 추출 (SRP, 캡슐화, Tell-Don't-Ask, 응집도)

## 1. 막힌 것의 종류

- [ ] 개념 자체를 모르겠다
- [ ] 개념은 알겠는데 코드로 어떻게 쓰는지 모르겠다
- [x] 코드는 돌아가는데 이게 맞는 건지 모르겠다
- [ ] 기타: ___

Waiting의 도메인 규칙이 Service에 퍼져 있고 Reservation과 공통점이 많은데, 어떻게 정리해야 할지 몰랐다.

## 2. 이번 타임의 학습 내용

### 진단

- Waiting에 있어야 할 도메인 규칙이 Service에 퍼져 있음
- Reservation과 공통으로 쓰는 필드/메서드가 많음 (`date/time/theme/storeId`)

### 가설과 탈락 (후보 탈락법)

```
가설: Waiting이 Reservation을 통째로 소유(composition)한다
  ↓ 라이프사이클 검증
Reservation이 사라지면 Waiting이 승격 → 둘의 생명주기가 엇갈림
  ↓
"통째 소유" 탈락 → 소유가 아니라 공통 개념을 따로 추출해야 한다
```

### 착지: Slot 값 객체 추출

- `date/time/theme/storeId`를 직접 들고 있던 건 **원천처럼 보이지만 파생** — 두 도메인에 복사돼 있어 서로 어긋날 위험
- 공통 개념에 이름을 붙임 → `Slot` (Reservation은 "차지", Waiting은 "대기")
- 데이터를 가진 객체가 판단도 한다(응집도/Tell-Don't-Ask) → `isPast(now)`를 Slot으로

### 적용 전략 (안전한 리팩토링)

- 팩토리/생성자 시그니처는 유지하고 내부에서만 Slot 조립
- 기존 getter(`getDate/getTime/getTheme/getStoreId`)를 Slot에 위임
- → 캡슐화 덕분에 내부 표현을 바꿔도 외부 계약은 그대로 → DAO/DTO 무변경, 테스트 green 유지

```
Slot (불변 값 객체): date, time, theme, storeId + isPast(now)
Reservation = Slot + 예약자(Member) + status ...
Waiting     = Slot + 대기자(Member) + rank
```

- 적용 순서: 안전망(테스트) 확보 → Slot 생성 → Waiting 적용 → Reservation 적용 → 테스트 재확인

## 3. 전략 평가

- 효과적이었던 것: 가설을 끝까지 밀어붙여 라이프사이클 검증에서 직접 탈락시킨 것. 구조 변경 전에 안전망(테스트)을 먼저 확보한 것.
- 아쉬운 것: 인출 연습에서 composition(소유)과 association(가짐)을 구분하지 못한 채 "has-a 관계가 아니다"라고 단정함.

## 4. 이전 "바꿀 것" 적용 여부

- 바꿀 것: 패턴 이름을 구조와 연결해서 기억하기 (3번째 미뤄짐)
- 적용: 오늘 실제로 컴포지션·값 객체 패턴을 썼으나 이름 연결을 명시적으로 하지 않음 — "상황 미발생"이 아니라 **기회를 놓친** 것

## 5. 다음 타임에 바꿀 것

**유지할 것**
가설을 끝까지 밀어붙여 탈락시키는 방식. 오늘 "Waiting이 Reservation을 통째 소유한다" 가설을 라이프사이클 검증으로 직접 깨뜨린 순간이 핵심 착지점을 만들었다. 결론을 의심하고 검증하는 흐름은 계속 가져간다.

**바꿀 것**

1. **개념을 정밀하게 쓰기 전에 결론부터 내리지 않기.** 인출 연습에서 "has-a 관계가 아니다"라고 단정했지만, composition(소유)과 association(가짐)을 구분하지 못한 채였다. 단정하기 전에 "지금 쓰는 이 단어, 정확히 어떤 의미지?"를 한 번 멈춰서 확인한다.

2. **"패턴 이름 연결"의 미룸 패턴 깨기 (3회째).** 매번 "상황 미발생"을 핑계로 미뤘지만, 오늘은 실제로 컴포지션·값 객체 패턴을 썼다. 즉 미발생이 아니라 *기회를 놓친* 것이다. 트리거를 바꾼다: "패턴 타임"을 기다리지 말고, **코드 적용 직후 '방금 쓴 구조에 이름이 있나?'를 묻는 습관**으로 전환한다. 또한 설계 결정(getter 위임 등)을 코치가 주도하기 전에, 먼저 사용자가 한 가지 안을 제시하도록 순서를 바꾼다.
