# 학습 로그 #22

**학습 범위**: JWT 구조와 동작 방식, 세션 vs JWT 트레이드오프

## 1. 막힌 것의 종류

- [x] 개념 자체를 모르겠다
- [ ] 개념은 알겠는데 코드로 어떻게 쓰는지 모르겠다
- [ ] 코드는 돌아가는데 이게 맞는 건지 모르겠다
- [ ] 기타: ___

JWT가 서버에 아무것도 저장하지 않는다면 어떻게 검증하는지 몰랐다.

## 2. 이번 타임의 학습 내용

### JWT 구조

```
xxxxx.yyyyy.zzzzz
헤더  .페이로드.서명
```

- **헤더**: 서명 알고리즘 정보
- **페이로드**: 사용자 정보 (memberId, role 등) — Base64로 인코딩된 JSON
- **서명**: 헤더 + 페이로드를 비밀키로 서명한 값

### 검증 흐름

```
로그인 시:
  서버가 헤더 + 페이로드를 비밀키로 서명 → JWT 발급 → 클라이언트에 전달

이후 요청:
  클라이언트가 JWT 전송
  → 서버가 헤더 + 페이로드를 같은 비밀키로 재계산
  → 서명 비교 → 일치하면 페이로드의 정보 신뢰
  → DB 조회 없음
```

- 복호화가 아니라 **재계산 후 비교** — 페이로드를 변조하면 서명이 맞지 않음
- 비밀키 없이는 올바른 서명을 만들 수 없으므로 위조 불가

### 세션 vs JWT 비교

| | 세션 | JWT |
|---|---|---|
| 서버 저장 | O (메모리/DB) | X |
| DB 조회 | 매 요청마다 | 없음 |
| 로그아웃 보장 | O | X (만료 전까지 유효) |
| 동시 로그인 방지 | 구현 가능 | 서버 저장 필요 (세션과 동일해짐) |
| 스케일 아웃 | 공유 저장소 필요 | 비밀키만 공유하면 됨 |

### JWT가 빛을 발하는 상황

- 서버를 여러 대로 늘려야 할 때 (스케일 아웃)
- 세션은 서버 A에서 로그인 → 서버 B로 요청 오면 세션 못 찾음
- 해결책: Redis 공유 세션 저장소 or sticky session → 복잡도 증가
- JWT는 어느 서버로 가도 비밀키만 공유하면 검증 가능

### JWT의 단점

- 매 요청마다 서명 계산 비용
- 페이로드의 정보가 바뀌어도 토큰은 만료 전까지 유효 (갱신 필요)
- 로그아웃/강제 만료/동시 로그인 방지 구현 시 서버 저장 필요 → stateless 장점 희석

## 3. 전략 평가

- 효과적이었던 것: 세션 방식을 직접 구현한 뒤라 JWT와 비교가 자연스럽게 됐다
- 아쉬운 것: 서명 방식을 "암호를 푼다"로 오해했다가 수정 — 복호화 vs 재계산 차이를 처음부터 명확히 할 수 있었을 것 같다

## 4. 이전 "바꿀 것" 적용 여부

- 바꿀 것: 패턴 이름과 구조를 연결해서 기억하기
- 적용: 이번 타임도 해당 상황 없었음 — 디자인 패턴 키워드 할 때 반드시 적용

## 5. 다음 타임에 바꿀 것

- 유지할 것: 세션 구현 후 JWT로 넘어간 것처럼, 직접 구현한 것과 연결해서 학습하는 흐름
- 바꿀 것: 패턴 이름 연결 — 3번째 미뤄지고 있음. 다음 디자인 패턴 타임에서 반드시 이름 먼저 떠올리는 연습
