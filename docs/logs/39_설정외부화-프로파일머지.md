## 0. 계기 — 401에서 갈라진 갈래
v1 결제창 → v2 결제위젯 마이그레이션을 끝낸 뒤 confirm이 `401 인증되지 않은 시크릿 키`. 원인: `local` 프로파일 미적용 → `application-local.yml`의 gsk가 안 실리고 `application.yml`의 빈 기본값(`${TOSS_SECRET_KEY:}`)이 나감 → `Basic base64(":")` → 토스 거절. (위젯은 떴는데 confirm만 깨진 건, client-key 기본값은 gck로 박아둬서 프로파일 없이도 위젯이 떴기 때문.) 여기서 "프로파일/설정 머지가 정확히 어떻게 도나?"를 파고듦.

## 1. 핵심 인출 — base + overlay 머지 (한쪽만/둘 다?)
- 초기 걱정: "secret을 별도 파일로 빼면 기존 application.yml(DB 정보 든)은 안 읽히나?"
- 추측으로 도달: `application.yml`(base)은 **프로파일과 무관하게 항상 로드**된다. `application-{profile}.yml`은 그 **위에 얹혀**(더 높은 우선순위) **자기가 가진 키만 덮어쓴다.**
- 키 단위 머지: 양쪽에 있으면 → 프로파일 승 / 한쪽에만 있으면 → 그 값.
- → **원래 걱정의 답**: DB 설정(`datasource.url` 등)은 application.yml에만 있으니 그대로 로드됨. application-local.yml엔 **복사 안 하고**, *덮어쓸 것(시크릿)만* 둔다. (그래서 application-local.yml이 toss 키 2줄뿐.)

## 2. 왜 굳이 분리? — 시크릿은 커밋되면 끝
- `application.yml`은 git에 **커밋·push → 공개**. 시크릿을 거기 박으면 그대로 노출(공개 repo면 영구).
- `application-local.yml`은 **`.gitignore`** → 로컬에만 남음. 시크릿을 여기 두고 `local` 프로파일이 "이 파일도 읽어라" 스위치 역할.
- 즉 **파일 분리 = 비밀을 커밋 경계 밖으로 빼는 수단.** 프로파일은 그 외부 파일을 *언제* 끼울지 정하는 스위치.

## 3. 연결("그러면 운영선?") — 3번째 소스: 환경변수
- 운영 배포 서버엔 `local` 프로파일도, gitignore된 그 파일도 없다. → 그럼 시크릿은 어디서?
- 답이 코드에 *이미* 있었음: `secret-key: ${TOSS_SECRET_KEY:}`. **`${VAR:기본값}`** 플레이스홀더가 **환경변수** 주입 갈고리.
- **우선순위(같은 키, 높은→낮은): 환경변수 > 프로파일 yml > base yml.** 같은 `secret-key` 한 줄이 환경마다 다른 소스에서 채워진다 — **코드 0 변경, 환경만 다름.**

## 4. 한 문장 봉인 (정의 + 방향)
> 스프링 설정은 base(항상 로드) + 프로파일(overlay, 중복 키만 덮음) + 환경변수(최상위)로 **키 단위 머지**되고, 시크릿은 *커밋되는 base에 박지 않고* gitignore된 프로파일 파일이나 환경변수로 **외부화**한다. (방향: 비밀이 커밋 경계 *밖에* 머물게.)

## 5. 곁가지 — 설정 주입: @Value vs @ConfigurationProperties (ISP 트레이드오프)
RestClient @Bean을 짜며 갈라진 갈래(코드리뷰 직전). 레퍼런스는 `@Value("${toss.base-url}")`(값 하나씩), 우린 `TossProperties`(toss.* 묶음 record).
- **@ConfigurationProperties(TossProperties):** 정의 응집(한 곳) + 타입안전 + prefix(`toss.`) DRY. 단 consumer가 *통째로* 의존 → 안 쓰는 필드까지 노출 = **ISP 약화**(오용 가능·과도한 책임·"이것도 써야 하나" 오해).
- **@Value:** consumer가 *쓰는 값만* 콕 의존(ISP 정밀). 단 `"${toss.~}"` prefix가 반복·흩어짐.
- **함정 회피:** "응집"을 ①정의 ②사용으로 쪼개라. 내가 처음 "전역적으로 쓰이면 응집 깨진 거 아니냐" 했지만, 그 "전역 사용"은 ②(어느 방식이든 게이트웨이/config가 서로 다른 값 씀 → 흩어짐)지 ①이 아님. ①(정의 모음)은 TossProperties가 이김. 반론의 *진짜* 알맹이는 ②가 아니라 consumer 과노출(ISP).
- **판단:** 작고 응집된 설정(toss 4필드)은 TossProperties로 충분(과노출 비용 작음). 크거나 오용 위험 크면 @Value/properties 분할. → 맹목 채택 아니라 트레이드오프 알고 고름.
