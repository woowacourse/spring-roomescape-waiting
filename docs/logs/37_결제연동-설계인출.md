# 학습 로그 #37

**날짜**: 2026-06-16
**학습 범위**: 토스 결제 연동을 roomescape에 **코드로 적용** — 전체 아키텍처(엔드포인트 3개, 주문 저장, PENDING 상태, 검증→승인 순서, 포트&어댑터, 에러 매핑, 경계)를 *추측으로 인출*해 설계. log_35·36 개념을 코드로 내린 i+1.

분류: 외부 API 연동 / 결제 — **코드 적용** (재방문: #35 인증·승인, #36 멱등성, #33 책임의 위치, #32 클라이언트 값 위조). 같은 세션의 메타 절반은 [[38_ACL적용-원칙은수단]].

## 0. 이 로그의 성격 — "받은" 게 아니라 "인출한" 설계
오늘 설계는 코치가 떠먹인 게 아니라, *추측 질문*에 답하며 거의 전부 내가 재구성했다. 그래서 큰 학습이었다. 아래는 그 인출 경로.

## 1. 흐름을 코드로 — create() 한 방이 3칸으로 쪼개진다
- 기존: `create()` → 즉시 BOOKED. 결제 낄 틈 없음.
- 인출: 결제를 끼우면 서버 진입점이 **3개**로 — ① 예약 생성(PENDING + orderId 발급·전달) ② success 콜백(승인) ③ fail 콜백(정리).
- 연결: log_35의 "서버가 *조용히* 하는 두 번([전]저장 / [후]검증+승인)"이 코드 엔드포인트로 그대로 앉았다. 그때 약한 고리였던 "보이지 않는 두 칸"을 이번엔 점프 없이 먼저 짚음(log_35 "바꿀 것" 적용).

## 2. 주문 조각 — 책임 분리 + amount 스냅샷
- Order를 Reservation과 **분리**(책임의 위치, log_33). orderId·amount(①) + paymentKey(②, log_36의 "토스가 successUrl로 *처음* 주는 것"이 여기 합류).
- **amount는 Theme.price의 스냅샷** = 검증 기준. 핵심 인출: 주문 생성과 결제 사이 테마 가격이 *바뀌어도* 검증은 저장값과 한다(라이브값 X). "왜 amount를 따로 저장하나"의 진짜 이유 = 가격 변동 + 할인 → 주문이 자기 값을 self-contained하게 들어야. (요구사항의 "최종 amount".)

## 3. 상태 — BOOKED 재사용 (의식적 트레이드오프)
- PENDING 추가, 승인 성공 시 PENDING → **BOOKED**(기존 재사용). 새 CONFIRMED 안 만듦.
- 이유 인출: `isActive()`가 BOOKED에 묶여 있어, 별도 CONFIRMED를 만들면 기존 코드가 전부 ripple. 재사용 = minimal change(단순함 먼저). autopilot 아니라 트레이드오프를 말로 풀어서 고름.

## 4. 검증 먼저 → 승인 (완료 기준의 코드화)
- `order.validateAmount`가 `gateway.confirm`보다 *앞*. 조작 금액은 게이트웨이 호출 *전* 차단 → 완료 기준 "게이트웨이가 호출되지 않는다"가 이 순서에서 나옴. 테스트로 `verify(never())`.
- throughline(log_32·35): "클라이언트가 들고 온 값은 못 믿는다 → 서버 진짜값으로 검증"이 코드의 이 한 줄 순서로 박힘.

## 5. 포트 & 어댑터 — 의존성 역전(DIP)
- `PaymentGateway`(포트, 도메인) ← `TossPaymentGateway`(어댑터, 토스 앎). PaymentService는 포트만 의존.
- 인출 함정 통과: 처음에 "PaymentService가 TossPaymentGateway에 요청"이라 했다가, *"이름에 Toss가 박히면 PG 바꿀 때 또 고친다 → 디커플링 아님"*을 스스로 교정. 포트 이름은 중립(PaymentGateway), 구현만 Toss = DIP(고수준·저수준 둘 다 추상에 기댐).
- 경계: 포트를 오가는 건 도메인 모델(PaymentConfirmation/PaymentResult). 토스 DTO는 어댑터(payment.toss) 안에서 나고 죽음. (엄격도/예외 이름 논쟁은 메타로 → [[38_ACL적용-원칙은수단]].)

## 6. 에러 매핑 — 처리(handling)별로
- 예외 종류 수 = 토스 *코드 수*가 아니라 *서로 다른 처리* 수. 카드거절(안내)/키오류(알람)/토스내부(재시도)/이미처리됨. 미정의 코드는 기본 예외로.

## 7. 검증
- 백엔드 전 구현 + 다각도 적대적 리뷰 + 228 테스트 그린. 완료 기준(테스트카드 승인→확정 / 조작 amount 차단 / 에러코드→예외·응답 / 미정의→기본 / secret 외부화 / failUrl null 가드) 충족.
