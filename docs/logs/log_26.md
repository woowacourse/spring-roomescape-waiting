# 학습 로그 #26

**학습 범위**: INSERT 시 락이 걸리는 순서 — insert intention lock → X record lock → 유니크 충돌 시 S락 → 동시 중복 INSERT 데드락 (MySQL InnoDB 기준)

## 1. 막힌 것의 종류

- [x] 개념 자체를 모르겠다 (정확히는: 조각은 있는데 **순서/관계가 엉켜 있었다**)
- [ ] 개념은 알겠는데 코드로 어떻게 쓰는지 모르겠다
- [ ] 코드는 돌아가는데 이게 맞는 건지 모르겠다

시작 시점의 지식: "Insert할 때 insertion-lock을 걸고, 롤백될 때까지 gap-lock이 걸린다. 다른 insertion은 유니크 제약 때문에 S락을 건다." → 세 조각(insertion-lock / gap / 유니크 S락)이 다 맞는 재료였지만 **시간 순서로 꿰여 있지 않았고**, "롤백까지 유지되는 건 gap락"이라는 한 곳이 어긋나 있었다.

## 2. 이번 타임의 학습 내용

### INSERT 한 줄의 락 타임라인

```
1. insert intention lock (gap 계열, 행 작성 "전"에 자리 확보)
   - 같은 gap에 서로 다른 값을 넣는 intention락끼리는 "호환" → 안 막음
   - 그래서 이름이 intention: gap 전체를 잠그는 게 아니라 "넣을 의도"만 표시
2. 새 행 작성 + X record lock
   - 이게 커밋/롤백까지 "유지"되는 락 (시작 지식에서 'gap락'이라 한 부분의 정체)
3. 다른 트랜잭션이 같은 유니크값을 INSERT 시도
   - 미커밋 상태라 "중복 확정"을 못 함 → T1 끝날 때까지 기다려야 함
   - 그 대기 수단으로 기존 행에 S락 요청 → T1의 X락과 충돌 → 블록
   - S로 거는 이유: 남의 행을 고치려는 게 아니라 "이 값이 살아남나"만 확인
```

### intention락이 일반 gap락과 다른 점

일반 gap락이 gap 전체를 잠그면, 같은 gap에 서로 다른 값(5, 7)을 넣으려는 INSERT 두 개가 불필요하게 서로 대기하게 된다 → 동시성 최악. 그래서 InnoDB는 insert intention lock을 **서로 호환되도록** 설계했다. "범위 vs 점"의 문제가 아니라 **"intention락끼리는 충돌하지 않는다(compatible)"** 가 핵심.

### 동시 중복 INSERT 데드락 (정확한 트리거)

```
T1: 5 삽입, X락 보유, 아직 미커밋
T2: 5 삽입 시도 → S락 대기 (T1 기다림)
T3: 5 삽입 시도 → S락 대기 (T1 기다림)
  └ 이 순간은 아직 데드락 아님: 둘 다 T1 하나만 기다림 = 사이클 없음

T1 롤백 → X 풀림
 → T2, T3 둘 다 S락 획득 (S끼리 호환이라 동시 가능)
 → 둘 다 진짜 INSERT 위해 X(insert intention)로 "승격" 시도
 → T2의 X승격은 T3의 S 해제 필요 / T3의 X승격은 T2의 S 해제 필요
 → 상호 대기 = 사이클 = 데드락
```

- 트리거는 "대기 2개"가 아니라 **"S 획득 후 X 승격이 맞물리는 순간"**.
- MySQL은 wait-for 그래프로 사이클을 감지 → **undo log가 적은(롤백 비용이 싼) 트랜잭션을 victim으로 롤백** + 예외 반환.

### 자기 락은 충돌하지 않는다

"intention락 걸린 상태에서 같은 트랜잭션이 X락을 걸 수 있나, 아니면 풀고 다시 거나?" → **한 트랜잭션은 자기 락끼리 절대 안 부딪힌다(항상 호환).** 락은 "트랜잭션 간" 충돌 장치이기 때문. 그래서 intention락 → 행 작성 → X락은 "풀고 다시"가 아니라 그냥 **얹는** 것. (자기-블로킹이 있었다면 INSERT 한 줄로 혼자 데드락 날 것.)

## 3. 전략 평가

**효과적이었던 것**

- 추측 먼저 방식이 잘 먹혔다. "intention락은 A(전)에 건다"는 **첫 직감이 정답**이었고, 본인이 잠깐 B로 뒤집었다가 그 근거 문장("gap락 있으면 삽입 못 함")이 *누구의* gap락이냐로 갈린다는 걸 짚으니 정리됐다.
- 데드락의 정확한 트리거(S→X 승격 맞물림)와 victim 메커니즘까지 스스로 도달.

**아쉬운 것 (학습 방식)**

코치가 "intention락 언제 걸려요?"와 "왜 intention일까요?"를 한 메시지에 묶어 던진 게 "한 번에 하나의 포인트만"과 부딪히는 듯 보였다. 하지만 본인 판단으로는 **insert intention lock을 아예 모르던 입장이라 방향을 지정해주는 게 오히려 도움이 됐다.** → 결함이 아니라 **calibration 이슈**였다. 다만 오늘은 그게 *우연히* 잘 맞았고, 코치가 친숙도를 먼저 가늠하고 입자 크기를 정한 건 아니었다 (그래서 5번 바꿀 것으로 연결).

## 5. 다음 타임에 바꿀 것

**유지할 것**

학습 중간에 **본인이 먼저 "이건 어떻게 동작하지?" 중간 의문을 던지는 것** (오늘 "자기 X락 가능?"). 코치가 다음 포인트로 끌고 가기 전에 스스로 의문을 만든 흐름은 계속 가져간다.

**바꿀 것**

1. **코치는 질문 입자 크기(granularity)를 사용자 친숙도로 조절한다.** 개념을 아예 모를 때 → 방향을 주는 스캐폴드(포인트 여러 개 묶기 허용). 어느 정도 발판이 있을 때 → 한 포인트씩 좁혀 집중. **Why**: "한 번에 하나의 포인트만"은 발판이 있을 때의 규칙이고, 거리가 먼(i+1이 큰) 새 개념에선 방향 제시가 사이클을 살린다. 오늘은 우연히 맞았을 뿐. **How**: 새 개념에 들어가기 전 "이거 아는 거예요, 처음 보는 거예요?"로 친숙도를 먼저 가늠하고 입자 크기를 정한다.

연결 메모: 오늘 배운 "유니크 충돌 S락 → 중복 INSERT 데드락"은 next-cycle의 **"동시성 race(finding #6) + DB unique index 도입 여부"**(log_25) 판단의 직접 재료다. unique index를 backstop으로 두면 이 데드락 패턴이 어떻게 바뀌는지가 그 키워드의 실전 버전. → 아래 7번에서 실전 적용으로 해결.

## 7. 실전 적용 — WaitingService.create 동시성 점검 (같은 날)

### 시작 의문 (user)

"WaitingService에서 insert할 때 동시성 문제 아닌가? Waiting 삽입 3개가 동시에 일어나면 전부 rank 1을 받는 거잖아."

- rank를 `waitings.size() + 1`로 계산(`Waitings.enqueue`) → **read-then-write race**. 직렬화 장치가 없다면 100% 터지는 게 맞다. (직감 정확)

### 코드 실제 분석

`WaitingService.create`에는 `ForUpdate`가 **두 개**:

```java
// 29줄: 예약(reservation) 행에 FOR UPDATE
reservationDao.findBySlotKeyForUpdate(...)
// 33줄: 큐(대기들)에 FOR UPDATE
waitingDao.findQueueBySlotForUpdate(slot)
```

- **함정**: 큐가 비어 있으면 33번 `SELECT ... FOR UPDATE`는 잠글 행이 0개 → **큐 락만으론 직렬화 불가**.
- **진짜 게이트는 29번**: 대기는 항상 존재하는 예약에 붙으므로, **예약 행 = 항상 존재하는 안정적 앵커**. 큐가 비었든 말든 같은 슬롯의 INSERT들을 그 한 행의 X락 앞에 한 줄로 세운다.

### 일반화 패턴

> **개수/순번(rank, count, max)을 읽고 쓰는 로직은, "항상 존재하는 부모 행"을 잠가서 직렬화한다.** (큐처럼 비어 있을 수 있는 대상은 앵커로 못 쓴다.)

### 오늘 배운 데드락은 이 코드에서 안 터진다

- `waitings`에 `UNIQUE (member_id, date, time_id, theme_id, store_id)` 이미 존재 → 중복 멤버 DB 백스톱 완비 (finding #6의 "unique index 도입 여부"는 이미 도입됨).
- 동시 중복 INSERT 데드락은 "같은 유니크값 두 개가 S락 단계에 **나란히** 도달"해야 터진다. 그런데 **같은 유니크값 = 같은 슬롯 = 같은 예약 행** → 29번에서 직렬화 → S락 단계에 나란히 못 선다 → **데드락 전제 붕괴.**
- 결론: **락의 순서/계층이 데드락 가능성을 결정한다.** 상위에서 직렬화하면 하위(유니크 체크)의 동시성 자체가 사라진다.

### 2단 방어 구조

1. **예약 행 X락** (1차): 정합성(rank/count/중복) + 직렬화
2. **유니크 인덱스** (2차): 코드 경로가 빠져도 중복을 막는 백스톱. 1차가 늘 먼저 직렬화하므로 평소엔 **발동조차 안 하는 잠든 백스톱**.

### 유일한 전제 (fragility)

모든 쓰기 경로가 **29번 줄(`findBySlotKeyForUpdate`)을 거쳐야** 이 보장이 성립한다. 배치/관리자/타 서비스가 이를 우회해 직접 INSERT하면 → 1차 직렬화가 사라지고, 그땐 동시 중복 INSERT가 유니크 인덱스 S락에서 오늘 배운 데드락을 낼 수 있다.
