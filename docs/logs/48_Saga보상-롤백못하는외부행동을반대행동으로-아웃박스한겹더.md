# 48. Saga 보상 — 롤백 못 하는 외부 행동을 *반대 행동*으로, 아웃박스 한 겹 더

**날짜**: 2026-06-18
**학습 범위**: hot arc '결제 능동적 실패/보상'의 #3 — `confirm()`이 토스 승인(돈 나감) 뒤 예약 확정에 실패하면 돈↔상태 불일치. DB 롤백으로 못 되돌리는 외부 행동을 어떻게 갚나. **#31(Saga 개념)의 재방문 — 추상에서 우리 코드 좌표로.**

분류: 분산 트랜잭션 / 비동기 — 개념 재방문 + 구현 설계 (재방문: #31 보상=반대행동, #28 아웃박스, #44 read=모름·catch→commit, #36 멱등키, #46 reconciliation, #47 상태가드)

## 0. 계기 — #31은 추상이었다
log_31에서 Saga를 *흐름 파악*까지 했지만 "룸이스케이프엔 외부 PG가 없어 실전 미적용"이라 §5에 **"Saga 코드 적용 — 어디에 적용해볼 수 있나"** 를 i+1로 남겨뒀다. 이제 토스(진짜 외부 PG)가 들어왔으니 그 빈칸을 정확히 꽂는다. 더해 #31이 예고한 **"보상 자체가 실패하면(보상의 멱등성)"** 까지 오늘 닿았다 — 설계된 재방문.

## 1. 롤백의 사정거리 — 외부·커밋후는 그 밖
- `confirm()` 3단계: ① `toss.approve()`(돈 나감, **외부·되돌릴 수 없음**) → ② `order.complete`(DB) → ③ `reservation.confirm`(DB). 셋이 한 `@Transactional`.
- ③이 터지면 DB는 ②·③을 깔끔히 롤백. 근데 ①은 *DB 행이 아니라 외부에서 이미 일어난 사실*이라 롤백 범위 **밖**. → **돈은 나갔는데 예약은 사라진다**(학습자 인출: "돈은 못 받았는데 예약이 취소돼버려").
- log_31 §4 재확인: Saga = A(원자성)를 한 DB 너머로 확장하되 **되감기 능력을 내주는** 트레이드오프. 그 "내준 되감기"가 여기서 실체로 드러남.

## 2. 더 위험 — 롤백이 *증거*까지 지운다
- ③에서 예외가 던져지면 ②(`order.complete`: CONFIRMED + paymentKey 저장)도 **같이 롤백**. → order는 PENDING/NEEDS_CHECK로 돌아가고 paymentKey도 안 남는다.
- 결과: 워커가 나중에 "돈 나간 주문 찾아줘" 해도 **안 잡힌다.** 돈은 토스에서 빠졌는데 DB엔 흔적이 0. **불일치를 탐지조차 못 하는** 게 진짜 구멍.

## 3. 증거 생존 트릭 — 예외 던지지 말고 catch→commit (#44 재사용)
- read timeout 때 똑같이 풀었다: 예외를 던지면 NEEDS_CHECK 마크가 롤백되니 → **잡아서** 마크하고 **리턴**(ConfirmOutcome). `@Transactional`은 *정상 리턴*엔 커밋하므로 마크가 살아남는다.
- 예약 실패도 같은 트릭: 예외 잡아서 `NEEDS_CHECK` 커밋 → 증거 생존 → 기존 reconcile 워커가 줍는다. **이미 가진 도구로 §2의 제일 무서운 부분이 풀린다.**

## 4. 구멍 — 워커엔 forward만 있고 backward(보상)가 없다
- 현재 `reconcile`: APPROVED → `complete`+`reservation.confirm`(forward), NOT_APPROVED → `markFailed`+`cancelPending`(DB 정리). **둘 다 "우리 DB 상태"만 바꾼다. 토스 돈을 되돌리는 행동(환불)은 0개.**
- 그래서 *APPROVED인데 `reservation.confirm`이 영영 실패*하면 → reconcile 트랜잭션 롤백 → NEEDS_CHECK 그대로 → 다음 사이클에 또 시도 → **무한 루프**, 환불은 영영 안 함.
- log_31 §2 재인용: **보상 = 롤백이 아니라 "반대 방향의 새 장(章)"**(결제→환불). reconcile에 그 backward 가지를 가르치는 게 남은 한 조각. ("해결할 게 너무 많다"의 정체 = 새로 만드는 건 결국 *이 가지 하나*.)

## 5. 재시도 vs 보상 — 한 묶음(경첩: "언제 포기하나")
- **일시적 실패**(DB 데드락 등) → 재시도(forward recovery). **영구적 실패**(예약이 죽음=슬롯 뺏김/취소) → 보상(환불, backward). log_31 §"실패 처리 두 갈래"의 코드화.
- 가르는 법: `reservation.confirm`이 *반환 타입*으로 "영구 불가"를 알리면 즉시 환불, 인프라 예외면 재시도. **깔끔히 갈리는 케이스엔 충분.**
- 한 군데는 타입으로도 **안 갈린다**: **외부 호출의 "모름"**(토스 환불이 됐는지 모름) → 거긴 **재시도 바운드**(N번/T분 넘으면 포기·판단)가 필요. → i+1 '재시도 정책'.
- 위험: 한 번 실패에 **바로 환불**하면, 그게 일시적이라 다음 재시도엔 예약이 *성공*하는데 돈은 이미 환불 → **거꾸로 불일치(공짜 예약)**. 그래서 함부로 환불 못 한다.

## 6. 재귀 — 보상도 외부 호출이라 아웃박스 한 겹 더
- 환불(toss cancel)도 외부 호출 → 또 "모름"·실패 가능. → **상태로 남기고 워커가 재시도**(학습자 인출: "환불 실패하면 데이터 남겨야 하고 아웃박스 패턴으로"). 보상은 특별한 기계가 아니라 **종일 쓴 그 패턴이 한 겹 더** 도는 것.
- 재시도하면 **이중 환불** 위험 → **Idempotency-Key**(환불당 고정 키)로 토스가 중복을 거른다. **step2 confirm 멱등키를 환불에 그대로 재적용**(#36). 이게 #31 §i+1 "보상의 멱등성"이 닿은 지점.
- 중요 구분: 우리 DAO 가드는 *우리가 두 번 시작*하는 걸 막을 뿐, "모름"이라 *같은 환불을 일부러 재시도*하는 중복은 **토스가** 멱등키로 걸러야 한다(중복은 우리 DB가 아니라 토스에서 일어나니까).

## 7. 한 문장 봉인
> 보상(Saga) = **DB 롤백으로 못 되돌리는 외부 행동(돈)** 을, *반대 방향의 새 행동(환불)* 으로 갚는 것. 라우팅은 우리가 이미 쓰는 모양(상태 찍기 + 워커 수렴) 그대로 — 단, 증거가 롤백에 안 지워지게 **catch→commit**(NEEDS_CHECK)이 출발점이고, 워커엔 forward만 있고 backward(환불)가 없는 게 구멍. 그리고 보상 자체도 외부 호출이라 **아웃박스 한 겹 더**(상태+워커+멱등키). "언제 재시도를 멈추고 보상으로 넘어가나"가 재시도 정책과 보상을 잇는 경첩.

## 8. 정직한 한계
- 오늘은 **개념 재방문 + 우리 코드 내 위치 확정 + 구현 설계**까지. 실제 코드는 미작성(커밋 없음). 구현(트랜잭션 밖 외부호출 분리 + 환불 가지 + 재시도 바운드 + 환불 멱등키)은 backlog hot에 남긴다.
