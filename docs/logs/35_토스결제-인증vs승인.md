# 학습 로그 #35

**날짜**: 2026-06-15
**학습 범위**: 토스 결제 연동(사전학습 1단계) — 결제 *인증* vs *승인*, 클라이언트/서버 역할 분담, "왜 서버가 confirm API를 호출해야 하나". 외부 API 문서 읽기.

분류: 외부 API 연동 / 결제 (재방문: #31 Saga·보상, #32 인증·secret·payload, #33 책임의 위치)

## 1. 지향점 (뿌리 질문)

> "카드사가 인증(카드 주인 확인)을 해줬는데, 왜 서버가 *승인*을 또 호출해야 하나?"

- 답: **클라이언트가 들고 돌아온 값(amount)을 못 믿기 때문.** 인증은 "이 카드 주인 맞아"까지. 돈은 아직 안 빠졌다.
- [[22_JWT구조-세션비교]] / log_32 "클라이언트가 들고 있는 값은 위조 가능"이 결제에서 그대로 재등장 → 이게 오늘의 알맹이.

## 2. 등장인물 3명 + 서버는 *두 번*만 개입

| ① 브라우저(위젯) | ② 우리 서버 | ③ 토스+카드사 |
|---|---|---|
| 화면·렌더링·요청·인증 | secretKey, 저장·검증·승인 | 외부, 못 건드림 |

- 화면 3단계(렌더링→요청→인증)는 ①③끼리. **서버는 구경.**
- 서버가 *조용히* 하는 건 눈에 안 보이는 **두 번**: [전]저장, [후]검증+승인. ← 처음 흐름 인출 때 이 둘이 빠져 안개였음(약한 고리).

## 3. 인증 ≠ 승인, 검증 ≠ 출금

- **인증**: 카드 주인 확인. 돈 그대로.
- **승인**: 서버가 `POST /v1/payments/confirm`(secretKey) → 그제서야 토스가 출금 → `status: DONE`.
- 검증만으론 돈 안 빠짐 — 검증은 *맞춰본 것*, confirm을 *날려야* 빠진다. (미션 2단계 = 이 confirm을 콘솔로 손호출.)

## 4. 저장(전) vs 검증(후) — 자꾸 섞인 지점

- 둘 다 "금액"을 다뤄 헷갈림. 하나는 **먼저 만들기(전)**, 하나는 **나중에 맞춰보기(후)**.
- **(가)결제 전 저장 vs (나)복귀 후 저장**: 차이는 *시점이 아니라 출처*.
  - 손님은 "무엇을"(테마·시간)만 고른다. **"얼마"는 서버가 자기 DB에서 안다** → 서버표 진짜값(ground truth).
  - (나)는 손님이 건넨 값을 손님이 건넨 값과 비교 → **공회전**(의미 없는 검증).
  - 그래서 진짜 금액은 손님이 손대기 *전*에 서버 데이터로 박아둬야 한다. "예약하기 찰나"가 신성한 게 아니라 **"결제 전"이라는 조건**이 핵심.

## 5. orderId vs paymentKey, clientKey vs secretKey

- **orderId = 우리 거.** 우리가 발급 → 위젯에 건넴 → successUrl로 *그대로* 복귀. **빈칸1(저장)과 빈칸2(검증)를 잇는 실(매칭 열쇠)** — "내가 저장한 그 주문이네" 찾는 용도. (← 토스가 발급하는 줄 오해했던 지점, 교정.)
- **paymentKey = 토스 거.** successUrl에서 *처음* 받아봄 → 저장(이후 조회·취소에 필수).
- **clientKey**(공개, 프론트 위젯) / **secretKey**(비밀, 서버 confirm — 노출되면 안 되니 ①에선 못 함).

## 6. throughline — log_32와 한 줄로 꿰기

- "클라이언트가 들고 있는 값은 위조 가능"(JWT Payload=Base64, 누구나 연다) → **서버가 secret + 자기 진짜값으로 검증**.
- 토스 amount 검증 = JWT signature 재계산·비교와 **같은 뼈대**. 못 믿는 입력을 서버만 아는 기준으로 받아낸다. [[16_세션-토큰인증]] [[32_인증세션-지향점]]

## 8. 학습법 회고

### 학습 *방식*의 아쉬운 점

오늘 가장 날카롭게 드러난 건 **"흐름을 시간 순서가 아니라 눈에 보이는 것 위주로 쥐는 습관"** 하나다. 두 증상이 같은 뿌리에서 나왔다.

- **저장↔검증 혼동(3번 반복)과 보이지 않는 단계 누락은 별개 실수가 아니다.** 둘 다 "amount를 다루는 화면 동작"은 또렷이 쥐는데, *화면에 안 뜨는* 서버의 두 동작(결제 전 저장 / 검증 후 confirm 호출)은 흐려진다는 동일 증상이다. 그래서 "검증하면 결제 완료"로 점프했다 — 검증과 출금 사이에 confirm이라는 *보이지 않는 한 칸*이 있는데, 화면엔 안 나오니 인출에서 통째로 빠졌다. 빈칸1(저장)을 묻는데 "검증해줘야 해"가 튀어나온 것도 같은 결: 두 칸을 *시점*으로 구분하지 못하고 'amount를 만지는 일'이라는 *모양*으로 한 덩어리로 쥐었기 때문. (4번이 결국 "차이는 시점이 아니라 출처"로 정착한 건 이 혼동을 정면으로 통과한 흔적이다.)

- **saga로 튄 점프는 강점과 회피가 겹쳐 있다 — 구분 기준이 필요하다.** "예약 생성 시점에 orderId 발급 + 실패 시 보상"으로 튄 건 BACKLOG에 parked돼 있던 'Saga 코드 적용'의 적용처를 *스스로* 찾아낸, 진짜 좋은 연결이었다(7번에 그대로 수확됨). 하지만 같은 점프가 **눈앞의 낯선 칸(저장/승인/orderId 주인)을 다 소화하기 전에** 일어났다는 게 신호다. orderId를 토스가 발급하는 줄 오해한 채로 saga 보상 설계까지 가 있었다 — 익숙한 영역(saga·도메인)이 *앞 칸의 안개를 가리는 알리바이*로 쓰일 수 있다. 강점인 점프와 회피인 점프의 차이는 **"점프하기 직전 칸을 정확히 말할 수 있느냐"**다. 오늘은 말할 수 없는 상태에서 튀었다.

(좋았던 것: 낯섦≠어려움을 등장인물 세우기로 풀어낸 것, "우리가 만드는 값"이라는 이름 추측으로 orderId를 한 방에 뚫은 것, 설계까지 번지자 스스로 "범위가 넓다"며 끊은 스코프 컷 — 이 세 개는 유지.)

### 다음 타임에 바꿀 것

**흐름을 인출할 때, 등장인물별로 "이 사람이 *조용히* 하는 일"을 한 칸씩 따로 짚어보기.** 오늘 등장인물 3명은 세웠지만, 거기서 멈추지 말고 각 인물이 *화면에 안 뜨는* 동작을 1개라도 갖는지 물어보는 단계를 추가한다. 서버가 "구경만" 하는 줄 알았는데 실은 *조용히 두 번* 개입한다는 게 오늘의 약한 고리였다 — 보이는 것만 나열하면 이 두 칸이 매번 샌다.

트레이드오프는 네가 판단할 몫:
- **칸을 다 채운 뒤 점프**하면 안개가 안 남지만, 오늘처럼 *튀는 순간 생기는 좋은 연결*(saga 적용처 발견)을 놓치거나 늦출 수 있다.
- 더 가벼운 대안: 점프를 막는 대신 **점프 직전에 "지금 떠나는 칸을 한 문장으로 말해봐"라는 자가 체크 하나만** 끼우기. 말할 수 있으면 강점인 연결이니 그대로 튀고, 막히면 그게 회피 신호다.

둘 다 깔지 말고 다음 사이클에 하나만 써보고 정한다.
