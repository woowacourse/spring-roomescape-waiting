# 학습 로그 #40

**날짜**: 2026-06-17
**학습 범위**: **패키지 구조 — package-by-feature(기능별) vs package-by-layer(타입별), 기능 경계, shared kernel, 의존 방향.** 토스 결제 정리 후 "구조가 커지니 길을 잃는다"에서 출발, 예제(기능별)를 모델로 *왜·어디·경계*를 추측으로 인출. (실제 이동은 다음 세션 — roomescape `HANDOFF_package-by-feature.md`.)

분류: 아키텍처 / 패키지 구조 — 개념 (재방문: #37 포트&어댑터, #28 아웃박스, #38 ACL)

## 0. 계기
layer-by-type(`controller/`·`dao/`·`dto/`·`domain/`)로 쌓다 보니 *한 기능* 고치려 4개 패키지를 왕복 → 커질수록 navigate 어려움. 예제(woowacourse payment)의 기능별 패키지가 "한눈에" 들어와 → 제대로 배우고 적용하기로.

## 1. 타입별 vs 기능별
- **타입별:** 같은 *종류*(컨트롤러끼리, dao끼리)를 모음 → 한 기능이 여러 패키지에 흩어짐.
- **기능별:** 같은 *관심사*(reservation 전부)를 모음 → 한 곳에서 한눈에. ← 목표.

## 2. 헥사고날 = 포트&어댑터 (이름만 몰랐던 재방문)
- "헥사고날 아키텍처" = 포트&어댑터의 별명(육각형 그림에서). 난 이미 #37에서 `PaymentGateway`(포트)/`TossPaymentGateway`(어댑터)로 *했음* — 새 개념이 아니라 이름을 안 것.
- 왜 포트를 거치나: PG사 교체(토스→카카오페이) 시 도메인 안 흔들리게 = DIP.
- **Controller·Dao의 정체:** Controller=들어오는(인바운드) 어댑터, JdbcDao=나가는(아웃바운드) 어댑터, `XxxDao`(인터페이스)=포트. → 둘 다 기능 패키지 안으로(`reservation/web/`, `reservation/dao/`).

## 3. 기능 경계 — "공유는 신호"
- 두 기능이 개념을 공유(Slot)하면 **경계 신호.** 별개면 → 공용(`common/` = **shared kernel**), 한 덩어리면 → 묶기(**bounded context**).
- reservation·waiting: "한 덩어리는 아니다"(각자 lifecycle·endpoint, *승격 지점에서만* 만남) → 별개 + Slot은 `common/`.
- Slot을 reservation에 넣으면 waiting이 reservation을 import = 기능→기능 결합. *어느 기능도 안 가리는* 공용에 둬야 둘이 대등하게 의존.
- 기준 한 줄: **각자 lifecycle·endpoint가 있고 특정 지점에서만 만나면 → 별개.**

## 4. 의존 방향 — 한 방향은 OK, 적은 사이클
- waiting→reservation은 *깊고 정당한* 한 방향: `Waiting.promote()`가 **Reservation을 생성**, enqueue/validate가 Reservation을 검사. waiting은 본질적으로 "예약을 기다림".
- 핵심: 한 방향 OK, **사이클**(서로 import)이 문제.
- 확인: reservation은 waiting을 import 안 함. 취소→승격은 `PromotionService.enqueuePromotion`(아웃박스 쓰기)뿐 → 실제 승격은 worker가 비동기. **아웃박스(#28)가 reservation↔waiting 사이클을 끊고 있었다** — 내가 짠 패턴이 결합 차단 역할을 하던 걸 이제 봄.

## 5. 한 문장 봉인
> 패키지는 *관심사(기능)*로 모으고(타입 아님), 기능이 공유하는 개념은 *어느 기능도 안 가리는 공용(shared kernel)*에, 기능 간 의존은 *한 방향*만 — 사이클은 아웃박스/이벤트로 끊는다.
