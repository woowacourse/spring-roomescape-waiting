# 학습 로그 #28

**학습 범위**: `@Scheduled` 워커 + 아웃박스 패턴 — 취소와 자동 승격을 비동기로 분리하고, 최종 일관성·멱등성·새치기 방지로 정합성을 지킨다.

## 1. 막힌 것의 종류

- [x] 개념 자체를 모르겠다 (`@Scheduled`/백그라운드 워커 완전 처음)
- [ ] 개념은 알겠는데 코드로 어떻게 쓰는지 모르겠다
- [ ] 코드는 돌아가는데 이게 맞는 건지 모르겠다

리뷰어 #6("promote 실패 시 취소도 실패되는 게 의도인가") + 본인 아이디어("재시도하면 금방 풀리니 묶는 게 낫다")에서 출발 → "안전하게 분리하는 법"으로 번지며 아웃박스가 등장. **구현 도중 "버겁다"고 느껴 한 단계 내려 개념 학습을 먼저 요청**(컴포트존 회피가 아니라 i+1 조절).

## 2. 이번 타임의 학습 내용

### `@Scheduled` = 시간 기반(polling), 이벤트 기반(Observer) 아님

- 첫 추측은 "이벤트 발생까지 대기(Observer/push)"였으나 **반대**였다.
- 도어벨(이벤트/push) ↔ 알람시계(시간/pull). `@Scheduled`는 알람시계: 이벤트를 안 기다리고 주기마다 깨어나 "PENDING 있나?" **폴링**.
- 그 추측(Observer)이 *틀렸지만* 반대 개념과의 대비를 만들어 오히려 이해를 가속.

### 아웃박스 패턴의 계기

취소 시 두 가지가 일어나야 함: ① 예약 취소(빠르게, 반드시 성공) ② 다음 대기자 승격(재시도 가능, 사용자 안 붙잡음). 묶기는 ②가 실패하면 ①도 실패. **떼고 싶다, 근데 그냥 던지면 승격이 증발**(스레드/서버 죽으면). → 승격 의도를 DB에 **취소와 같은 트랜잭션으로 기록**하고, 별도 워커가 나중에 확실히 실행 + 재시도.

### 핵심 오해 교정 — 워커는 다른 트랜잭션

"같은 트랜잭션에서 5초마다 폴링하면 락 문제"라는 걱정 → 사실 **워커는 취소와 다른 트랜잭션**. 취소는 outbox 한 줄 적고 즉시 커밋(락 바로 풀림), 워커는 5초 뒤 *자기 트랜잭션*으로 승격.

### 대가 = 최종 일관성, 그리고 두 문제

- **빈 창**(폴링 간격만큼): 취소됐는데 아직 승격 안 된 시간. strong(묶기) vs eventual(아웃박스).
- 문제①: 승격이 몇 초 늦음 → 이 도메인에선 견딜 만.
- 문제②: **새치기** — 빈 창에서 C가 대기자를 제치고 직접 예약. (정확성은 안 깨지지만 "대기자 우선" 규칙이 깨짐)

### 멱등성 (워커가 두 번 실행돼도 안전)

- "이 슬롯에 **활성 예약**이 있으면 승격하지 마라" → 첫 실행 후 조건이 거짓이 되어 두 번째는 멈춤.
- 코드 가드를 깜빡해도 슬롯당 활성 예약 1개 **UNIQUE** 제약이 두 번째 insert를 막는 백스톱.

### 구현 + 회귀 잡기 (실전)

- 파일: `PromotionTask`/`OutboxStatus`, `PromotionOutboxDao`, `WaitingDao.findFirstBySlotKeyForUpdate`, `WaitingService.enqueuePromotion`/`promotePendingSlot`, `PromotionOutboxWorker`(`@Scheduled`), 새치기 가드(`ReservationCreator.validateNoWaitingQueue`).
- 회귀 3종: ① `@JdbcTest @Import`에 새 빈 추가(의존성 바뀌면 슬라이스 컨텍스트가 깨진다 — log_27 Q2에서 배운 그 성질) ② 테스트에서 스케줄러 끄기(`scheduling.enabled=false`, 백그라운드 타이머가 검증을 흔들지 않게) ③ e2e `truncateAll`에 새 테이블 추가(공유 H2 오염 + FK 충돌).

## 3. 전략 평가

**효과적이었던 것**

- "버겁다"는 신호에서 스스로 한 단계 내려 **개념 학습을 먼저 요청**한 것. i+1을 본인이 조절. (코치가 끊은 게 아니라 학습자가 페이스를 잡음)
- 이름값 추측(Observer / outbox / 멱등성)이 매번 연결고리를 만들었다. 틀린 추측(Observer)도 대비로 작동.

**아쉬운 것 (학습 방식)**

개념 학습 땐 추측을 활발히 던졌지만, **구현 단계(파일 8개+)에선 학습자 인출이 0**이었다. 코치가 전부 작성하고 학습자는 지켜봄. 이번엔 목표가 "@Scheduled 개념"이라 괜찮았으나, 코드 적용 단계에 인출이 빠지면 체득이 약해질 수 있다. (회귀 디버깅을 코치가 혼자 처리한 건 기계적 이슈라 아쉬움 아님 — 학습자 판단.)
