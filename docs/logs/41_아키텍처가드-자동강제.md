# 41. 아키텍처 가드 — 왜 자동 강제가 필요한가 (ArchUnit)

**날짜**: 2026-06-17
**학습 범위**: 아키텍처 가드(ArchUnit)의 **동기(왜)**. 수동 확인의 한계 → 자동·반복 강제. pre-commit 훅 vs 빌드 테스트의 견고함 차이. ArchUnit의 정체(값이 아닌 *구조*를 검사하는 테스트). 가드는 *안 만들고* "왜"만 닫음.

분류: 아키텍처 / 테스트 — 개념+동기 (재방문: #38 ArchUnit 예고(ACL 경계), #40 의존 방향·사이클)

## 0. 계기
package-by-feature 실제 이동(#40 HANDOFF) 중, 의존 방향을 `grep`으로 확인하다 **패키지 사이클 2개**를 발견:
- `reservation ↔ waiting` (reservation→waiting: `ReservationCreator`의 새치기 방지 읽기 / waiting→reservation: 승격)
- `payment ↔ reservation` (PaymentService ↔ ReservationOrder/Service)

"발견"이 핵심 단서였다 — *우연히, 손으로* 알아챘다.

## 1. 발견은 우연이고 수동이었다
사이클의 존재를 알게 된 건 **재구조화하는 김에 내가 손으로 `grep`을 1회 돌렸기 때문**. 재구조화를 안 했으면 몰랐을 것. → 탐지가 사람의 행동·기억에 의존.

## 2. 가드가 없으면 구조가 무너진다 (왜 중요한가)
다음에 누가 `reservation`에 `waiting` import를 새로 추가하면 → **아무도 못 알아챈다.**
- 신규 위반이 조용히 누적 → 기능 경계가 흐려짐 → 우리가 *탈출한* 엉킨 구조로 회귀 (극단엔 "reservation 밑에 waiting을 합쳐버리는" 식으로 경계 붕괴).
- 즉 **재구조화는 1회성 이벤트가 아니라 유지돼야 가치가 있다.** 사람이 매번 기억해서 지킬 수는 없다.

## 3. 가드 = 자동·반복 검사 / ArchUnit의 정체
손으로 한 `grep`을 **기계가 매번 대신** 돌리게 하는 것 = 가드.
- **ArchUnit** = 이름 그대로 *Architecture + Unit(test)*. JUnit으로 돌아가는 평범한 테스트인데, `assertEquals(5, sum)`처럼 **값**을 검사하는 대신 **코드의 구조·의존 관계**를 검사한다.
- "패키지 사이클 금지"는 그 규칙 중 하나. (다른 예: "reservation은 waiting을 import 금지", "토스 타입은 payment.toss 밖으로 못 나감"(#38 ACL), 네이밍 규칙.)

## 4. 어디 둘까 — 빌드 테스트 > pre-commit 훅
- **pre-commit 훅**: 로컬에서 커밋 직전 실행. but 각 개발자가 *깔아야* 하고 `--no-verify`로 *건너뛸 수* 있음 → 사람 의존.
- **빌드/테스트에 박기**(`./gradlew test`): 누구든 빌드만 돌리면, 특히 **CI에서 강제** → 건너뛸 수 없음.
- → 가드의 본질은 **"사람의 기억에 의존하지 않는 자동 강제"**. 그래서 빌드 테스트가 더 튼튼.

## 5. 한 문장 봉인
> 아키텍처 규칙(사이클 금지 등)은 문서·기억으로는 안 지켜진다 — **깨지는 순간 빌드를 실패시키는 테스트(ArchUnit)로 박아야** 유지된다.

## 6. 이번엔 가드를 만들지 않은 이유
기존 사이클 2건은 **작업플로우 보존**을 위해 유지하기로(같은 결정 일관). 가드를 지금 켜면 그 2건이 바로 빨간불 → "전부 금지 vs 기존 허용·신규만 차단"이라는 설계 결정이 먼저 필요. 그래서 오늘은 **"왜 필요한가"만 닫고** 코드 적용은 i+1로.
